## Bottlenecks / friction points (ranked)
1. **DISK 91% (CRITICAL, dominant).** `/dev/sda1` chronically near-full (6.9GB free).
   This is the single real bottleneck — it throttles cargo target cache, tmp, and any
   large build. CPU (load <0.4) and RAM (9%) are healthy, so compute is NOT the limit.
   - Fix (operator-documented): free non-/root space (apt/journal/docker-prune) and move
     non-critical caches to the 50GB Hetzner volume `/mnt/volume-fsn1-1` via rsync+symlink
     (e.g. `/usr/share/ollama`). `/root` is OFF-LIMITS — never touch.
2. **`TELEMETRY_NO_TG` vs real send latency.** Telegram send is 3-retry × 2s = up to 6s on
   failure. `monitor` loop must not block on a failed send (uses `|| true` — correct).
3. **eqc generation depends on a venv with sympy.** Not installed in system python; CI installs
   sympy via pip. Fine, but bare `python3 tools/eqc/...` fails — document the venv requirement.

## What is fast (not a bottleneck)
- cargo build/test, markov scan, telemetry dispatch — all sub-second, tiny RSS.

## Recommendation
The only high-value action is **disk reclamation on /dev/sda1**; everything else is healthy.
Automate `monitor` (backgrounded, 15s interval) to keep the friction signal live in Telegram.
