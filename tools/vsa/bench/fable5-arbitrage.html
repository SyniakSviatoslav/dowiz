Why it holds: fixed cost vs a growing business

At 2× (Fable-10: 10 hubs, 100 drivers, 900 orders) the naive feed doubles to $24,515/mo and even the compact feed climbs to $720/mo, crossing the image. The frame does not grow — it just packs denser. That crossover is the moat: the more the client scales, the more the image saves.

Driving the floor — 256² macro + delta frames

Claude tokenizes an image at ≈ (w·h)/750, so shrinking the canvas is a direct token lever. A 256² frame costs 88 tokens; a 384² frame, 197. The honest catch is that the same formula is the information ceiling — ~88 perceptual tokens can't carry 450 distinct micro-states — so the low-res tier drops the per-order matrix and encodes only the decision signal (zone heat + per-driver status + spares). Two more levers: a minified legend (telegraphic, 113 tokens vs 235 prose, cached once) and delta frames (when <10% of drivers change, send a tiny strip of only the movers — 11 tokens — and let the model hold the rest from prompt cache).

The true floor — don't send the state at all

Every tier above still pays the model to look at the system. The last step stops doing that. Deterministic code — spatial cull, ETA math, VSA cosine — does the whole calculation at $0 and auto-assigns every clear match with no API call. The model wakes only for a genuine judgment call it alone can settle: a VIP whose nearest driver is poorly rated, a shift ending mid-route, two orders and one last slot. That arrives as a ~50-token micro-prompt against a cached 40-token judge — {task, vip, options:[{d, score, risk}]} → "reply with the driver id".

In steady state the model is never called — most ticks cost nothing, and the monthly bill for the orchestration core falls toward the price of a coffee. The client still sees a system solving hard non-linear logistics; under the hood it's matrix algebra doing the work and the model arbitrating only the handful of real dilemmas. Honest caveat: this is calibration-dependent — a busier fleet (demand > slots) produces more genuine scarcity, and the deterministic resolver has to be right (a wrong silent assignment is worse than an API call, so when truly unsure, it escalates).

The honest fine print

✓ Verified, not asserted

A Claude vision model read the Fable-5 frame and correctly flagged HUB3 as the critical alert (all-red) — the #1 dispatch decision — at fixed token cost. The small-scale frame was read down to each driver's exact matrix.

Legibility ceiling

Past ~16 vehicles per frame, micro-cells drop below ~20px (and Claude downscales >1568px). At 450 orders you reliably get the macro decision — which hub is on fire — not every order. For per-driver detail, drill into the alert hub with a second frame.

It's a decision-support view

The image is read-at-a-glance, not a lossless store. Keep the authoritative JSON server-side; send the image, act on the returned command, verify against source. The renderer runs in ~5 ms, pure-stdlib, zero deps.

Which baseline is fair

The big number is vs the full-object JSON feed most orchestrators actually send. A team already distilling to compact JSON saves less until ~560+ orders — after which the fixed image wins permanently. Measure your real payload before quoting a figure.
