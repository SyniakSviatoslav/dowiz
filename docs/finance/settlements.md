# Courier Settlements

This document outlines the architecture for courier cash settlements and payouts in DeliveryOS.

## Overview
DeliveryOS employs an immutable, append-only cash collection flow for couriers handling cash on delivery. 

### Core Concepts

1. **Idempotent Background Jobs (`settlement-cron`)**: 
   - Runs daily at 02:00 UTC.
   - Collects all un-settled `delivered` assignments with `cash_collected = true` in the prior period.
   - Uses `FOR UPDATE SKIP LOCKED` to prevent duplicate processing.
   - Creates a unique `settlement_items` record per assignment (enforced by DB `UNIQUE` index).

2. **Cash Immutability**:
   - `courier_assignments.cash_collected` and `cash_amount` are the absolute sources of truth.
   - Once written, these fields cannot be updated due to the `prevent_cash_mutation` trigger.
   - The only exception is an explicit settlement reversal which requires the `app.settlement_reversal` flag.

3. **Approval Flow**:
   - **pending**: Generated by the system.
   - **approved**: Owner verifies and locks the amount.
   - **paid**: Owner marks the payout as transferred to the courier.
   - **disputed**: The owner flags discrepancies.

4. **Claim-Check Events**:
   - PII is strictly excluded from `MessageBus` payloads. 
   - `settlement.approved` only emits `locationId`, `courierId`, and `totalEarned`. Listeners (e.g., Telegram bots) fetch PII securely at runtime.
