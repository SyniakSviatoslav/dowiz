## Context

Сповіщення керуються per-event boolean у `owner_notification_targets.prefs`, gating розкиданий по двох воркер-шляхах (`handleTelegramSend`/`handleDispatch`), quiet-hours hardcoded глобально UTC у `event-registry`. Виявлено два пре-існуючі баги: (A) `notification_outbox_audit` CHECK не містить gating-статусів, які `writeAudit` пише (`prefs_disabled`, `quiet_hours`, …) → кожен gating-no-op кидає `23514` і ковтається per-target try/catch → **тихі дропи без сліду**; (B) `owner_notification_targets` має RLS `ENABLE` без `FORCE`, а `notification_outbox_audit` — без RLS. Немає per-category UX, немає inbound-дії storefront open/close, немає consent-логу на зміну prefs.

## Decision

1. **Категорії (3) як новий канон prefs.** `EVENT_REGISTRY[event].category ∈ {transactional, operational, quality}`. `prefs = {"operational":bool,"quality":bool}` (transactional non-mutable, не в prefs). Backfill зберігає поведінку (operational=true, quality=false).
2. **A2 — Policy Gateway:** єдиний категорійний gating у наявному воркер-шарі. Порядок: status → transactional завжди → `prefs[category]` → quiet-hours hold → dispatch. Кожен крок лишає audit (нуль тихих дропів).
3. **B2 — Deferred-deliver:** non-transactional у quiet-вікно → audit `quiet_hours` + re-enqueue (pg-boss `startAfter` = window-open), max 1 re-hold. Без digest (non-goal). 🔴 ніколи не тримаються.
4. **C2 — Idempotent compare-set + RLS-visibility розрізнення для storefront-toggle (RESOLVE BR-23, РАУНД 4):** спільний сервіс `setAcceptingOrders(client, locationId, accepting, {messageBus})` за взірцем `openShift`. `locations` — FORCE RLS вже сьогодні (`core-identity.ts:85-87`), policy на **`app.user_id`** → обгортка ставить `set_config('app.user_id', targetUserId, true)` (НЕ `app.current_tenant` — той GUC policy не читає; raunд-1 хибний GUC давав deny на КОЖНОМУ toggle). NULL-user store-mutating → reject. **0-rows двозначність (idempotent-noop vs RLS-deny) розв'язана `cur FOR UPDATE`-CTE:** `WITH cur AS (SELECT delivery_paused AS was FROM locations WHERE id=$l FOR UPDATE) UPDATE locations l SET delivery_paused=$v FROM cur WHERE l.id=$l RETURNING cur.was, l.delivery_paused AS now`. **3-way:** 0 rows (`cur` порожній під RLS)=**deny→бот ПОМИЛКА+web-fallback (ніколи success)**; 1 row & `was=now`=idempotent-noop (бот «вже …», **не помилка**); 1 row & `was≠now`=changed (бот «✅ HH:MM»+publish). `GUARDED_NOOP`-throw з BR-22 непридатний (store 0-rows ЛЕГІТИМНО=double-tap), тому розрізнення через visibility-сигнал CTE, не throw. **Дві guarded-площини консистентні:** order і store обидві «рядок невидимий (0-rows)=fail». Web `PUT /owner/settings` мігрує на цей же сервіс. Telegram inbound `store.open:<locationId>`/`store.close:<locationId>` його викликає; **authority = (chatId↔target) ∩ (locationId з callback) ∩ membership** — locationId У callback_data + явний target-scope (RESOLVE BR-3). **`store.close` — inline-confirm крок; `store.open` — один тап** (Counsel 3-2).
5. **Per-category prefs write через atomic per-cell `jsonb_set` БЕЗ попереднього SELECT** (RESOLVE BR-4): кожна змінена категорія = окремий statement під row-lock у read-committed → lost-update класу не існує. Web PUT `notifications.ts` видаляє read-merge-write.
6. **Фікс BUG-A (RESOLVE BR-2):** audit CHECK = рівно статуси, що код пише — **`prefs_disabled`** (НЕ `pref_off`), +`held`. `AuditStatus`-union розширюється тими ж рядками.
7. **Фікс BUG-B + RESOLVE BR-20/BR-21 (РАУНД 3 — СТРУКТУРНЕ, замінює worker-escape-GUC раундів 1–2):** RLS `ENABLE+FORCE` на targets, audit, prefs-audit, nonce-таблицях. **Корінь, доведений Ламачем за 3 раунди:** notif-worker і web ділять ОДИН operational-пул під ОДНІЄЮ роллю `deliveryos_api_user` (`server.ts:346`) — будь-яка спроба розрізнити їх self-settable GUC `app.notif_worker` = **self-grantable крос-tenant backdoor** (BR-20: web ставить GUC, читає чужі owner-таргети), а policy на `request.jwt.claim.sub` (0 production-setters) = **outage + тихий prefs data-loss** під FORCE (BR-21). Дисципліною шляхів не лагодиться. **СТРУКТУРНЕ РІШЕННЯ:**
   - **(BR-21) policy-GUC = КАНОН `app.user_id`** — кожна policy 4 таблиць = `location_id IN (SELECT app_member_location_ids())`, рівно як `orders`/`locations` (`core-identity.ts:70-79`). `request.jwt.claim.sub` ВИДАЛЕНО (його прод не ставить). `app.notif_worker` ВИДАЛЕНО. Web prefs-шлях `owner/notifications.ts` мігрує на `BEGIN; set_config('app.user_id', request.user.sub, true); …; COMMIT` **ПЕРЕД** FORCE (enable-gate ordering — інакше 100% web owner-notif операцій мертві).
   - **(BR-20) worker-escape = РОЛЬ, не GUC** — notif-worker конектиться окремим пулом `createNotifWorkerPool()` під **`deliveryos_notif_worker` з verified BYPASSRLS** (`...052` міграція БЕЗ swallowed-EXCEPTION → FATAL-fail якщо платформа не дає grant; boot-assert `rolbypassrls=true` → FATAL-exit). Web-роль `deliveryos_api_user` структурно не може обійти RLS (нема self-settable значення; BYPASSRLS — атрибут окремої ролі, якою web не конектиться). **BR-13 суперседед:** немає GUC → немає ні autocommit-outage (роль обходить FORCE без GUC) ні session-leak; транзакція потрібна ЛИШЕ для prefs+audit-атомарності (BR-16). **Fallback (R-ROLE):** якщо Supabase не дає BYPASSRLS не-superuser-ролі → `policy TO deliveryos_notif_worker` (role-based, BYPASSRLS не потрібен). **DoD:** web живий під FORCE (тест на `app.user_id`); web-роль `set_config('app.notif_worker')`→0 rows (escape мертвий); notif-роль БЕЗ GUC→всі рядки (BYPASSRLS легітимний).
7b. **RESOLVE BR-22 (HIGH, РАУНД 3) — webhook order-action authority:** R-PRE-обгортка ставить `set_config('app.user_id', targetUserId, true)`, де **`targetUserId`** = вже-верифікований active-member (`telegram-webhook.ts:147-157` резолвить, `:185-194` перевіряє `memberships(active)`). NULL-user order-mutation → reject (НЕ порожній app.user_id). **Guard-on-rowcount:** 0-rows guarded-UPDATE під FORCE (RLS-deny не кидає) → `throw GUARDED_NOOP` → бот рапортує **помилку**, НЕ «confirmed»; order лишається PENDING і оператор це бачить.
8. **Operator-action prefs-audit** (`notification_prefs_audit`, RESOLVE Counsel 3-3 — перейменовано з `consent_log`, бо це не data-subject consent) — INSERT у тій же транзакції що зміна prefs; rollback-разом; `source='migration'` для BR-9 backfill-сліду.
9. **`/settings` inline у Telegram** — наявний `handleMessage`, кнопки несуть лише `category`; toggle через той самий atomic per-cell шлях + prefs-audit.
10. **MVP: лише Telegram-канал.** Модель готова до push/email тим самим рядком `owner_notification_targets` — рантайм НЕ вмикаємо ("схема багата, рантайм мінімальний").
11. **BR-6 + BR-14 (timezone, раунд 2):** `locations.timezone text` **NULLable** (НЕ NOT NULL DEFAULT — той мовчки нав'язував намір «я в Тирані» всім рядкам) + **CHECK проти `pg_timezone_names`** (валідація, інакше `AT TIME ZONE 'BadZone'` ламає gating) + **code-fallback `?? 'Europe/Tirane'` з `quiet_tz_fallback`-audit**. Незворотне (transactional) не залежить від TZ. quiet-window math = local minutes-of-day з wraparound. Backfill пілотних локацій — явним UPDATE з `source='migration'`, не німим DEFAULT.
12. **BR-8 + BR-18 (strict secret, раунд 2):** **allowlist default-deny** (НЕ denylist-prefix): явний read-only enum `INBOUND_LENIENT`; усе інше (mutating + link-flow `/start <token>`) вимагає збіжний `x-telegram-bot-api-secret-token` → новий mutating-handler fail-CLOSED. Link-flow завжди потребує secret (це state-write). Flag `TG_WEBHOOK_STRICT` керує лише read-only гілкою.
13. **BR-10 (outbound timeout):** `AbortController` 10s на кожен Telegram fetch — у скоупі MVP.
14. **BR-7 + BR-17 (раунд 2):** order-create при paused → **атомарний `INSERT…SELECT WHERE delivery_paused=false`** → 0 rows = `409 store_paused`. TOCTOU-вікно елімінується (не звужується) — «екстрено стоп» authoritative на момент INSERT. Hard capacity-gate (stock-aware) — окремий скоуп.
15. **BR-15+BR-19 (confirm, раунд 2):** `store.close` → one-shot **nonce** `(nonce,action,expires_at)` + confirm echo'ить `location.name`; крок 2 = `DELETE…RETURNING` consume (прямий/replay/expired confirm → reject). `location_id` з nonce-store (trusted), не з callback.

## ⚖️ ETHICAL-інваріант (RESOLVE Counsel умовний STOP)

> **Категорія = зворотність наслідку, НЕ гучність сповіщення.** Будь-яка подія, наслідок якої незворотний у вікні тиші (`order.timeout_cancelled`, `cash.reconcile_discrepancy`, `delivery.flag_raised`, `order.substitution_needs_human`, та всі order-lifecycle/aging/pickup, courier, ops, shift), лишається `transactional` — пробиває тишу, ніколи не held, не в prefs. `quality` (held-дозволено): `rating.low_received`, `shift.close_reminder`. `operational`: наразі порожня для order-flow (fail-safe: нічого незворотного не held).
>
> **STOP-ETHICS-1 (ОЧІКУЄ людського підпису — НЕ знімається дизайном):** Counsel раунд 2 підтвердив: це **справжній людський акт** про сон живої людини, не технічне питання. Вузько: чи `order.pending_aging` пробиває тишу як **передвісник** (transactional), чи чекає ранку, поки лише його незворотний наслідок `order.timeout_cancelled` будить (operational/held). Обидва етично захищені — різниця у кількості нічних дзвінків оператору. Product+Arch письмово підтверджують ЦЕЙ вузький `pending_aging`-рядок у ADR перед `TG_CATEGORY_GATING=on`. **Fail-safe (поки підпис не поставлено):** `order.pending_aging` лишається `transactional` (пробиває), бо operational-order-flow порожня → нічого незворотного не held навіть якщо акт забудуть. Дизайн НЕ намагається зняти STOP — фіксує очікування підпису.

## Альтернативи (відхилено)

- **A1 scattered inline guard** — дублювання policy по producer-ах, розсинхрон, саме так BUG-A і виник.
- **B1 silent-drop** — оператор втрачає 🟠-контекст за ніч; суперечить "нуль тихих дропів" духу.
- **C1 read-then-write** — гонка read↔write, подвійний тап = два UPDATE, не атомарно.

## Verification (DoD — design-time)

- Forward-only міграції; idempotent (`IF NOT EXISTS`); RLS FORCE на 3 таблицях; integer-гроші не торкнуто.
- **BR-20/BR-21 gate (несучий, РАУНД 3, тестує ТОЙ САМИЙ GUC що policy читає — `app.user_id`):** (1) під FORCE, `set_config('app.user_id', member-of-A)` → SELECT targets WHERE loc=A → **рядки є** (web read-after-write живий, BR-21); (2) `set_config('app.user_id', non-member)` → той самий SELECT → **0 rows** (ізоляція); (3) web PUT prefs під FORCE (member) → UPDATE rowCount=1 + prefs-audit ok (BR-16); (4) на operational-пулі (web-роль), `set_config('app.notif_worker','on'); SELECT targets WHERE loc=чужа` → **0 rows** (BR-20: GUC-escape мертвий); (5) на notif-пулі (`deliveryos_notif_worker`), SELECT targets БЕЗ GUC → **всі рядки** (BR-20: BYPASSRLS легітимний); (6) boot-assert `rolbypassrls=true` інакше FATAL; (7) `grep request.jwt.claim.sub` нових міграцій → 0. FORCE-міграція НЕ деплоїться без 1+3; gating без 4+5.
- **BR-22 gate (webhook authority):** order-confirm з active-member targetUserId → rowCount=1, бот «confirmed»; з non-member → membership-resolver `:185-194` reject; FORCE-deny (порожній app.user_id) → 0-rows → `GUARDED_NOOP` throw → бот «помилка» НЕ «confirmed», order PENDING; `targetUserId IS NULL` + order-mutation → reject.
- **BR-23 gate (store-toggle deny-vs-noop, РАУНД 4):** `setAcceptingOrders` через `cur FOR UPDATE`-CTE під FORCE: (a) member, paused=false, close → 1 row `was=false`≠`now=true` → changed («✅», storefront реально закрито); (b) повторний close (вже paused) → 1 row `was=now=true` → idempotent-noop (бот «вже на паузі», НЕ помилка); (c) **non-member/порожній app.user_id (RLS-deny)** → `cur` порожній → 0 rows → бот **ПОМИЛКА+web-fallback**, storefront НЕ змінено (**deny НІКОЛИ не success**); (d) `targetUserId IS NULL` + store-mutating → reject до CTE.
