# ADR — Go/No-Go Pre-Launch Remediation

- Status: **Proposed** (draft, awaiting review)
- Date: 2026-06-20
- Deciders: Owner, System Architect
- Companion design: `docs/design/golive-remediation/proposal.md`
- Relates to / does not contradict: existing ADRs (monolith-first, transactional outbox, RLS FORCE, integer-money, JWT RS256), `ADR-NOTIFICATION-CONSOLIDATION.md`.

## Context

Pre-launch audit flagged 6 defects + 1 data task. Live source review refined several headline claims (see Decision); the severities stand. A round-2 System-Breaker pass (`docs/design/golive-remediation/breaker-findings.md`, C1–L2) proved two round-1 claims FALSE — "no schema change" and "single transition authority / no logic fork" — plus a wrong prod slug and a non-durable notification dedup; round-2 introduced two additive migrations to fix them. A **round-3 re-attack (RC1/RC2/RH1)** then proved BOTH round-2 migrations broken against the live tree — `CREATE INDEX CONCURRENTLY` is incompatible with the single-transaction migration runner (RC1), and the partial UNIQUE crashes the legitimate multi-row attempt-log (RC2) — and that the detection query had no executor (RH1). The honest resolution: **both round-2 migrations are REMOVED, returning the launch to "Schema changes: NONE"**, and detection is folded into the live standalone sweep. We are pre-launch at pilot scale (1 active tenant, ~3-10 locations, peak ~0.5 orders/sec). Constraints we will not break: integer-money, RLS `ENABLE`+`FORCE`, JWT RS256, no raw runtime `CREATE` on `pgboss` (revoked by migration 0009 — but pg-boss's own `boss.createQueue`/`boss.schedule` DO work at runtime), Postgres-backed idempotency.

## Decision

Ship six forward-only fixes with **NO schema change** (round-2's two additive indexes are BOTH REMOVED — breakers RC1/RC2) and one PII-free data load.

1. **Order auto-cancel (CRITICAL).** Keep the existing dedicated `apps/worker` consumer (`apps/worker/src/handlers.ts:15`, deployed as the `worker` process in `fly.toml:9`). Close the real gaps:
   - (a) emit `order.timeout_cancelled` when the order is observed in CANCELLED-by-timeout state (NOT gated on the emitter's own `rowCount>0` — breaker C4: that made the recovery sweep *silent* on the handler-crash case → zero notification), via the existing `NOTIFY_TELEGRAM_SEND` outbox. Dedup is at the **notification layer** — in-process LRU `Set` (`notifications/workers/index.ts:57`) + `singletonKey` — which is exact within the single web VM. The round-2 partial UNIQUE on `notification_outbox_audit` is **REMOVED** (breaker RC2: that table is a multi-row attempt-log with NULL `target_id` and a bare `INSERT` in `audit.ts:31-44`; a UNIQUE crashes the handler on `23505`). Emitting on *observing* CANCELLED-by-timeout closes the zero-notification hole regardless; the residual cross-restart duplicate is accepted at single-instance pilot (R12).
   - (b) the guarded `WHERE status='PENDING'` UPDATE **IS** the transition authority for this one fixed `PENDING→CANCELLED` edge — **NOT** routed through `assertTransition`, because `apps/worker` cannot import `@deliveryos/domain` (breaker H2 — `apps/worker/package.json` lacks the dep). The round-1 "single transition authority / no logic fork" claim is dropped as structurally false; what is single-source is the *identical SQL+emit contract* in handler and sweep, test-enforced.
   - (c) the worker/sweep UPDATE is documented-RLS-exempt (relies on `deliveryos_api_user`/`BYPASSRLS`) + a guardrail test — NOT `app.current_tenant` (verified no-op: no policy reads that GUC). A future `NOBYPASSRLS` flip is the hard re-validation gate (R4).
   - (d) **a 1-minute reconciliation sweep is MANDATORY (not deferred)**, mounted as its OWN pg-boss schedule in `apps/api` beside the live `NotificationWorker` — **NOT** on the `ReconciliationWorker`, which is REMOVED in prod (`server.ts:382`, breaker C3). **No backing index** (breaker RC1 removed it; pilot seq-scans cheaply). Lost-job detection is a **`pgboss.job` overdue-count query run INSIDE that same sweep job** (breaker RH1 — the round-2 standalone query had no executor: `LivenessChecker` reads only the heartbeat, the `ReconciliationWorker` is removed), NOT the heartbeat (keyed by `FLY_MACHINE_ID`, cannot bind `order.timeout` — breaker C2). Recovery and detection share one live executor.

2. **DB pool wedge (CRITICAL).** **Keep ALL reads INSIDE `BEGIN…COMMIT`** (round-1's "move reads out" is DROPPED: it breaks idempotency — `menu_version` is in `request_hash`, `orders.ts:194,197` — and the money snapshot, since `:376`'s "FOR UPDATE" comment is false and there is no lock). Win via **batching the only N-fan-out**: the per-line `modifier_groups` query (`orders.ts:430-436`) collapses to one set-based query into a `Map<product_id, groups[]>` **broadcast per line-item, never consumed once** (breaker H3 — a 2-line same-product cart would otherwise skip validation). Add `SET LOCAL statement_timeout` **~4.5s** (just inside `connectionTimeoutMillis=5000`, sized for a ~60-insert worst-case cart — breaker M2; round-1's 4s had no headroom and would abort valid large orders). Keep `max: 8`. Double-order on retry is closed at the idempotency layer (content-derived `request_hash`), not by the timeout.
   - Concept: short transaction via round-trip batching + bounded hold + server-authoritative MVCC snapshot. Hold-time roughly halves and stops scaling with cart size.

3. **Health truthfulness (HIGH).** The **pg** check specifically classifies a connection failure/timeout as `down` → HTTP 503 (the existing `hasDown→503` path fires); soft checks stay `degraded`/200. Strip BOTH `data` (raw node-postgres internals) AND `detail` (raw driver error text — breaker L2) from the unauthenticated `/health`; return only `{status, checks:{name:{status,latencyMs}}}`; `detail`/`data` move to authenticated `/health/detail`. The 503 is invisible to Fly (it reads `/livez`, breaker M1) → its observability value needs an external monitor on `/health` (R10, Owner).

4. **Checkout fallback + dev-mock containment (HIGH).** On order-place failure surface a `location.phone` "call the restaurant" CTA for **all non-422 errors** (5xx/network/timeout — breaker H4; round-1 only handled MIN_ORDER, and #2's abort makes a 5xx the likely failure), with `fallback-config` **fetched and cached on page mount** so the CTA doesn't depend on a catch-time fetch under load. Cart preserved. Gate the `o_mock_123` dev-mock behind compile-time `import.meta.env.DEV` so it is dead code in any prod build regardless of session.

5. **Checkout privacy notice (ETHICAL-STOP, human-approved).** Short warm sq/en notice above the submit button: what is collected, why, who sees it (this restaurant + its courier), retention (`locations.retention_days`, displayed), how to remove. New i18n keys under `checkout.privacy.*`. No new data collected, no new column.

6. **ESM test-collection fix (quick).** Replace `__dirname` in `e2e/tests/flow-ui-admin-branding.spec.ts:26` with `fileURLToPath(import.meta.url)` + `dirname`.

7. **Data task.** Seed staging `demo` from prod **`sushi-durres`** (the live slug per `1790000000021:5` — breaker C1; round-1's `durres-sushi` was a typo that would 404 every read) via the **public** menu/theme/fallback APIs (structurally PII-free), transform to a new `demo` tenant. Scrub ALL free-form identifying fields — not just the top-level phone but the `fallback_config` JSON and branding name/contact (breaker M3, since `fallback_config` is owner-authored free-form). Idempotent loader upserts on `slug='demo'` (the only stable natural key cross-DB — breaker M3). Load into `dowiz-staging-db` via `flyctl proxy`. Direct prod-DB read rejected as default (blast radius); allowed only with a column allow-list excluding `customers`/`orders`. Verify post-load: `customers`=0, `orders`=0, `demo.phone ≠ sushi-durres.phone`, and no prod identifying string in `fallback_config`/branding.

## Consequences

- **Positive:** small forward-only diffs; **NO schema change** (round-2's two indexes removed — RC1/RC2); pool wedge fixed at the cause (hold-time, batching); auto-cancel notification gap closed (emit on observing CANCELLED → never zero); lost-job detectable <1 min via a pg-boss overdue count co-located in the live sweep (RH1); outage pages external monitors; checkout degrades gracefully on any non-422 failure; CI suite collects again; staging gains realistic PII-free data.
- **Negative / accepted:** no migration → H1's sweep query is a pilot-scale seq scan (sub-ms over low-thousands of rows, off the hot path; re-index at 10× volume, R11); notification dedup is in-process so a rare api-restart inside the ~60s emit-gap can duplicate one owner Telegram (R12, breaker RC2); #2 keeps a ~intra-tx TOCTOU window (mitigated by the same-snapshot `is_available` re-check + optional `FOR SHARE`, R1); #1 emit is post-COMMIT so a crash after COMMIT but before pg-boss-accept loses one attempt, recovered next sweep tick (R2); `max:8` unchanged (R5); O1/O3 detectors dead in prod until `ReconciliationWorker` is re-enabled (R9); #3's 503 needs an external monitor to have observability value (R10).
- **Open before Go:** point an external uptime monitor at `/health` (R10); re-enable `ReconciliationWorker` to restore O1/O3 in a follow-up (R9); the worker RLS posture is accepted-and-flagged for the `NOBYPASSRLS` re-validation gate (R4); if a 2nd web instance is ever added, promote notification dedup to a durable `event_id`-keyed table (R12). Tracked in the resolution risk table with owners. (R3 "verify queue registered" is CLOSED — subsumed by the mandatory sweep.)

## Alternatives considered

- Rewrite a new in-API timeout worker — rejected: a consumer already exists; the gap is the missing emit + a safety-net sweep, not a missing worker.
- Mount the cancel sweep on the `ReconciliationWorker` — rejected: it is REMOVED in prod (`server.ts:382`, breaker C3); the sweep rides a standalone `apps/api` schedule instead.
- Route the worker cancel through `assertTransition` for "single transition authority" — rejected: `apps/worker` cannot import `@deliveryos/domain` (breaker H2), and one fixed `PENDING→CANCELLED` edge needs no state-machine guard; the guarded WHERE is the authority.
- Detect a lost timeout-job via the heartbeat-watched set — rejected: the heartbeat is keyed by `FLY_MACHINE_ID` with no queue identity (breaker C2); detection is a `pgboss.job` overdue query instead.
- Durable notification dedup via a partial UNIQUE on `notification_outbox_audit` (round-2's M-2) — **rejected in round 3 (breaker RC2)**: that table is a multi-row attempt-log with NULL `target_id` written by a bare `INSERT` (`audit.ts:31-44`); a UNIQUE there throws `23505` on legitimate rows and crashes the handler. The zero-notification crash-case is fixed by emitting on *observing* CANCELLED-by-timeout (not by the UNIQUE); dedup is the in-process `Set` + `singletonKey`, exact within the single web VM. Accepted at single-instance pilot (R12); a durable upgrade is a SEPARATE dedup table (`event_id PK`), never a UNIQUE on the attempt-log.
- A backing index for the sweep (round-2's M-1, `CREATE INDEX CONCURRENTLY orders_pending_timeout_idx`) — **rejected in round 3 (breaker RC1)**: `CONCURRENTLY` is incompatible with the single-transaction migration runner (`1790000000011:18-30`), and a non-concurrent build locks `orders`. The pilot seq-scans a few-thousand-row table cheaply off the hot path; the index is re-added via `pgm.noTransaction()` only at 10× volume.
- Run the lost-job overdue query on `LivenessChecker` or a re-enabled `ReconciliationWorker` — rejected (breaker RH1): `LivenessChecker` reads only the heartbeat and the recon worker is removed; the query is co-located inside the standalone sweep job so detection cannot lose its executor.
- Move reads outside `BEGIN` (round-1) — rejected: breaks idempotency (`menu_version` in the key) and the money snapshot; batching achieves the hold-time win without relocating reads.
- Bump pool `max` — rejected: does not fix head-of-line blocking and spends the scarce Supabase connection budget; makes a wedge bigger.
- Direct prod-DB read for the data task — rejected as default for blast radius; public-API read is PII-safe by construction.
