- **Date:** 2026-06-30
- **Slug:** `menu-characteristics-model`
- **Design doc:** `docs/design/menu-characteristics-model/proposal.md` (FULL BUILD)
- **Relates to / does not contradict:** ADR-001 (monolith-first), the v1 ADR, acquisition read-gate (mig 068/070/072), prep-time/ETA (mig 065), `products.attributes` model, modifier model (mig 1780338982010).

## Context

v1 shipped three surfaces behind `VITE_MENU_CHARACTERISTICS_ENABLED`: L1 taste chips (live), an L2 descriptive band (allowlist EMPTY → dormant), and the allergen **detail-floor** (`MenuPage.tsx:1089-1144`). Comparison, filter, the card character badges, the card allergen unit, and live modifier recompute were all deferred. `packages/ui/src/lib/characteristics.ts` already holds the pure derivation primitives (`DESCRIPTIVE_ALLOWLIST` empty, `REGULATED_ANCHORS` empty, `computeAllergenSurface`, `compareDishes`). The FULL build delivers the deferred surfaces — the hardest being honest recompute on modifier selection, where the data (`modifiers` carry only `price_delta`) does not yet exist.

## Decision

Build the FULL characteristics model as **presentation + one optional additive migration over existing dish data**, on a single pure derivation reused by every surface, in **three independently flag-gateable steps**:

1. **L2 BADGES (card)** — `VITE_MENU_CHARACTERISTICS_ENABLED` (extends v1). Populate `DESCRIPTIVE_ALLOWLIST` with human-reviewed (EN/sq/uk) labels; render the card descriptive band above a per-tenant **descriptive-coverage** gate (descriptive-axis only — governs no allergen element); curate 2–3, "only when notable". Turn on the card **allergen unit** (`…_CARD_ALLERGEN`) — warning + no-data marker **together, ALL-OR-NOTHING** — above an **allergen-coverage** gate keyed on the allergen denominator **plus an absolute authored-dish floor**. Gating guardrails: #1, #4-positive, #5, #5d, #6, #7.
2. **COMPARE-VIEW** — `VITE_MENU_CHARACTERISTICS_COMPARISON`. Exactly two dishes, one mobile screen. **Entry = long-press gesture PLUS a visible "compare" affordance** (a11y/discoverability — the gesture is never the only path). Directional indicator on **price + prep-time ONLY**; **no arrow on kcal/any regulated axis** (a kcal ↓ is the lightness verdict). Prep-time as **"~N min"** point estimate (one int; no range fabrication). Macros side-by-side raw (missing → "—"); taste two profiles side-by-side, **no winner**; allergen cells **always explicit + reliance bound, never blank** (guardrail #8). **No global winner.** Gating: #8.
3. **FILTER** — `VITE_MENU_CHARACTERISTICS_FILTER`. Client predicate over the loaded menu (O(≤200), no backend); non-allergen, non-regulated lenses only (sort-by-protein/energy, taste facets, declared-vegan once the diet field exists); a no-data dish shows "no data", never silently dropped. **Allergen filter stays DEFERRED** (positive-only, NEEDS-HUMAN); guardrail #1 forbids any dish-removing allergen predicate; sort-by-"lightness" deferred with the regulated subset.

**Derivation + threshold source (the named choice).** **Option A — client-derived, hard-coded thresholds in the shared `characteristics.ts` lib** is chosen, over **Option B — server-computed + cached on the menu payload** (rejected: re-versions the 🔴 hot-path `read_public_menu` for a sub-ms pure compute, SQL-localizes thresholds, cache thrash on every edit, and the live selection is client-state anyway) and **Option C — owner-tunable thresholds** (rejected: lets the owner manufacture a verdict with no number changing and hands the regulated-claim threshold to a marketing dial — the exact honesty-anchor/authority inversion the design forbids). Descriptive thresholds are versioned platform config (relative-composition margins, NEEDS-HUMAN to set); **regulated** thresholds come from a verified EU 1924/2006 + AL legal-anchor table (`REGULATED_ANCHORS`, citation + `verifiedBy`) — exact numbers transcribed from the regulation, **NEVER from memory**, red-on-disk OFF until populated.

**Live modifier recompute (two honest contracts).** **v1 ships Contract A only; Contract B is DEFERRED.** **Contract A (data absent, today):** characterize the base dish; **suppress reassuring labels** on modifiable dishes (never caption them); **never suppress the allergen warning** (a modifier can only ADD an allergen). **Contract B (`MODIFIER_NUTRITION_ENABLED`, DEFERRED):** macros = base fold + Σ selected-option deltas → thresholds re-evaluate live; allergen surface = conservative UNION (selection adds, never removes a base allergen); pure & idempotent; if any selectable option lacks a nutrition delta the recomputed reassuring label is suppressed (the allergen union still adds known option allergens). **Honest delivery cost (FB-H1):** modifiers reach the client only via `read_public_menu` (🔴 hot-path), which emits `{id,name,price_delta}` only — so Contract B is **not "additive columns" alone**; it needs a **delivery path for the deltas. We do NOT re-version the hot path**; the deltas ride a **lazy per-product detail fetch on modal-open** (the modal already lazily fetches media on open), preserving the grid "+0 endpoints" budget and the hot-path spine LAW.

**STEP 0 — allergen single-source-of-truth (NO FLAG, lands first).** Two allergen red-line crossings are already in production: a recipe-only live menu filter that ignores owner `declared_allergens` (`MenuPage.tsx:193-194`, FB-C1) and a recipe-only card chip + quick-add path that diverge from the modal's `computeAllergenSurface` (`:854`/`:865`, FB-C2). STEP 0 converges **every** allergen surface (filter, card prop, quick-add, modal, comparison) onto the single pure `computeAllergenSurface(attributes, bomAllergens)` → `{known: declared ∪ recipe, hasInfo}`; `bomToNutrition().allergens` is never read for a safety decision again; the unguarded recipe-only card chip is removed (v1 = detail-floor-only, so no bare card warning manufactures "clean by contrast"). Gated by guardrail **#12**, independent of every feature flag.

**Owner control (asymmetry).** HIDE is allowed and stored as `attributes.char_hidden: string[]` (enum-subset of the derived vocabulary, Zod-validated, `jsonb` passthrough — **no migration**) and applied as a **subtraction** from the true derived set. ADD-a-false-badge has **no representation** (no `char_added`, no union path). Presence→absence flip is impossible (`computeAllergenSurface` never returns absence). Regulated opt-in is a separate owner authority, anchor-gated — not a threshold dial.

**Data / migrations.** SHIP-presentation needs **zero** migrations. The **only** schema change is the optional **modifier nutrition/allergen deltas** (additive `kcal,proteinG,fatG,carbsG int` + `allergens text[]` columns on the tenant-scoped modifier-option table) — 🔴 **a RED-LINE forward-only migration in `packages/db/migrations/`**: atomic, idempotent, must re-assert `ENABLE+FORCE` RLS, integer money/macros half-up, staging-applied before boot, gated dark behind `MODIFIER_NUTRITION_ENABLED`. `attributes.diet` (deferred) and `attributes.char_hidden` ride `jsonb` — no migration. All additive `jsonb` keys are backward-compatible.

## Red lines preserved (carried from v1, LAW)

- **Allergen presence-only / absence-never**, on every surface, **including never by contrast** (card allergen unit is ALL-OR-NOTHING, guardrail #7); a no-data card/cell never reads "clean".
- **Authored allergen WARNING is never suppressed** by any mechanism (modifier / coverage / curation); its unconditional floor is the detail modal.
- **Reliance bound always attached** wherever allergen content renders ("declared to contain… not a complete list — confirm with the venue", EN/sq/uk) so a partial declaration never reads as exhaustive (guardrail #5d).
- **No platform-asserted regulated claims** — owner-authority + verified-anchor + server-sourced, red-on-disk OFF until prerequisites exist. Body-effect phrasing denylisted (CI red). Descriptive vocabulary is a closed human-reviewed allowlist (guardrail #6).
- **Honesty-anchor** — a real qualifying number is necessary (not sufficient for regulated). Borderline → no label.
- **Owner authority for declarations; hide-not-fabricate asymmetry** (subtractive `char_hidden` only).
- **Facts, not a verdict** — no global winner; arrows only on price + prep-time, never on kcal/any regulated axis.
- **Never fabricate composition** — diet/allergen never inferred from ingredients.
- **Hot-path untouched** — `read_public_menu` not re-versioned; `allergens_confirmed` read as provenance, never "reviewed".

## Build order (gate-by-gate)

0. **STEP 0 — allergen single-source-of-truth (NO FLAG, first/independent).** Converge filter + card prop + quick-add + modal + comparison onto `computeAllergenSurface`; converge the live filter (+ coverage disclosure, or gate it OFF — recorded human decision); remove the unguarded recipe-only card chip (v1 detail-floor-only). Gate: **#12** red→green. Fixes FB-C1/FB-C2 (live production red-line crossings) and underwrites every later step.
1. **L2 badges** — flip `VITE_MENU_CHARACTERISTICS_ENABLED` card surfaces only after #1, #4-positive, #5, #5d, #6, #7, **#12** are red→green; descriptive band needs ≥1 reviewed allowlist label; card allergen unit (`…_CARD_ALLERGEN`) needs real allergen coverage + absolute floor, the card-gestalt guardrails **#13/#14**, and a **launch-gate human perception validation** (floor copy + badge-stack gestalt with allergy-affected users).
2. **Compare** (`…_COMPARISON`) — after #8, #11 red→green. **Entry = visible affordance only; long-press removed** (FB-M4).
3. **Filter** (`…_FILTER`) — non-allergen lenses; no-bom dishes in an explicit "no data" group (guardrail **#15**, FB-M3); the live "contains X" allergen view is converged in STEP 0 (keep-with-disclosure vs gate-off is the recorded human decision); any hide→safe transform stays forbidden (#1).
4. **NEEDS-HUMAN gates:** regulated subset (**named legal/Counsel signatory + verified-from-text anchors** + owner authority + server supplies + AL legal review — FB-STOP-1); the `…_CARD_ALLERGEN` perception-validation launch-gate; allergen filter re-enable (positive-only); diet track (`…_DIET_DECLARATION`); modifier deltas (`MODIFIER_NUTRITION_ENABLED` + the RED-LINE migration **+ the lazy per-product delivery path**).

## Guardrails (red→green before each respective flag; SPECIFIED, NOT BUILT — design-time)

Carried verbatim-in-force from v1, plus the FULL-build bindings: **#1** no-hide-allergen (grep, permanent ratchet, gates all); **#2** regulated-subset-off (red-on-disk, gates regulated flag); **#3** body-effect denylist (CI red, all flags); **#4-positive** warning-never-suppressed + reassuring-suppressed-on-modifiable (gates card); **#5** no-clean-from-empty + unconditional detail floor (gates card); **#5d** reliance-bound-rendered in both branches incl. non-empty (gates card); **#6** descriptive ⊆ closed reviewed allowlist (gates descriptive band); **#7** card-allergen ALL-OR-NOTHING anti-contrast (gates `…_CARD_ALLERGEN`, permanent ratchet); **#8** comparison-allergen explicit-both, never blank (gates `…_COMPARISON`). FULL-build additions/revisions to author:
- **#9 (REVISED → permanent ratchet, FB-STOP-3)** `char_hidden` vocabulary = closed `DESCRIPTIVE_ALLOWLIST ∪ TASTE_AXES`, **provably disjoint** from allergen tokens, free-from/reassurance tokens, and any label whose removal manufactures a reassuring absence; Zod `.strict` rejects out-of-set; `computeAllergenSurface` takes no `char_hidden`; the subtraction is in the one shared derivation so a hidden label cannot resurface on compare/filter (FB-M2) — gates the hide feature.
- **#10** modifier-recompute monotonicity (allergen union never removes a base allergen; reassuring label suppressed if any option lacks a delta) — gates `MODIFIER_NUTRITION_ENABLED`.
- **#11** compare arrows only on price/prep, never kcal/regulated — gates `…_COMPARISON`.
- **#12 (NEW, FB-C1/FB-C2)** single-allergen-source: no allergen render path reads `bomToNutrition().allergens` (only `computeAllergenSurface`); no bare card warning without the all-or-nothing marker; a declared-only allergen dish is retained by a "contains X" view and surfaced — gates **STEP 0 + SHIP** (permanent ratchet).
- **#13 (NEW, FB-H2)** card presence coupling: a card showing a reassuring descriptive badge while its allergen unit (warning or no-data marker) is absent is RED — gates descriptive band + `…_CARD_ALLERGEN`.
- **#14 (NEW, FB-STOP-2b)** card layout: the allergen unit precedes/equals the descriptive-badge container in DOM order, is present whenever any descriptive badge is, prominence ≥ the descriptive badge, never evicted/shrunk by the 2–3 cap — gates `…_CARD_ALLERGEN`.
- **#15 (NEW, FB-M3)** sort-by-macro no-data bucket: a no-bom dish renders in the explicit "no data" group, never numerically ranked as 0 — gates `…_FILTER`.

## Consequences

**Positive.** One pure deterministic derivation reused by STEP-0 allergen surface + card + compare + filter → surfaces cannot disagree (allergen single-source-of-truth, #12); +0 DB connections/queries/endpoints for all **presentation** (Contract B's deferred delta delivery is a **lazy per-product fetch on modal-open**, not a grid/hot-path read — the "+0 endpoints" budget applies to the menu grid, which is never touched); instant flag rollback with zero data change; degrades to today's behavior when data is absent; the only migration is additive, FORCE-RLS-safe, and gated dark.

**Negative / costs.** "Computed, zero owner work" holds only where a `bom` exists; **diet declaration and modifier deltas are genuinely new owner work** (deferred/gated). Regulated subset needs verified legal anchors before any label ships. Supply-Library nutrition is `localStorage`-only (snapshots, non-reproducible) — a prerequisite blocker for the regulated subset. Long-press compare needs a visible affordance to stay accessible.
