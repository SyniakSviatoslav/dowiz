- **Red-lines:** money, state-machine, `packages/db/migrations/`
- **Design detail:** `docs/design/audit-fix-money/proposal.md` (v2), `resolution.md` (dispositions), `breaker-findings-r2.md` (re-attack; its N2/N3/N5/N6/N7/N8 are binding implementation pins)
- **Findings:** audit-money-orders 2026-07-03 C1/C3/H5 ≡ synthesis LC1/LC6/B9; relates ADR-0005 (fee mirror), ADR-0017 (crypto C2), ADR-deliver-v2-cash-as-proof (R2-3).

## Context

Three verified money defects:

1. **LC1 (live):** `orders.ts` §9 and the FE mirror (`packages/ui/src/lib/money.ts`) ADD the tax that `applyTax` correctly EXTRACTS from a tax-inclusive subtotal (`price_includes_tax` schema-DEFAULTS true) → every taxed inclusive order overpays by `r/(1+r)` of the cart (16.7 % at 20 % VAT). `fee-parity.test.ts` certified the bug: its expected values were computed FROM the implementation under test (mirror-oracle).
2. **LC6 (dark, blocks crypto flag-flip):** `deliveryCompletion.ts` was the ONLY writer of `refund_due`. Five other sanctioned terminal paths (timeout sweep, owner PATCH, mark-no-show, grace-cancel, courier abort) cancel PAID orders with no obligation; the Plisio webhook flips already-CANCELLED orders to `paid` with no status check. Customer money silently kept; the owner refunds queue never learns.
3. **Settlement (live):** `app_generate_settlements` scanned only `[period_start, period_end)` with `FOR UPDATE … SKIP LOCKED` and each period is generated once → a locked row or a crashed 2 AM run lost cash deliveries from reconciliation FOREVER; the payout upsert then unconditionally bumped `deliveries_count/total_earned` — on a `paid` payout this now provably CRASHES the whole generation run against the immutability trigger (worse than the audit stated; demonstrated red locally).

## Decision

**D1 — Inclusive tax is never additive.** `chargedTax = price_includes_tax ? 0 : taxTotal`; `total = subtotal + deliveryFee + chargedTax - discountTotal`; `taxTotal` stays persisted/displayed as informational extraction. Shipped as (1) the in-place hotfix at both callsites, later (2) Option-B consolidation into one shared `composeOrderTotal` (separate commit; parity-matrix demotion happens ONLY there — breaker M5). Correctness is proven by independent-constant vectors (data-only, zero-import vector file with hand-derivation comments — breaker M4), the definitional invariant `inclusive ⇒ total === subtotal + fee`, and a route-level POST /orders composition matrix (P3b) so the inline server composition has combinatorial coverage before Option B. **FE receipt coherence (CC-3, breaker M7):** `estimateOrderTotal` gains additive `chargedTax`; inclusive receipts render "includes VAT (r%) — X" as an informational line, never an addend. Historical overcharges are enumerable (`overcharge == tax_total` on inclusive venues pre-fix); **restitution is ESC-1**: not closed until a VAT-trace (do `tax_total/total` feed any filing/export?) and an operator DECISION RECORD in `docs/decisions/` exist — silence must not default to keep-the-money. Historical money rows are never retro-mutated.

**D2 — Terminal-state refund obligation is structural: FOUR decoupled layers** (v1's single fail-closed fold is superseded — it wedged the fleet-wide sweep, breaker C1):

- **L-A app-fold in `updateOrderStatus`:** entering `CANCELLED`/`REJECTED` inserts `refund_due` for every `payments` row `status='paid'`, same tx. Fail-closed **per order only** + mandatory surfacing (Sentry + DRIFT log + ops-bus alert); **ESC-2 operator escape hatch** (`opts.forceTerminal`) swallows the fold consciously, writes an audit row, fires the same alert. The R2-3 binding-terminalize fold widens to REJECTED. `completeDelivery`'s own `refund_due` insert is removed (single writer restored).
- **L-B webhook fold:** the 'completed' branch keeps flipping `paid` (money truth — suppressing would hide received funds) and inserts `refund_due` in the same tx when the order is already terminal (CC-2: `payment_status='paid'` on a CANCELLED order = expected "paid-awaiting-refund" state).
- **L-C structural floor — `SECURITY DEFINER` AFTER-UPDATE trigger on `orders` (migration M-1):** covers EVERY writer, present and future, raw UPDATEs included. **Deliberately NON-THROWING** (N8 — inverts the repo's throwing-trigger convention): per-row `BEGIN/EXCEPTION` swallow so one poisoned row can never wedge `app_sweep_timeout_orders()`'s atomic CTE (C1). Subtransaction cost gated behind `IF EXISTS (… payments … status='paid')` so cash cancels pay one index probe (N3). Per-row tenant-GUC dance with pinned semantics (N6, PG16-verified): success restores `COALESCE(saved,'')` ('' ≡ unset for every consumer — audited), error path relies on subxact rollback (exact restore). The sweep fn itself is byte-untouched.
- **L-D reconciler — `app_reconcile_refund_due()` (migration M-3):** called by the timeout-sweep worker EVERY tick; per-row exception isolation; returns `inserted/failed/mismatch` rows; `failed` alarms every tick until resolved (un-recorded but never un-alarmed); `mismatch`-class terminal orders are **surfaced, never auto-obligated** (M3 scope — amount is ambiguous; follow-up `M3-mismatch-disposition` gates "crypto refunds done").

Idempotency does NOT lean on the NULL-able `provider_payment_id` unique (N5): M-1 adds a partial unique `payment_events(payment_id) WHERE type='refund_due'`, and every writer uses arbiter-less `ON CONFLICT DO NOTHING` — at most one obligation per payment across L-A/L-B/L-C/L-D.

**Owner PATCH guard (CC-1, both arms — breaker M6):** PATCH → `DELIVERED`/`PICKED_UP` returns **409 `ASSIGNMENT_ACTIVE`** when an active binding exists, **409 `USE_DELIVER_FLOW`** when the order is IN_DELIVERY without a delivered assignment (was silent-200 + permanent strand). Never-dispatched orders stay PATCH-able (phone/manual flow). FE surfaces "complete via /deliver" for both codes. The same PATCH read is now authorized by a live owner-membership JOIN (LC2, authz council) — 0 rows → 404 before any transition logic.

**LC3 customer cancel (DEP-1, breaker-confirmed):** drop the phantom `cancelled_at/cancellation_reason` writes (columns never existed → live 500); resolve `location_id` from the ownership-verified read; `set_config('app.current_tenant', loc, true)` immediately before the mutation with a minimal GUC window (N7 — only ownership-scoped statements run while set); route through `updateOrderStatus` so the fold + WS + audit + assignment terminalization all apply.

**D3 — Settlement generation becomes catch-up, immutable-once-paid, idempotent** (migration M-2, forward-only `CREATE OR REPLACE`, exact signature match): scans are lower-bounded by a **LITERAL deploy watermark** (`2026-07-10T00:00:00Z`) — **N2 pin: the literal MUST be ≥ the actual prod apply date; erring late is safe (rows defer to operator backfill), erring early double-pays; re-check at apply time.** Post-watermark misses (SKIP LOCKED, crashed run, missed day) self-heal into the next run; SKIP LOCKED is kept (a skip is a deferral now). Paid payouts: row locked `FOR UPDATE`, pair skipped unless `pending` — a paid payout's numbers never move; late items roll to the next period's pending payout. Totals recomputed as aggregates over `settlement_items` (immune to phantom counts); `pg_advisory_xact_lock` single-flights all callers. **Pre-watermark history moves ONLY through the operator-gated flow** (breaker C2): read-only `app_backfill_precount_settlements()` magnitude report → explicit `app_backfill_historical_settlements()` run (never called by cron — structurally asserted) → rows land `pending` with additive `settlement_items.backfilled = true` → `/pay` stays human-gated. Doctrine seam: *auto-mutating money-out: never; auto-surfacing an existing obligation into a human-gated pending queue: allowed — and pre-watermark history does not even auto-surface.*

## Consequences

- Customer-visible: inclusive-venue totals drop by the extracted tax (the correct price); inclusive receipts show "includes VAT (r%)" informationally — visible lines always sum to the total.
- **CC-1:** owner PATCH to `DELIVERED`/`PICKED_UP` on a bound/IN_DELIVERY order returns 409 (`ASSIGNMENT_ACTIVE` | `USE_DELIVER_FLOW`) — FE affordance required for both codes.
- **CC-2:** `payment_status='paid'` on CANCELLED/REJECTED is an expected paid-awaiting-refund state, always paired with an unmatched `refund_due` in the owner queue (≤1 reconciler tick, or an alert).
- **CC-3:** `estimateOrderTotal` return gains additive `chargedTax`.
- Settlement payouts may contain older post-watermark deliveries (caught up); backfilled history is flagged and legible to owner and courier.
- Three forward-only migrations — **M-2** (`1790000000085` settlements catch-up + backfill pair + `backfilled` column), **M-1** (`1790000000086` refund_due trigger + N5 partial unique), **M-3** (`1790000000087` reconciler) — authored as operator-placed drafts in `docs/design/audit-fix-money/migration-drafts/` (migrations dir is red-line gated); staging first, preflight against prod schema per `docs/lessons/2026-07-03-prod-staging-schema-drift.md`.
- LC6 layers ship dark (flags off ⇒ zero paid payments ⇒ all layers match zero rows); crypto flag-flip is gated on the ESC-2 checklist item ("escape hatch + alert path proven on staging") and `M3-mismatch-disposition`.
- DEP-2 (pre-existing, NOT this change): the sweep's own `order_status_history` insert and the settlement fns' cross-tenant writes still need a post-B3 RLS strategy — B3-flip checklist; the L-C/L-D per-row GUC pattern is the template.
- Proofs: P1–P16 in the design doc, red→green with independent expected values (literal constants, definitional invariants, DB-state counts) — never mirror==mirror; local DB-level proof suites `apps/api/tests/settlements-catchup.test.ts` + `refund-due-spine.test.ts` (skip without `MONEY_TEST_DATABASE_URL`), route-level `order-total-route-composition.test.ts` + `orders-status-patch-guards.test.ts`.

## Alternatives considered

- **Single fail-closed fold everywhere (v1):** rejected — a throwing insert inside the sweep's atomic CTE wedges fleet-wide cancellation (breaker C1); liveness of a batch must never couple to a money-ledger write.
- **App-fold only, no trigger:** rejected — the grep-gate exempts exactly the DEFINER-fn class where LC6's original leak lived; "remember to add the fold everywhere" is the model that already failed. The trigger needs no allowlist and no memory; its non-throwing inversion is documented (N8) so it is never "fixed" back to the throwing template.
- **Suppressing the webhook `paid` flip on terminal orders:** rejected — hides received funds; recording payment + obligation is truthful and drives the existing refunds surface.
- **Unbounded catch-up scan (no watermark):** rejected — auto-asserting weeks-old unsettled rows that were plausibly reconciled in person pays couriers twice (breaker C2); history is operator-pulled with a magnitude pre-count instead.
- **Dropping SKIP LOCKED in settlements:** rejected — blocks the fleet-wide run on any long app transaction; catch-up scanning makes skipping safe instead.
- **Auto-`refund_due` for `mismatch` funds:** rejected for now — the obligation amount is ambiguous (over/under vs recorded); mis-stating a money obligation is worse than alert-plus-human. Surfaced-only; follow-up `M3-mismatch-disposition`.
