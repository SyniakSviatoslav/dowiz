### [SEVERITY: CRITICAL] [MATH/FIELD-OPERATOR] FEYNMAN-01
**Where:** `engine/src/field_frame.rs:9-14,156-179` vs `docs/design/physics-ui-capture-blueprint.md:13,29,41`, `docs/design/internal-retrieval-living-memory-blueprint.md:357,434`, `docs/design/MASTER-EXECUTION-PLAN-2026-07-13.md:54`, commit `9106534c1`
**What:** The canon claim "ONE physical operator `MÜ+ΓU̇+c²LU=S` draws the whole UI" is actually FOUR different equations across the corpus, and the dowiz field integrator's own header derivation is algebraically a non-sequitur from its own first line.
**Evidence:**
- Canon (physics-ui blueprint:29): `MÜ+ΓU̇+c²LU=S` — **second-order** damped wave, "roots −γ/2±i√(ω²−γ²/4)".
- Commit `9106534c1` subject: `M∇²U+ΓU̇+c²LU=S` — a third variant (M multiplies ∇²U, dimensionally different).
- `field_frame.rs:10` header: `M·U̇ = -ΓU̇ - c²·L·U + S` — **first-order**, and then claims `⇒ U_next = (U + dt·(Γ·U̇ + c²·L·U) + dt·S)/(1+dt·M)`. That implication is false: rearranging the header's first line gives `U_next = U + dt/(M+Γ)·(−c²LU+S)` — the "⇒" line flips the sign of BOTH `ΓU̇` and `c²LU` and moves M from a U̇-coefficient to a U-coefficient.
- What the code (lines 170-179) actually integrates, recovered by algebra: `(U_next−U)/dt + M·U_next = +Γ·U̇ + c²·∇²U + S`, i.e. continuous limit `(1−Γ)U̇ = −M·U + c²∇²U + S` — a **screened-diffusion/decay equation with an anti-damped momentum term** (+ΓU̇ ADDS velocity; in every damped-oscillator equation Γ opposes it). "M" here is a decay rate (units 1/time), not mass. There is no Ü anywhere; `u_prev` exists only to form U̇.
- Only bebop's `crates/bebop/src/field_physics.rs:334 step_wave` (`a=(c²Lu−damp·v+src)/m; v+=dt·a; u+=dt·v`) genuinely implements the claimed second-order damped wave.
- `internal-retrieval-living-memory-blueprint.md:434` additionally claims `motion.rs` IS "`MÜ+ΓU̇+c²LU=S` (FE-08)" — `motion.rs` is an uncoupled per-property scalar spring `ẍ+2ζωẋ+ω²x=ω²x_target`; there is no Laplacian L in it at all.
**Why it matters:** The "grand unification — ONE Laplacian, ONE operator" is the arc's central architectural claim (memory: "UI=ONE operator"); P51 builds route-glow on `compose()` "by construction" of this operator. In reality three implementations called by one name obey three different dynamics; anyone transferring analysis (energy decay rates, resonance, Lyapunov structure) from the documented equation to `field_frame` will get wrong predictions. The header's broken derivation means nobody re-derived it after writing it.
**Fix guidance:** Pick the actual equation per module and write it honestly: `field_frame` = semi-implicit screened diffusion `U̇+MU = ΓU̇_back + c²∇²U + S` (note the equilibrium is the nice fact: `U* = (M−c²∇²)⁻¹S`, a genuine screened-Poisson Green's function — better math than the false wave claim); `motion.rs` = uncoupled damped oscillator per property (no L); `step_wave` = the damped wave. Delete "ONE operator" or scope it to "one Laplacian primitive, three dynamics".

### [SEVERITY: HIGH] [MATH/STABILITY] FEYNMAN-02
**Where:** `engine/src/field_frame.rs:18-27,56-73` (`FieldEquilibrium::assert_stable`)
**What:** The "fail-closed" stability assertion `dt ≤ M/(Γ+2c²)` is not a stability bound — it admits timesteps under which the scheme provably diverges — and the eigenvalue-range comment is wrong for 2D.
**Evidence:** Comment claims "the Laplacian's discrete eigenvalues live in [-4, 0]" — that is the 1-D 3-point range; the implemented 2-D 5-point stencil (`left+right+up+down−4u`) has eigenvalues `2cosθx+2cosθy−4 ∈ [−8, 0]`. Von Neumann analysis of the actual recurrence `(1+dtM)·u_{n+1} = (1+Γ+dt·c²λ)·u_n − Γ·u_{n−1}` at defaults (M=1, Γ=0.2, c²=1): claimed bound = 1/2.2 ≈ **0.4545**; true stability boundary (Jury condition p(−1)>0) = (2+2Γ)/(8c²−M) ≈ **0.3429**. Independently recomputed this audit: spectral radius ρ(dt=0.4545, λ=−8) = **1.588** (unstable); direct simulation of the exact update at dt=0.45 reaches **|u| ≈ 1.2×10³⁹ after 200 steps** while `assert_stable` passes. All existing tests use dt=0.02 (ρ≈0.76), so the region [0.343, 0.455] where the assert lies is never exercised.
**Why it matters:** A guard whose name is "fail-closed" and whose comment says "a divergent dt can never reach the integrator" admits divergent dt. Any future caller trusting the assert (e.g., adaptive-dt work, P38a GPU port re-using the constant) inherits a latent blow-up. The bound formula `M/(Γ+2c²)` matches no derivation (dimensional numerology).
**Fix guidance:** Derive the real bound from the two-step recurrence's Jury conditions with λ∈[−8,0]: `dt < (2+2Γ)/(8c² − M)` (for 8c²>M), or simply assert `dt ≤ 0.5·(2+2Γ)/(8c²)` with margin; fix the eigenvalue comment to [−8,0]; add a RED test at a dt inside the currently-admitted unstable band.

### [SEVERITY: MEDIUM] [MATH/CITATION] FEYNMAN-05
**Where:** `kernel/src/spectral.rs:404-413` (`graph_energy`)
**What:** The citation appears fabricated and both stated bounds are false; the computed quantity itself is the standard definition.
**Evidence:** Comment: "Graph energy E = Σ|λᵢ| … (Gutman–Adrić, 2001) … Bounds: 2(n−1) ≤ E ≤ 2n√(n−1) (empty / complete extremal graphs)". Graph energy is **Gutman 1978** ("The energy of a graph"); no co-author "Adrić" exists in this literature (Gutman's 2001 piece is a solo survey). ### [SEVERITY: MEDIUM] [MATH/ROUTING] FEYNMAN-06
**Where:** `kernel/src/router.rs:52-61,168-219` (`Shortcut`, `build_shortcuts`) vs roadmap P51 §12 / P31a ("Dijkstra/A*/CH landed")
**What:** "Contraction Hierarchies" is present in vocabulary only — the three mechanisms that make CH be CH (importance-ascending contraction, witness search, bidirectional upward-only query) are all absent; the shortcuts as implemented can only make queries slower, never faster.
**Evidence:** (a) Contraction order is **degree-DESCENDING** (`order.sort_by(|&a,&b| deg(b).cmp(&deg(a))…)`, line 178) — real CH contracts least-important nodes first precisely so shortcut fan-out stays small; contracting hubs first generates O(deg²) candidate shortcuts around the largest hubs. (b) No witness search: a shortcut is added whenever `wa+wb < direct(a,b)` (line 208) without checking whether a cheaper path through other nodes already exists — real CH adds a shortcut only when the contracted node is on the sole shortest path. (c) The query (`route`, lines 102-139) never uses `rank`: shortcuts are just appended as extra edges to a **unidirectional full-graph** Dijkstra/A* — the entire CH speedup (bidirectional search restricted to upward edges) does not exist. Correctness is safe (each shortcut's weight is a real path's cost, so optima are preserved — the `ch_preserves_cost` test verifies exactly and only that), but the data structure is pure overhead.
**Why it matters:** P51's routing design cites "the already-landed zero-dep kernel router (Dijkstra/A*/CH)" as its engine and rejects OSRM/Valhalla partly on that basis. At venue scale it won't matter; at city-scale MapPacks the promised CH speedup is not there to be had — the honest label is "Dijkstra/A* + optional precomputed shortcut edges (no speedup)".
**Fix guidance:** Either implement the real thing (ascending importance, witness Dijkstra, upward bidirectional query) or rename `Shortcut`/`build_shortcuts` to drop the CH claim and delete the dead `rank` machinery; add a bench falsifier (query node-pops with vs without shortcuts) — today it would show shortcuts ≥ baseline.

### [SEVERITY: MEDIUM] [GUARD/MONEY] FEYNMAN-07
**Where:** `engine/src/money_guard.rs:8-14,52-62,98-112`
**What:** "Money never tweens" claims TWO enforcement layers; the runtime layer is provably dead code (its Err branch is unreachable for every possible input) and its mirror test is vacuous.
**Evidence:** `present_money(amount_minor: i64)` rejects when `amount_minor as f64 != (amount_minor as f64).round()`. Every i64 casts to an f64 whose value is an exact integer (large magnitudes lose precision but remain integral), so the condition is false for all 2⁶⁴ inputs — the guard has never rejected and can never reject anything. The docstring "a caller that tried to pass an interpolated float (e.g. lerp result) is rejected" is impossible: such a caller must cast to i64 first (truncating), after which the guard accepts. The test at 104-107 asserts `is_ok == (fract < 1e-9)` — always `true == true`. The type-level layer (Money does not impl `FieldValue`, so `interpolate(Money…)`/`Spring<Money>` won't compile) is real and IS the actual enforcement.
**Why it matters:** This is a 🔴-labeled red-line module. The stated defense-in-depth is single-depth; a reviewer counting two independent layers is counting one. (Also `Money(pub i64)` has a public field, so `Money(lerp(...) as i64)` bypasses everything — only convention prevents it.)
**Fix guidance:** Either make `present_money` take a type that CAN be fractional (`f64`) so the check is live, or delete the runtime layer and state honestly that enforcement is compile-time-only; make the Money field private with an integer-only constructor.

### [SEVERITY: MEDIUM] [MATH/RETRIEVAL] FEYNMAN-08
**Where:** `kernel/src/retrieval/ppr.rs` vs `docs/design/internal-retrieval-living-memory-blueprint.md:50,113,222,379`
**What:** The blueprint claims PPR "via the local **push algorithm** (Andersen-Chung-Lang)… local-push (no eigendecomp)"; the implementation is dense power iteration — a different algorithm with a different complexity class — and the doc's resolvent formula uses α with the opposite meaning from the code's α.
**Evidence:** `Ppr` stores a **dense** `Vec<Vec<f64>>` and `rank()` does K full passes `nxt[j] += pii·((1−α)·w[i][j])` — O(n²) per iteration, touching every node (the entire point of ACL local-push is touching O(1/ε) nodes independent of n; nothing resembling a residual/push queue exists). The doc's spine table (line 379) writes the resolvent as `(I−αW)⁻¹` where its α is the damping/continue probability; the code's `alpha` is the **restart** probability, so the implemented fixed point is `π = α·e_seed·(I−(1−α)W)⁻¹` — same symbol, complementary meanings (doc α ↔ code 1−α). The unification claim itself (PPR fixed point = a resolvent = Green's-function analogy, eigenvalue map 1/(1−αλ)) is mathematically legitimate, and the code's mass-conservation argument (Σπ stays 1 exactly given row-stochastic W, enabling divide-free determinism) checks out. P26's `O(n²)→O(nnz)` CSR-delegation note is honest about the density.
**Why it matters:** "Local-push, no eigendecomp" is cited as the scalability argument for L3 retrieval; what exists scales as dense n². The α convention flip is exactly the "same symbol, two meanings" trap that produced the prior pass's Legendre finding.
**Fix guidance:** Amend the blueprint: "power iteration now (dense, small-n), ACL push = the named upgrade path"; pick one α convention and state it in both places; land the P26 CSR delegation.

**What:** "No central issuer at runtime" is honestly scoped, but the trust root is a genesis-frozen, runtime-irrevocable anchor set — decentralized *liveness* with a centralized, unrepairable *trust* root, on the exact key path the roadmap itself flags as classically-hardened-only.
**Evidence:** `roster.rs`: "the roster is enrolled **exactly once, at genesis, and then frozen**. There is exactly one set of anchors and it **never grows or shrinks during runtime**." So an anchor-key compromise cannot be revoked or rotated without a new genesis (capability revocation exists; anchor revocation does not — :221 concerns delegation revocation being irreversible, the opposite direction). Compose with the roadmap's own §1.1 finding: "the trust root itself is classically forgeable (`roster.rs` delegation chain is Ed25519-only, making the ML-DSA leg vacuous)" and the still-open C4b `mod_l` constant-time leak (§1.7) on that same Ed25519 path. **Why it matters:** For long-term viability this is the honest single point of failure: not availability (mesh keeps running) but authority — one leaked anchor key = unlimited capability minting until every node re-bootstraps a new genesis by hand. The architecture documents the tradeoff's halves in different places but never composes them into this sentence.
**Fix guidance:** Name it in canon as the residual centralization (M7 is about liveness-SPOF; add a trust-SPOF note); design an anchor-rotation/epoch mechanism (even operator-gated re-genesis with signed hand-off) before the mesh is load-bearing; Phase 3's hybrid-chain fix addresses forgeability but not irrevocability — they are separate holes.

### [SEVERITY: LOW] [MATH/NAMING] FEYNMAN-16
**Where:** `engine/src/motion.rs:85-92` (`heat_kernel_delay`)
**What:** "Heat-kernel stagger" implements ballistic (wave-front) scaling, not heat-kernel (diffusive) scaling, and the claimed unit is dimensionally wrong.
**Evidence:** `delay = graph_distance / α.sqrt()` — linear in distance with propagation speed √α: that is a WAVE arrival time. **Why it matters:** Same decorative-language class as the exemplar's "contour-integral stability"; it also means anyone "tuning α like a diffusivity" gets linear, not quadratic, response.
**Fix guidance:** Rename to `wavefront_delay`/`stagger_delay`, or actually implement d²/α if diffusive feel is wanted.

### [SEVERITY: LOW] [MATH/ROUTING] FEYNMAN-18
**Where:** `kernel/src/router.rs:12-14,71-72,90-91,117-132`
**What:** The A* optimality justification cites admissibility, but the implementation's closed-set/no-reopen discipline requires the strictly stronger property (consistency) — the code is correct under its stated precondition for the unstated right reason, and silently wrong outside it.
**Evidence:** Lines 117-121 and 130-132 never reopen a visited node. Textbook fact: admissible-but-inconsistent heuristics can make closed-set A* return sub-optimal paths. The comments ("admissible lower bound… so A* never returns a sub-optimal path", "MUST measure i→dst **or** the heuristic is inadmissible") state admissibility as the sufficient condition. What actually saves the code: haversine-to-a-fixed-dst satisfies the triangle inequality, hence is *consistent* whenever every edge weight ≥ the haversine between its endpoints. But `road_graph_from_ways` accepts arbitrary `(u,v,cost)` triples — feed travel-time costs (numerically smaller than metres) and the heuristic silently becomes both inadmissible and inconsistent with no guard or warning.
**Why it matters:** P51 plans time-ish costing evolutions ("dynamic costing" is the named Valhalla boundary); the first person to put minutes in `cost` gets quietly wrong routes, not an error.
**Fix guidance:** Document consistency as the operative property; add a debug assertion (or a constructor flag) that edge weight ≥ haversine(u,v) when `heuristic=true`, else force `h≡0`.
