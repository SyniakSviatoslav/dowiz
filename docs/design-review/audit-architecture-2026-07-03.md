# Architecture & Tech-Debt Audit — dowiz / DeliveryOS

**Date:** 2026-07-03 · **Type:** READ-ONLY, higher-altitude structural review (no code changed)
**Repo state:** branch `integrate/merge-to-main`, ~197k NLOC, apps/api (Fastify) + apps/web (React) + apps/worker + packages/{domain,db,platform,ui,config,shared-types,voice}
**Method:** first-hand reading of the crown-jewel files (server.ts, orders.ts, spa-proxy.ts, packages/db, packages/platform, config, shared-types, domain) + 4 parallel fan-out lanes (backend-route consistency, contract layer, frontend god-files, dead-code/coupling). Every claim is grounded in a real `file:line`.

> **Already-known (not re-reported here):** server.ts/orders.ts decomposition is a known-open item; NOBYPASSRLS deferred; raw-query-vs-`withTenant` *security-leak* angle (covered by the security lane — this doc covers the *architectural seam* angle); feature-branch/main divergence.

---

## The shape of the debt (executive summary)

The codebase is **not** over-engineered — the dominant failure mode is the opposite: **the right shared primitives exist but adoption is partial and uneven**, so the same concept is modeled 2–5 ways and the "shared" layer is bypassed by inline copies. Four structural themes recur:

1. **Missing data-access layer.** ~777 raw `.query(` calls live directly in 65 route handlers; the `withTenant` seam is used in only 22 of them. There is no repository. This is the *root* that forces most of the duplication below.
2. **Aspirational contract layer.** `packages/shared-types/src/contracts/` holds 124 Zod schemas — with **zero** importers in `apps/api/src`. Request/response shapes are re-defined inline per route (42 files) and re-declared again in the FE, so FE↔BE agreement is by convention, not by a shared source of truth.
3. **God-hubs that everything must touch.** The top churn files (server.ts 91 commits/90d, MenuPage 89, spa-proxy 59, MenuManager 56) are hot *because* they are catch-all hubs. spa-proxy.ts is the clearest: a "proxy" that is actually the de-facto home of ~19 owner-domain endpoints.
4. **Naming/abstraction lies.** `RedisMessageBus` is Postgres LISTEN/NOTIFY; `legacy.ts` holds the *live* order contract; `stubs.ts` is dead and shadows the real provider; `spa-proxy.ts` proxies nothing. Each misdirects the next contributor.

Ranked findings follow (leverage = impact ÷ effort, most central first).

## TIER 1 — Highest leverage (structural, central, silent-drift)

### F1. `spa-proxy.ts` is a misnamed god-file and a split-brain owner API — HIGH · effort M–L
- **Where:** `apps/api/src/routes/spa-proxy.ts` (885 lines, **59 commits/90d**, 99.4%ile churn). Its header comment says *"maps React SPA URL patterns to real backend routes"* — but it **defines ~19 first-class owner endpoints**: `/api/owner/analytics` (:296), `/api/owner/orders` (:393), `/api/owner/couriers` (:452), `/api/owner/brand` (:528/562/616), `/api/owner/settings` (:667/701), `/api/owner/customers` (:838/856), `/api/owner/onboarding` (:758), `/api/public/theme/:slug` (:506), plus image/media serving.
- **Structural problem:** the owner API has **two homes and two URL conventions**. Flat scheme `/api/owner/couriers` (spa-proxy.ts:452) vs nested `/api/owner/locations/:locationId/couriers` (routes/owner/couriers.ts:18). `/api/owner/orders`, `/api/owner/settings`, `/api/owner/brand`, `/api/owner/analytics`, `/api/owner/customers` exist **only** in spa-proxy — there is no `routes/owner/orders.ts` etc.
- **Why it's a liability:** every owner feature must edit this one file → it is a permanent churn/merge hotspot; the tenant-resolution helper is copy-pasted inside it twice (:57 and :133, comment *"Mirrors getLocationId above"*); it leaks internals in errors (:228/:249/:254/:290 return `{ error, detail: err.message }` — ADR-0010 B4 violation). New contributors cannot predict which file an owner endpoint lives in.
- **Refactor:** dissolve spa-proxy — move each endpoint into its correctly-named `routes/owner/*.ts` under one URL convention; keep only the genuine static/image passthrough (rename to `static-assets.ts`). **M–L.**

### F2. No repository layer — 777 raw queries, the tenant seam applied in 22/65 files, one authz helper cloned 5× with divergent failure — HIGH · effort L (M per surface)
- **Where:** ~777 raw `.query(` in `routes/` vs ~110 `withTenant`; 22 of 65 route files import `withTenant`, 38 hand-roll `db/pool/client.query` with no scoping helper. The `FROM orders` column list is copy-pasted across ~30 sites (dashboard.ts ×8, spa-proxy ×6, orders.ts ×6, telegram-webhook ×3, customer/orders ×3); `mapOrderRow`-style row mappers are re-defined in orders.ts, customer/orders.ts, owner/signals.ts, owner/gdpr.ts.
- **The seam is inconsistent *inside a single file*:** `orders.ts` POST uses raw `db.connect()`+`BEGIN`+manual `client.query` (:104–118) while the PATCH `/status` handler in the same file uses `withTenant` (:860). `courier/assignments.ts` copy-pastes the `connect→BEGIN→set_config→COMMIT→release` boilerplate ~8× (:81,111,139,190,251,317,431,496) — and `GET /assignments/:id` (:109) calls `set_config(...,true)` with **no BEGIN/COMMIT** to bind the local scope to. `customer/orders.ts` uses a *third* isolation model (bare `WHERE customer_id=$2`, no `set_config`).
- **Authz drift (security-adjacent):** the ADR-0004 live-membership check `getOwnerLocationId` (`lib/get-owner-location.ts`) is reimplemented ≥5 times with **divergent failure semantics** — `owner/promotions.ts:14` **throws** (→ generic 500) where `owner/products.ts`/`categories.ts` return `null` (→ clean 401). Same authz condition, different HTTP outcome per route; a fix to one won't propagate.
- **Why it's a liability:** duplication of SQL + tenancy + authz is the root that makes every other route-layer inconsistency possible; a schema change touches ~30 hand-written SELECTs; an authz fix must be applied 5×.
- **Refactor:** promote `withTenant` to the mandatory data seam; extract per-aggregate repositories (`OrderRepo`, `CourierAssignmentRepo`, `NotificationTargetsRepo`); delete the 5 tenant-resolution copies in favor of `getOwnerLocationId` with a single `null→sendError(401)` convention (add a regression test — authz red-line). **L overall, M per surface.**

### F3. The `contracts/` layer is an orphaned parallel type system — FE/BE agree by convention, not contract — HIGH · effort M
- **Where:** `packages/shared-types/src/contracts/{common,courier,customer,owner,public}/*.ts` exports **124 Zod schemas**. `grep contracts/ apps/api/src = 0 importers.` Only 5 of 65 route files import `@deliveryos/shared-types` at all (orders, order-messages, customer/orders, owner/menu-import, public/menu) — and they import the **legacy** re-exports, not `contracts/`. Spot-check: `CreateProductBody`/`UpdateProductBody`/`ProductTranslationBody` (contracts/owner/products.ts) have **0** app consumers; `routes/owner/products.ts` (472 lines) defines its own inline validation instead. 42 route files hand-roll `z.object(...)`.
- **The *real* shared contract is mislabeled `legacy.ts`:** `CreateOrderInput`, `StatusUpdateInput`, `OrderStatusEnum` live in `packages/shared-types/src/legacy.ts` (279 lines, imported by 10+ files incl. `orders.ts:2`, `websocket.ts`, `plugins/auth.ts`) but the barrel calls them *"Legacy exports kept for backward compatibility"* (index.ts:4). A migration into `contracts/` stalled; the name now signals "don't use" while the whole system depends on it.
- **Enforcement is inverted and one-sided.** The contracts that *are* parsed live in `legacy.ts` (order shapes); the organized `contracts/` are decorative. Where a `contracts/` schema *is* used, only the **FE** parses it (via `apiClient({ schema })`) — the API validates a *different* inline body schema, so the two hand-copies drift with no cross-check. **Live, already-drifted example:** `contracts/owner/promotions.ts:15` makes `valid_from` **required**, but `owner/promotions.ts:103` re-declares it `.optional()` with a `new Date()` default (:129); the FE form types against the contract and believes it mandatory — the API disagrees, and the POST response (`promotions.ts:138` `send(res.rows[0])` raw snake_case `RETURNING *`) is validated by neither side. Same pattern for the `Product` entity, declared **3× across two casing conventions** — camelCase `ProductResponse` (contracts/owner/products.ts:57, not `.strict()`), a narrower camelCase FE admin interface (`hooks/useMenuData.ts:9`), and a **snake_case** storefront interface (`MenuPage.tsx:73`, importing no contract and casting blindly).
- **No WS contract at all:** no shared discriminated union for ~18 realtime event types — matched by duplicated string literals across the API publisher and the `useWebSocket` client, frames `JSON.parse`→`any` never validated. This has produced **live latent bugs**: the server emits `task_offered` (owner/dashboard.ts:336) but the courier client only handles `task_assigned` (TasksPage.tsx:72) → an offered task never appears live (~8 emitted event types are handled by no client); `orderStatusService.ts:193` emits a camelCase `statusAtField` column-name used blindly as an object key at `OrderStatusPage.tsx:303` → a rename silently stops the stepper timestamp filling; `order.status` is overloaded with two incompatible payload shapes disambiguated only by which room it arrives on.
- **Why it's a liability:** 124 schemas of dead documentation give false confidence; shapes drift silently because each side owns its own copy (e.g. `customer/orders.ts:5` imports `CustomerOrderStatusResponse` but hand-builds the 60-line payload at :150 and never parses it).
- **The cheap root:** the project already depends on `fastify-type-provider-zod` but never feeds contracts into route `schema.body`/`schema.response`. Wiring the existing contracts into those blocks converts them from decorative to server-enforced in one mechanical pass, and makes the FE `apiClient({ schema })` and the server validate the *same* object.
- **Refactor:** pick ONE source. Either (a) delete `contracts/` and formalize inline+`legacy.ts` as the SoT, or (b) finish the migration — handlers register `contracts/*` as Fastify body+response schemas, FE imports the same, and a `contracts/ws-events.ts` `z.discriminatedUnion('type', …)` (reusing the domain status enum) types both `publish` and `useWebSocket` ingress. **M–L.**

### F4. Order status is modeled 4+ ways with a live label bug; the state machine is not a chokepoint — MED–HIGH · effort M
- **Where:** `ORDER_STATUSES` (packages/domain/order-machine.ts:3) is the intended SoT, but the enum is independently re-declared as: `OrderStatusEnum` literals (legacy.ts:82), the DB enum (migrations/…extensions-and-enums.ts), and FE literal maps. `OrderStatusPage.tsx` alone holds **4 parallel maps** (`STATUS_LABELS_KEYS`:22, `STATUS_VARIANTS`:33, `STATUS_ACCENT`:45, `STATUS_MESSAGE_KEYS`:57); more in `StatusBadge.tsx`, `OrderCard.tsx`, `DashboardPage.tsx:294`, courier `CouriersPage.tsx:50`.
- **Concrete bug:** `PENDING` → label key `order.pending` ("Pending") in `StatusBadge` but `order.placed` ("Placed") in `OrderStatusPage:23` — the same status renders **different words** on the badge vs the status page.
- **`order_type` is modeled 3× and the copies *disagree*:** the DB enum has `('delivery','pickup','scheduled')` (migrations/…enums.ts:13) — 3 values; the contract `CreateOrderInput.type = z.enum(['delivery','pickup'])` (legacy.ts:42) — **2, no `scheduled`**; the FE `DeliveryType = 'delivery'|'pickup'|'scheduled'` (checkout/types.ts:1) — 3. So the FE type system permits a `scheduled` order the API rejects with a 400, and the DB carries a value the contract forbids. Reconcile whether `scheduled` is real (add to contract) or dead (drop from DB/FE).
- **Machine bypassed:** `updateOrderStatus`→`assertTransition` guards the main path, but `orders.status` is also mutated by raw SQL that skips the machine: `apps/worker/src/handlers.ts` (`SET status='CANCELLED'` timeout), `customer/orders.ts:310`. Courier-assignment / shift / payment statuses have **no** machine (ad-hoc `SET status='...'`); a *second* state machine lives outside the domain package (`modules/acquisition/state-machine.ts`). `OrderStatusEnum` and `ORDER_STATUSES` are in sync today but nothing enforces it — a status added to the machine won't reach the contract enum.
- **Refactor:** one `packages/ui/orderStatus.ts` SoT (`status → {labelKey, variant, accentToken, messageKey}`) derived from the domain enum; route every `orders.status` write through `updateOrderStatus`; add an ESLint-local rule banning `SET status` on `orders` outside the service. **M.**

## TIER 2 — High leverage (contract/consistency, broad blast radius)

### F5. Error-envelope drift: 87 raw ad-hoc error bodies bypass ADR-0010, some leak internals — MED–HIGH · effort S–M
- **Where:** 87 raw `reply.status(n).send({error/code…})` vs 298 enveloped `reply.sendError(...)`; only 0–2 `throw new ApiError` in routes (the `throw ApiError` half of the contract is effectively dead — the `setErrorHandler` at server.ts:443 mostly catches Zod/validation, not domain errors).
- **Three different 429 shapes in `orders.ts` alone** (:80, :267, :292) — `retryAfterSeconds` (ad-hoc) vs the contract's `retryAfterMs`, none carrying `correlationId`. FE cannot reliably branch on throttle.
- **B4 leaks:** `spa-proxy.ts:228/249/254/290`, `owner/menu-import.ts:54/610`, `courier/me.ts:119` serialize `err.message` / raw `ZodError.format()` to the client — the exact thing the envelope exists to prevent. Wrong-slot/lowercase codes: `admin/backups.ts:79`, `payments-webhook.ts:14…`, `owner/promotions.ts:350`.
- **Refactor:** route all error returns through `sendError`/`rateLimitError`; drop `detail`/`details`; lint-ban raw `reply.status().send({error` in `routes/`. **S–M.**

### F6. Feature-flag sprawl across 4 surfaces; payments/voice/courier flags bypass the Zod schema entirely — MED–HIGH · effort M
- **Where:** flags live on four disjoint surfaces with **no registry**: (1) validated in `packages/config` (11 `z.enum(['true','false'])`); (2) **unvalidated raw `process.env`** — `PAYMENTS_PREPAID_ENABLED`/`PAYMENTS_CRYPTO_ENABLED` (lib/payments/registry.ts:6 — config defines *no* payments flag), `VOICE_CONTROL_ENABLED`/`VOICE_KILL` (lib/voice-flag.ts:12), `COURIER_OFFER_HANDSHAKE_ENABLED` (owner/dashboard.ts:323); (3) 14 `VITE_` twins in apps/web that must be hand-synced with the backend (the documented "flag bakes OFF unless build-arg passed" gotcha); (4) per-tenant DB columns (`pickup_enabled`, `widget_enabled`, `require_phone_otp`, `busy_mode`). 51 distinct `process.env.*` keys across 38 api-src files.
- **Why it's a liability:** unvalidated flags default silently and skip the boot-guard; FE/BE toggles drift; there is no place to enumerate "what flags exist." One logical flag (MEDIA_RICH) is expressed 3× (config + VITE_ + DB).
- **Refactor:** make `packages/config` the single flag registry — all `*_ENABLED` through the Zod schema (incl. payments/voice/courier); derive the `VITE_` public subset from it; read flags only via `loadEnv()`. **M.**

### F7. `server.ts` god-file with 257 lines of inline dev handlers duplicated in `routes/dev/mock-auth.ts` — MED · effort M
- **Where:** `apps/api/src/server.ts` (890 lines, **91 commits/90d**, the #1 churn file) mixes ~15 concerns (process guards, Fastify config, Zod validator/serializer compilers, CORS, static/cache headers, subdomain rewrite, correlation-id, 5 pool/bus/queue constructions, metrics, worker boot-budget, rate-limit, the 75-line `setErrorHandler`, route registration, SPA fallback, shutdown) **plus three full inline dev route handlers** (`/api/dev/mock-auth` :549, `/api/dev/create-assignment` :653, `/api/dev/seed-data` :701 — ~257 lines).
- **Duplication:** those inline handlers duplicate `routes/dev/mock-auth.ts` (which registers `/dev/mock-auth`, `/dev/create-assignment`, `/dev/seed-visual-state`) — the synthetic-courier mint block is copy-pasted near-verbatim between server.ts:563 and mock-auth.ts:27. Two homes for the same dev logic differing only by an `/api` prefix.
- **Refactor:** extract a `bootstrap/composition.ts` (pools/bus/queue/providers) and move all dev handlers into the single `routes/dev/*` plugin; `main()` should read as register-and-listen. **M.**

### F8. `orders.ts` POST is a ~640-line god-function — MED · effort M–L
- **Where:** `routes/orders.ts:71–709`. Already *partially* decomposed (pricing, persistence, preflight, dispatch, signals live in `lib/`), but the POST handler is still one monolithic transaction script mixing: rate-limit config, validation, connection+tx management, OTP verify (in two places, :145 and :307), phone+IP throttle (near-verbatim duplicate blocks :254 and :279), signals, idempotency, product/modifier verification, delivery/tax rules, customer upsert, persistence, bus publish, customer token, and a crypto-payment fork.
- **Refactor:** extract `ThrottleService` (collapse the two velocity blocks), `IdempotencyStore`, `CartValidationService`; the handler becomes an orchestrator. **M–L.**

### F9. FE god-files: state-soup pages + a 4,296-line flat i18n catalog — MED · effort L
- **Where:** `MenuPage.tsx` (1768, biggest file — 31 useState/10 useEffect/5 raw fetches + safety-critical allergen logic inline at :118–188 + a ~350-line product-detail modal at :1272); `MenuManagerPage.tsx` (1292, 99.7%ile) whose churn root is an **11-field product-form `useState` soup reset in 3 separate places** (:362/:380/:398) — every menu-field change touches all three + the JSX; `CheckoutPage.tsx` (764, sections extracted but a 33-field form soup remains); `DashboardPage.tsx` (835, 2 WS subscriptions + soup). None use a reducer or store.
- **The real i18n hotspot moved:** `packages/ui/src/lib/i18n.ts` is now a 74-line runtime (the CLAUDE.md/Repowise "#1 churn i18n.ts" note is **stale** — indexed 2026-06-14). The churn magnet is `packages/ui/src/lib/i18n-catalog.ts` = **4,296 lines, 1,424 keys × 3 locales, one flat file** (45 commits/90d) — a permanent merge hotspot that ships all admin+courier strings to the client bundle.
- **Refactor:** `useProductForm` reducer + `<ProductFormModal>` (kills the MenuManager churn root); extract `<ProductDetailModal>`/`useMenuPageData`/`menuFilter.ts` from MenuPage; split `i18n-catalog.ts` by the existing `client.*`/`admin.*`/`courier.*` prefixes for per-app bundles + edit de-contention. **L.**

## TIER 3 — Real but narrower (misleading abstractions, dead code, dup rules)

### F10. Misleading abstractions & phantom infra — MED · effort S each
- **`RedisMessageBus` is Postgres LISTEN/NOTIFY.** `message-bus.ts:242` — `export const RedisMessageBus = PgMessageBus` with comment *"aliased to make the test pass since Redis is inaccessible."* `server.ts:227` instantiates `RedisMessageBus` (passing a PG session pool). `packages/platform` declares `ioredis` as a dependency but **never imports it** (phantom dep). Meanwhile the real Redis (`new Redis(env.REDIS_URL)`, server.ts:352) is used as a **KV store** for OAuth state/import progress (auth.ts:46/76/161), *not* pub/sub — the inverse of the config comment `REDIS_URL // upstash, pub/sub only` (config:10). Realtime is Postgres LISTEN/NOTIFY (an intentional, ADR-0001-documented choice for the *queue*) wearing a Redis name. Rename to `PgMessageBus`; drop the `ioredis` dep or the phantom; fix the config comment. *(NB: the queue-in-Postgres decision itself is sound; the liability is purely the naming lie.)*
- **`stubs.ts` (31 lines) is fully dead and name-collides with the real provider.** `StubNotificationProvider`/`StubGeocodingProvider`/`StubThemeRenderer` have zero live importers; the `NotificationProvider` interface here *shadows* the actively-used one at `notifications/provider.ts:75`. Delete + drop from the platform barrel.
- **`spa-proxy.ts`** (F1) and **`legacy.ts`** (F3) are the other two naming lies.

### F11. DB pool construction is not centralized; the backup pool escapes the RLS guard — MED · effort S
- **Where:** `packages/db` exposes `createOperationalPool` (:6543, max 20, with a connect-time RLS-superuser guard at index.ts:32) and `createSessionPool` (:5432, max 3, no guard). But `server.ts:214` builds a **third pool inline** — `new Pool({ connectionString: DATABASE_URL_MIGRATIONS, max: BACKUP_POOL_SIZE })` — bypassing the factory, so it has **no `statement_timeout` and no RLS-superuser guardrail**, and it runs on the privileged migrations role. pg-boss constructs a fourth (port-mutated) pool; `scripts/bootstrap-pgboss.ts` a fifth on `DATABASE_URL_ADMIN`. Four distinct DB URLs in play.
- **Latent:** ADR-0001 pinned the safety budget at 14 peak connections (op 8 / session 3 / pg-boss 3) on Supabase free tier; config has since raised the operational pool to 20 — the budget the ADR relied on is exceeded.
- **Refactor:** add a `createBackupPool()` factory in `packages/db` carrying the same guard; centralize all pool construction there. **S.**

### F12. Allergen taxonomy fragmented across 3 vocabularies — SAFETY-adjacent · effort S–M
- **Where:** canonical `EU_ALLERGENS` (shared-types/allergens.ts, 14 entries incl. `milk`) vs UI `ALLERGEN_COLORS` (ui/constants/allergenColors.ts:7 — 9 entries, uses `dairy` not `milk`, missing celery/mustard/sulphites/lupin/molluscs) vs fixtures using `seafood`/`milk`. `dairy` never matches a `milk` value; the UI package doesn't import `EU_ALLERGENS` so nothing enforces agreement.
- **Refactor:** `EU_ALLERGENS` is the only allergen key set; UI derives its color/label map from it. **S–M.**

### F13. Dual worker topology — `apps/worker` is deployed but nearly vestigial (SPOF) — MED · effort M
- **Where:** `apps/worker` is real and deployed (fly.toml:19/40) and is the **sole** consumer of the `ORDER_TIMEOUT` queue — yet it registers only 2 handlers (one, `health-job`, is a dead `console.log` no-op). Meanwhile `bootstrap/workers.ts` runs a ~20-worker fleet **in-process inside the API**. "API hosts most workers, a separate deploy hosts exactly one load-bearing timeout" is architecturally muddy and a latent SPOF (if `worker` isn't running, order timeouts never fire).
- **Refactor:** either fold `ORDER_TIMEOUT` into the API fleet and retire `apps/worker`, or move the fleet into `apps/worker` and make the API request-only; delete `health-job`. **M.**

### F14. Cross-layer business-rule duplication — MED · effort M

### F15. Validation & pagination idiom divergence across sibling routes — LOW–MED · effort S

## Positives / dismissed hypotheses (to prevent false alarms)
- **Package graph is a clean DAG** — no app→app imports, no package→app upward imports, `packages/ui` never imported by `apps/api`, no package-level cycles.
- **`money.ts` vs `currency.ts` are complementary,** not duplicates (arithmetic vs currency selection). Money formatting via `PriceDisplay`/`formatMoney` is a clean single authority.
- **`RoutingProvider`/`QueueProvider`/`MessageBus` are legitimate 2+-impl / multi-caller abstractions** (not ponytail) — only their *unused* members (F10/F16) are findings. The one real leak: `queue.boss` is accessed directly 82× across the API, so the `QueueProvider` interface (enqueue/work/start/stop) is a partial fig-leaf.
- **`mockData.ts` is DEV-gated and tree-shaken** from prod (main.tsx:111) — not a leak. (The real mock leak is `SettingsPage`'s `MOCK_SETTINGS` used as a *production* fetch-failure fallback, :152/:156.)
- **`apps/worker` and `packages/voice` are used** — not orphans (F13 is about topology, not deadness).
- The apiClient (`lib/apiClient.ts`) is a well-built central client (transparent single-flight refresh, envelope-aware `ApiError`, optional Zod validation) — a positive; note only that response-schema validation is opt-in and its `ApiError` name collides with the backend's.

## Top-3 highest-leverage refactors

1. **Dissolve `spa-proxy.ts` + collapse the 5 tenant-resolution clones (F1 + F2's authz half).** These are the #2 churn hotspot and an authz-adjacent silent-drift pair; both are S–M and unblock a coherent, predictable owner API surface. Do the tenant-helper collapse first (one authz truth, regression-tested), then relocate spa-proxy's endpoints into correctly-named `routes/owner/*`.
2. **Pick one home for contracts and status (F3 + F4).** Decide whether `contracts/` or inline+`legacy.ts` is the SoT and delete the other; add a shared WS event union; derive one `orderStatus.ts` map from the domain enum (fixes the live PENDING label bug). This removes the largest body of dead-but-authoritative-looking code and the highest FE↔BE drift risk.
3. **Introduce the missing data-access seam (F2 core).** Extract per-aggregate repositories so the ~777 raw queries and ~30 copied order-SELECTs collapse behind `withTenant`. This is the root that keeps regenerating the duplication in F5/F8/F14 — the single change with the widest downstream payoff.
