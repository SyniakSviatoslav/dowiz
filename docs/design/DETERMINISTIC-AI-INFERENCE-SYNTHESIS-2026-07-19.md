## 1. Grounding Results — What Is Real in This Codebase Today

### 1.1 There is NO AI/ML inference engine in either repo — but the adjacencies are real and specific

GROUNDED (exhaustive grep this session across `kernel/`, `engine/`, `apps/`, `tools/`, every workspace `Cargo.toml`, and `Cargo.lock`, plus `/root/bebop-repo`): **no tensor inference engine, no quantized model, no model-weight artifact, and no ML framework dependency exists anywhere.** `candle`, `burn`, `tract`, `ndarray`, `onnx`, `tch`, `ggml` — zero hits in any manifest or lockfile of either repo. The space-grade synthesis §18(a)'s honest non-finding ("no ML/tensor quantization exists anywhere in the kernel") is re-confirmed.

What DOES exist is a ring of real, deterministic, hand-rolled adjacencies — none of which is an inference engine, all of which are the exact substrate one would need:

- GROUNDED: `kernel/src/attention.rs` — a deterministic scalar softmax + one-step scaled-dot-product attention, documented as a *lens* ("attention = one step of diffusion over a learned affinity matrix"), with the explicit stance in its own module doc: "the kernel stays non-AI (deterministic pure functions); learning lives in `online` / `micrograd` at the edge if ever needed."
- GROUNDED: `kernel/src/simd.rs` — a real AVX2 `core::arch::x86_64` f64x4 batch lane (`_mm256_max_pd`/`_mm256_sub_pd`/`_mm256_add_pd`, runtime `is_x86_feature_detected!`, scalar fallback) with a **bit-identity design rule**: vectorize *across* independent rows, never *within* a row's reduction, so each lane replays the exact scalar op order. Falsifier tests enforce it. `kernel/src/householder.rs:33-43` adds an `_mm256_fmadd_pd` dot-product kernel. bebop2 carries `core/src/keccak_x4_avx2.rs`. **SIMD-intrinsics-not-compiler-autovec is already house practice.**
- GROUNDED: `kernel/src/mat.rs` — ONE contiguous row-major dense matrix + ONE matmul implementation (f64), by its own doc "the ONE backing store," created to kill `Vec<Vec<f64>>` pointer-chasing.
- GROUNDED: `kernel/src/micrograd.rs` — a minimal reverse-mode autodiff engine ("WITHOUT pulling in a vendor ML runtime (TF/XLA — explicitly rejected in the Master-Integration plan)… same inputs ⇒ bit-identical grads") and `kernel/src/online.rs` — deterministic local-only learners (`LinearSGD`, `ScalarAdam`). These are *training-side edge substrate*, deliberately outside the kernel's decision paths.
- GROUNDED: `llm-adapters/` — LLM inference exists in this project **only as an external-process boundary** (Ollama/vLLM/managed-API over HTTP, `route_model` in `ollama.rs:33-45`). And the Layer-F blueprint (`BLUEPRINT-P-F-local-ai-mesh.md` §0) **rejects distributed LLM inference on measured physics**: `llama3.1:8b` Q4 = 4.9 GB weights, throughput flat at 9.21/9.36/9.80 tok/s for 1/2/4 concurrent inferences — memory-bandwidth-bound; the settled design is "gossip the compiled OUTPUT of inference — DecisionUnits — not the inference."
- GROUNDED — the single most important precedent: **the kernel has already executed the IEEE-754-rejection move once, in production.** `docs/regressions/REGRESSION-LEDGER.md` row 25: float `sin`/`cos` in `empirical_identify` had platform-dependent ULP error → replaced by `tools/eqc-rs/src/cordic.rs` — integer-only Q30 fixed-point CORDIC, digest-pinned, consumed by the kernel (`d692c59fc`, `a0a375ad8`). The raw prompt's central thesis ("ви не отримаєте детермінізму, поки використовуєте float") is not a new philosophy here; it is a regression-ledger entry with a commit hash.

**Verdict (a), stated plainly:** an in-kernel deterministic inference engine is **genuinely greenfield as a subsystem** — but it is *not* greenfield in discipline. Fixed-point emission (`eqc-rs emit_fixed_rust`, Q-format, refuses-never-falls-back), integer CORDIC, `core::arch` SIMD with bit-identity falsifiers, the BumpArena, golden-signature drift gates, from-scratch Keccak, hand-rolled CRC32 (in `kernel/src/fdr/`), and ML-DSA code-signing (`kernel/src/pq/codesign.rs`, pinned-root update blobs) all exist and are tested. The new work is composition, not invention.

### 1.3 eqc-rs is the natural generator seam — via an extension that is already ruled PURSUE

## 2. The Five Open Questions, Resolved Under the Ruling

The operator's ruling — **safety and predictability over speed** — is applied as the tie-breaker to each. None is left open.

**Q1 — Own-kernel engine vs. TVM/Burn with hard settings → RESOLVED: own-kernel, zero-dep.** Three independent forcings, each sufficient: (i) GROUNDED — the zero-dep gate is live with an *empty* allowlist (§1.4); TVM or Burn would be the largest dependency this kernel ever carried, reversing today's Tier-1 close on the day it landed. (ii) The ruling itself: a minimal hand-rolled engine is auditable to the instruction level (the dialogue's own §5 argument); a framework's codegen is precisely the "compiler-chosen" surface the house discipline distrusts (binary-audit item 14 exists because of it). (iii) Precedent: Keccak-f[1600], the FDR logger, the regex-replacing pattern matcher, and CORDIC were all hand-rolled under identical reasoning. **Scope honesty (see §3): this rules out LLM-class models permanently** — the P-F physics measurement (§1.1) stands; LLM inference remains an external boundary via `llm-adapters`. The own-kernel engine targets small frozen models (KB–MB weights) only.

**Q2 — Statically pin `fold_transitions` vs. move it to a separate core → RESOLVED: moot pending re-measurement; if real, static data layout wins; separate-core is rejected.** Contingent finding honored: §1.2 found **no committed evidence the +16.6% regression exists** — no action may be taken on an unreproducible number (item 33 re-measures). If a fresh tracked run reproduces it: (i) the "statically pin the data" answer **already exists on the roadmap as item 3** — the `order_machine` const-adjacency rework (`[u16; 12]` compile-time bitmask) moves fold_transitions' working set into const memory, the most predictable placement possible; (ii) separate-core is rejected under the ruling *and* under GROUNDED precedent — `kernel/src/core_pinning.rs` is a deliberate no-op seam whose DECART decision already found "on a single-socket host there is no locality to exploit"; cross-core scheduling adds host-scheduler nondeterminism, the opposite of the ruling. Cache-line data layout is deterministic; core migration is not.

**Q3 — Train-time determinism vs. inference-only → RESOLVED: inference-only, permanently, with training outside the kernel.** The dialogue's own argument (parallel training is not bit-reproducible at this guarantee level) is confirmed and sharpened by the kernel's standing law: space-grade §16(b) already outlaws runtime-mutating learned weights as "exactly the non-reproducible cause→effect mapping P6 outlaws." GROUNDED: the kernel's own module docs pre-draw this boundary — `attention.rs`: "learning lives in `online`/`micrograd` at the edge if ever needed." The shape: **any** learning/fitting is an offline, build-time act (edge substrate: `micrograd.rs`/`online.rs`) whose output is a frozen artifact admitted through the item-41 signing path; the kernel only ever *executes* frozen artifacts. This is the same shape as P-F's "gossip the compiled OUTPUT of inference" — compiled artifacts move, computation doesn't.

**Q4 — Weight loading: embedded-in-binary vs. external signed blob → RESOLVED: embedded, per the ruling — with the in-repo-correct mechanism and honest costs.** The dialogue's lean is confirmed: code+weights as one immutable, atomically-deployed, signed unit eliminates the I/O-error, parse, version-skew, and load-time-attack classes entirely — predictability's definition. Mechanism, corrected to this repo's actual precedent (§1.5): **v1 is a generated, committed Rust `static WEIGHTS: [i8; N]`** emitted by the same generator pipeline as `eqc_gen.rs` (hand-inspectable, diff-reviewable, zero build machinery), wrapped `#[repr(align(64))]` (first in-repo use — named as new), with an `init`-time SHA3-256 self-check against a golden hash (reusing `event_log`'s from-scratch Keccak) and ML-DSA code-signing via the existing `pq/codesign.rs` when shipped as an update blob. The objcopy/`link_section` "Linker Way" is **deferred with a named reopening trigger** — weights outgrowing committed-source practicality (≈ >1–2 MB or a measured build-time regression) — per the item-25 procedure's park-the-alternative discipline. **Honest costs accepted under the ruling:** binary size grows by weight size (acceptable at pilot scale, a permanent ceiling on model size); every model change is a full kernel release (under this ruling that is a *feature* — atomic deploy, no skew); the zero-dep CI gate is untouched (no crates involved; `objcopy`, if ever adopted, is CI-time tooling in the same category as Kani).

**Q5 — Quantization format: i8-symmetric vs. exotic → RESOLVED: i8-symmetric, per-tensor scale, integer-domain end-to-end.** Predictability reasoning: symmetric quantization eliminates zero-points — fewer moving parts, simpler proofs; per-tensor (not per-channel) single scale keeps every layer's law a *scalar* equation eqc can express; scale constrained to a power of two where accuracy permits (requantization = arithmetic shift with `div_half_up` rounding — GROUNDED: already generated and tested in `eqc_gen.rs`'s money organs), falling back to Q-format fixed-point multiply otherwise. Accumulators i32 (the AVX2 `_mm256_madd_epi16`-class path the dialogue names) with proven no-overflow bounds per layer (bounded input domain × weight range — the bound is part of the format spec, checked by the emitter, refused if unprovable — eqc's refuse-never-fall-back stance). Exotic formats (Q4_K blocks, asymmetric affine, f16) optimize size and accuracy-per-bit — speed/size-axis wins the ruling explicitly subordinates. **The exhaustive-proof dividend, named:** the i8×i8 product space is 65 536 pairs — the *same* "65536-pair exhaustive proof" scale the source dialogue cites as the house standard; the core multiply-accumulate law is exhaustively provable, not merely sampled. Dequantize-on-the-fly (float domain) is rejected outright — it reimports IEEE 754 into the hot path.

**Cross-references binding this arc to session precedent:** any dependency question follows the item-25 10-step standing procedure (BINDING per its own §3). The §4 hardening checklist (oracle + dudect-with-planted-leak + debug-differential + item-14-triggered binary audit) applies to every kernel of this arc **unchanged — no parallel checklist is invented**; the NTT wire-in remains the template. Hermetic principles applied where they genuinely fit: P2 Correspondence — the golden-layer-checksum reuses `fdr`'s hand-rolled CRC32, the init self-check reuses `event_log`'s SHA3, the oracle reuses `mat.rs`'s matmul shape (one primitive each, never a second copy); P6 Cause-and-Effect — the entire arc is P6 made executable (replayed input ⇒ bit-identical activations); P7 Gender — golden checksums must be *re-executed* in CI, never presence-checked. P3/Rhythm and the rest are not force-fit.

## 3. Scope and Feasibility — Honest Assessment

**This is a genuinely new subsystem, not a bolt-on** — the largest single build proposed since the breaker (item 9) and the FDR (items 4+29): a number-format spec, a generator extension, a reference oracle, an arena layout, SIMD kernels, a checksum oracle, a weight pipeline, and a scheduler. Mitigations, both real: (i) every hard sub-problem has a live in-repo precedent (§1.1's list); (ii) the roadmap items are sized so each lands independently green, same as Tier 0/1 did.

**Relation to the zero-dep constraint:** strictly additive-safe — the arc introduces zero crates by construction, and the live empty-allowlist gate would catch any drift structurally (§1.4).

**Relation to the item-9 breaker (the roadmap's declared pivot):** the golden-checksum oracle's "hard-fail to safe state" (item 40) *wants* the breaker as its receiver — a checksum mismatch is exactly a `CommitError`-class "must alarm" event. Until item 9 lands, the fail is a typed trap (panic-handler path) plus an FDR entry; when the breaker exists, the mismatch routes through `Result<Permit, Tripped>`. The design work does **not** gate on the breaker; the *composition* is named so nobody re-derives it. Items 33–38 are breaker-independent; only item 40's final wiring composes with item 9.

**What this arc is NOT:** not an LLM runtime (P-F physics closed that — `llm-adapters` remains the boundary); not a training system (Q3); not a GPU story (space-grade §18(c)'s CPU/host scope holds — GPU matmul is the "чорна скринька" the dialogue itself rejects); and not yet a committed model — **no model exists in-repo today**, which is why item 34 (pilot workload selection) precedes all build items. Candidate real-product pilot surfaces were presented for the ruling: a retrieval reranker head (consuming `attention.rs`'s existing softmax shape), an anomaly/drift scorer over `Verdict`/`DriftClass` features, or an ETA-adjacent regressor — each small enough for embedded weights and bounded-domain proofs. **Operator's ruling on item 34 (recorded 2026-07-19, CLOSED): SYNTHETIC/TOY PILOT FIRST** — a small hand-built synthetic classifier (toy MNIST-style / hand-authored pattern classifier, KB-scale weights, zero product data, zero PII, zero product risk) that proves the *whole* determinism pipeline (quantization, arena, oracle, embedded weights) end-to-end BEFORE any real product workload is attempted. The three real-product surfaces above are therefore **DEFERRED to a follow-on second pilot gated on the toy pilot landing green** — NOT selected now, NOT a design-only exercise (the toy pilot is built). A consequence worth naming: a toy classifier's inputs are **public/synthetic by construction**, so the arc's items carry no PII/product-data handling and item 43's constant-time gate takes its cheap-but-optional branch for this pilot (the mandatory dudect/secret-plane branch is reserved for the deferred real-product pilots).

**Alignment worth naming (PROPOSED):** P-F's "domain experts as DecisionUnit FAMILIES — compiled `decide()` functions, hub-only build-time oracle" is philosophically the same object this arc builds: a frozen, verified, compiled decision artifact. If a numeric model ever joins a DecisionUnit family, this arc's engine is its execution substrate, and `import_unit()`'s replay-before-persist admission applies to model artifacts as it does to decision units. That convergence is a design option for the future, not a dependency of this arc.
