Slug: `owner-token-revocation`
Role: Counsel (Philosopher / Physician). Advisory. Aesthetics + strategy are NON-BLOCKING.
Reviews: `docs/design/owner-token-revocation/proposal.md`, `docs/adr/0004-owner-token-revocation.md`.
Date: 2026-06-23

`apps/api/src/plugins/auth.ts:44-92` (owner path is pure-JWT, courier path is checkout-per-request),
`apps/api/src/routes/auth.ts:234-305` (refresh single-use rotation + family reuse-detection EXISTS and works),

## 0. One correction that moves the whole argument

The proposal (§1.2) and ADR-0004 (Context.2) both say a removed owner "rolls forward **indefinitely** as long as they keep refreshing." **That is not what the live code does.** The refresh handler:

- mints from `memberships WHERE ... status = 'active'` (`auth.ts:289`) — so a *membership-revoke* is already honored at the next refresh, with `activeLocationId` resolving to `null` for a removed owner;
- writes refresh tokens with **`now() + interval '7 days'`** (`auth.ts:300`, `local.ts:153`), **not 30d**. The refresh *family* itself expires in 7 days.

So the true, current blast radius is not "indefinite." It is: **a leaked access token is valid up to 7 days; a revoked membership keeps its baked `activeLocationId`/role until the access token's exp or the next refresh, bounded by the 7d refresh family.** The roll-forward is bounded by 7 days, not unbounded, and membership status is *already* re-read on refresh. This matters because the entire framing of "build a revocation layer or the owner rolls forward forever" rests on a claim the code contradicts. The real gap is narrower and slower-burning than the prose. The Architect should correct §1.2 before this ADR is cited as justification — an over-stated threat is how a small SaaS talks itself into infrastructure it doesn't need.

## 1. Reasoning by lens (only what's load-bearing)

### Care / harm — who gets hurt when this fails, and how
The honest failure question is not "does it revoke" but "what does a non-expert operator *believe* it does." The design has three humane strengths: fail-open is correctly chosen (a DB blip must not log out every owner mid-shift — that turns a maybe-theft into a certain self-inflicted outage during someone's dinner rush), `/health: degraded` tells the truth, and the courier-path 500-not-401 invariant is preserved. Good.

The harm vector is the **gap between the marketing of "revocation" and its delivered semantics**: a coarse, eventually-consistent (≤30s), fail-open control. If an owner clicks "log out all devices" because they think their phone was stolen, and the design serves a fail-open allow for ≤30s on a Redis+DB double-fault, the *thief is still in* for that window while the owner has been told "done." For a small delivery SaaS with no compliance mandate, ≤30s is genuinely fine — **but only if the operator and the UI never over-promise.** A "Signed out everywhere ✓" toast that is actually "signed out within 30 seconds, unless our database is having a bad moment, in which case maybe not" is a small dishonesty. Pin the truth to the copy, not just the metric.

### Honesty / consent — the flag is the trap
`OWNER_REVOCATION_ENABLED` defaults **off**, and off ⇒ `tv` ignored ⇒ *exactly today's behavior: no revocation at all.* This is correct as a dark-deploy stance. The danger is **stage 2**: once the staff-management UI ships a "Remove owner" / "Log out" button, an operator can ship that button while the flag is still off — and the button will **silently no-op**. The owner is "removed" in the UI, the row says removed, and their token keeps working for up to 7 days. That is a soft-confirm-as-pasture pattern: the UI claims an effect the server does not deliver. This is the one place I'd put a real guard (see ETHICAL-STOP below). It is not a smell — it is a UI that lies about a security action.

### Justice / stakeholders
Cost/benefit is well distributed and modest. The owner pays nothing on the hot path (cache hit ≈ free). The platform pays ~2.3 reads/s today. Nobody's dignity is on the line — this is owner-side session hygiene, not courier surveillance. The folded membership recheck actually *tightens* tenant isolation (today's baked `activeLocationId` outliving a membership is a real cross-tenant staleness). No autonomy/dignity red-line is touched here; this is owners managing their own accounts.

### Long horizon / strategy — the thing I'd regret in a year
This is a **hand-rolled session/revocation layer** (token-version column + per-machine LRU + Redis pub/sub + sticky hard-revoke flag + fail-open semantics + a 4-mint-site claim change + an RLS self-read policy that doesn't exist yet) being built **before the staff-management endpoints that would call it even exist** (§Non-goals: "membership delete / role change endpoints do not exist yet"). You are building the lock before the door. The danger is not that any single piece is wrong — each is defensible — it is the **compound maintenance surface** for a 50-location SaaS: every future auth change now has to reason about cache TTL, pub/sub liveness, fail-open vs sticky-deny, and legacy `tv`-absent tokens, *simultaneously*. Auth is the one place where compound cleverness ages worst, because the failure mode is "everyone locked out" or "thief stays in," and both are discovered in production. The proposal is genuinely high-craft. My worry is that high-craft in a place that wanted boring is itself a smell.

### Aesthetics / conceptual integrity
The design's own aesthetic ideal — "schema rich, runtime minimal," inert column, flip-a-flag reversibility — is real and admirable, and the proposal invokes it explicitly. But the *runtime* is not minimal: cache + TTL + pub/sub + sticky-deny + dual-path fail semantics is four moving parts where the stated ideal is one. There is a tension between the proposal's self-description ("the boring, proven, pool-safe choice") and its actual surface area. The single most elegant thing in the whole document is the one borrowed line — folding the membership recheck into the refresh query that *already runs*. That's the move that costs nothing and closes the realest gap. The rest is the expensive part.

### Epistemic — the unexamined assumption
The load-bearing unexamined assumption is **"the per-request DB read (Option B) would starve the pool, therefore we must not do it, therefore we need a cache."** But the envelope's own numbers undercut this: 15 req/s today, 150 at 10× growth, against a Supabase pool. A single indexed PK read at 15/s is not pool starvation — the 503 guards exist for *login + order-create bursts*, not for a sub-millisecond owner-auth PK read. The proposal reasons as if Option B's cost is prohibitive at *today's* scale when it is prohibitive only at the *hypothetical 10×*. This is premature optimization wearing a back-of-envelope as armor. (See steel-man §4 — the cheapest correct design may be simpler than A *or* B.)

## 2. ETHICAL-STOP — grounded red lines only

Two grounded stops. Both are **friction, not verdict** — they require a recorded human decision, they do not block a conscious operator, and they do not block forever.

### ETHICAL-STOP 1 — silent-no-op revocation an operator could mistake for "revoked"
**Grounded line:** *soft-confirm-not-a-trap · server-authoritative · UI-tells-the-truth.*
With `OWNER_REVOCATION_ENABLED` default-off and off ⇒ `tv` fully ignored, any "Remove owner" / "Log out everywhere" action shipped while the flag is off **silently no-ops for up to 7 days** while the UI and the DB row both say "removed." That is a security control that fails *silently* and a UI that asserts an effect the server does not deliver. **Required recorded human decision before the staff-management UI ships:** either (a) the revocation action is hard-gated to the same flag and is *not surfaced in the UI* when the flag is off (no button that lies), or (b) shipping the UI flips the flag on as a coupled, non-optional act. A "removed" owner who is still logged in for a week, with no signal to anyone, is the exact shape of harm this red line names.

### ETHICAL-STOP 2 — fail-open must never be presented to a human as "revoked"
**Grounded line:** *server-authoritative · UI-tells-the-truth (no false sense of revocation).*
Fail-open is the *correct engineering choice* — I am not contesting it. The stop is narrower: the **success copy must not out-run the guarantee.** During a Redis+DB double-fault, a force-logout is *accepted but not enforced* for ≤30s, yet the operator/owner will have seen a success state. **Required recorded human decision:** the force-logout / "log out everywhere" confirmation must reflect eventual semantics ("signing out within ~30 seconds") OR the operator runbook (§9) must explicitly document that the *only* synchronously-guaranteed kill is `DELETE FROM auth_refresh_tokens` + waiting out the access exp — and that the tv-bump is best-effort-fast, not instant-guaranteed. The design already knows this (the sticky hard-revoke exists precisely for "known-compromised"); the stop is to make sure the *human-facing words* know it too.

**Not raised as stops** (so the friction stays proportional): fail-open itself (correctly chosen, mirrors courier path, grounded-line-compatible); coarse granularity (a product choice, not a red line); the RLS self-read policy (a correctness item, R1 — the Architect already flagged it; logging out a legitimate owner is *not* what this design does as long as fail-open + the membership predicate are scoped right, which is R6's job, not mine).

## 3. Non-blocking advice (aesthetic / strategic — take or leave)

1. **The 7d access TTL is the real lever, and it's barely examined.** The whole edifice exists to bound a 7-day theft window without regressing the 7d *no-relogin* UX. But the no-relogin UX win comes from the **refresh** lifetime, not the **access** lifetime — the client silently refreshes on 401. **Access TTL and refresh TTL are decoupled.** You can cut the *access* TTL to 24h (or even a few hours) and keep the 7d *refresh* family, and the owner never sees a relogin prompt — the refresh round-trip is exactly the mechanism that already makes 7d-access feel seamless. Cutting access to 24h shrinks the leaked-token window from 7 days to 24h **with zero new infra, zero new column, zero pub/sub, zero fail-open dilemma.** That is the cheapest 7×-improvement available, and it should be priced explicitly against Option A before A is built.

2. **`token_version` vs reusing `courier_sessions`'s shape:** `tv` is the *right primitive for the coarse "sign-out-everywhere / membership-changed" semantics* — it is genuinely simpler than a per-device session table, and the proposal is right to reject B as the hot-path mechanism. Don't second-guess that. The primitive isn't the problem; the *caching machinery around it* is.

3. **If you build A, consider building it without the cache first.** At 15 req/s, an indexed PK read on `users.token_version` through the operational pool is ~1-3ms and will not starve anything you've measured. Ship A as a plain per-request read, gated by the flag, and *add the LRU + pub/sub only if a metric says you need it.* That removes the four most fragile parts (TTL floor reasoning, pub/sub liveness, eviction, the 30s window) from day one. You can always add the cache later — it's a pure optimization. This is the "boring" the proposal claims to want, actually delivered.

4. **Strategic horizon:** the launch trigger is the *first real paid order* (red line). None of this serves that trigger. This is owner-account hygiene for a HIGH audit finding, not a launch blocker. Sequence it *after* launch unless an actual owner-token-leak incident is on the table. Building a bespoke auth-revocation layer pre-launch, for door-less endpoints, is polish wearing a security badge.

## 4. Steel-man of a rejected option (Option D, and a stronger sibling)

**The rejected option, stated at its strongest — "do nothing new; cut access TTL to 24h; lean on the refresh rotation + family reuse-detection that ALREADY exists, accept a 24h revocation gap."**

The proposal rejects D as "regresses the 7d no-relogin UX" — but that rejection conflates *access* TTL with *refresh* TTL, and the live code shows they're decoupled (`local.ts:146` mints access `'7d'`; the *refresh* family at `auth.ts:300` is what survives 7 days). The no-relogin UX is delivered by silent refresh, not by access longevity. So D's "regression" is largely illusory if you cut *access* to 24h while keeping *refresh* at 7d: the owner still never sees a login prompt.

Now price what D-at-24h actually delivers for *this* threat model — a small, no-compliance-mandate, 50-location delivery SaaS:
- **Leaked access token:** valid ≤24h instead of ≤7d. A 7× improvement. The existing reuse-detection (`auth.ts:262-283`, verified working) already kills the *refresh* family on replay — so a thief who tries to *extend* a stolen session past 24h must use the refresh token, which trips family-revoke. The realistic theft window collapses to 24h *for the access token alone* and is actively defended at the refresh boundary.
- **Membership revoke / role change:** already honored at the next refresh (`auth.ts:289`, `status='active'`), so within ≤24h with D — and you can fold the *same* membership recheck the proposal invented into the refresh handler that *already runs the query*, closing it further at zero new infra. That single borrowed line is D's, not A's, to keep.
- **Force-logout a known-compromised owner:** `DELETE FROM auth_refresh_tokens WHERE user_id = $x` (the mechanism *exists today*) + waiting out ≤24h on the access token. Synchronous, no cache, no pub/sub, no fail-open dilemma. For the one acute case, the guaranteed kill is the refresh-family delete the design's own runbook (§9.2) already relies on.

**What D-at-24h costs you that A buys:** the residual 24h access-token window for the *specific* case of a *leaked-but-not-refreshed* access token whose owner you want gone *now*. For a 50-location pre-launch SaaS with no regulator asking, **a 24h gap on a control that today is a 7-day gap, achieved with a one-character config change and no new failure modes, is very plausibly good enough** — and it is *honestly* good enough, with no UI that can lie, no fail-open window to mis-describe, no new operational dependency to monitor. The bespoke revocation layer buys you ~23.99 hours of additional tightening on one sub-case, at the cost of a permanent four-part runtime surface on the most safety-critical path in the system.

D-at-24h is not obviously worse. It may be the more honest, more maintainable, more launch-aligned answer. It deserves to be costed head-to-head in the ADR, not dismissed on a UX-regression claim the code doesn't support.

---

## 5. One open question for the human

**Has an owner token actually leaked — or are we hardening against a finding, not an incident?** The answer changes the right design. If there is a real or imminent owner-token-compromise on the table, build the synchronous kill *today* with tools that already exist (`DELETE FROM auth_refresh_tokens` + cut access TTL to hours) and defer the elegant layer. If this is purely closing a HIGH audit row for a pre-launch product with no incident and no compliance mandate, then the question to put on the record is: *are we spending our scarcest pre-launch craft on a bespoke auth-revocation layer for endpoints that don't exist yet, when a 24h access TTL closes 85% of the same gap with one config line and zero new ways to be locked out or to lie to an operator?* Nobody in the proposal asked whether this should be built **now**, or built **at all at this size** — only how to build it well. It is built very well. The unasked question is whether it should be built.

# REV-2 RE-EXAMINATION — the design pivoted to the steel-manned answer

Date: 2026-06-23 (rev 2). Reviews the rewritten `proposal.md` (Option D′), `resolution.md`, and ADR-0004 rev 2. Grounding re-verified against live source: `auth.ts:288-294` (refresh already runs the memberships SELECT and hard-codes the owner role via `refreshedOwnerClaims` — so P-c is genuinely 0 new queries and H4 is real), `local.ts:59-69` (dev-bypass mints `signDevToken('7d')` and returns **only** an `access_token`, with **no refresh-token issuance**).

**Bottom line up front: the design is now ethically clean and well-sized. I confirm it. Both ETHICAL-STOPs are resolved as red-line stops; the residual that carries forward is a single recorded human gate tied to a UI that does not exist yet, which is exactly where a *future* stop belongs.** This rev does not raise a new ETHICAL-STOP. It records one genuine new footgun (dev-bypass, §R2-new) as non-blocking, and tightens the framing of "honest enough."

## R2.1 — Are the two ETHICAL-STOPs resolved?

**ES-1 (silent-no-op "Remove owner" / "Log out"): RESOLVED at the substrate level for the present design; correctly carried forward as a recorded gate for the future UI.** This is the right disposition, and I confirm it. The reason ES-1 was a *stop* in rev 1 was structural: a default-off enforcement flag meant the read could be ignored, so a button could be wired to a control that silently did nothing. The lean design **has no enforcement flag and no gated read** — there is nothing that can be "off." The logout button (P-b) is wired to a real synchronous `DELETE FROM auth_refresh_tokens`; it cannot no-op silently (it either deletes rows or surfaces an error). The membership-revoke path is enforced at the ≤24h refresh boundary by code that *already runs the query* (`auth.ts:289`). So the trap is not mitigated — it is *structurally absent*. That is the strongest possible resolution: you cannot ship the lying button because the mechanism that made it lie no longer exists.

The residual recorded gate is correct and I want to be precise about why it is **not** a present-tense stop but a real future one. The staff-management "Remove owner" UI does not exist (grep-confirmed in §Non-goals). When it ships, it will assert "this person no longer has access," and the *server-true* statement is "this person loses access at their next refresh, within ≤24h, and loses the in-flight access token at its exp, within ≤24h." Those are close but not identical, and a UI that renders "Removed ✓" with an implied *instant* kick is the same honesty defect ES-1 names — just with a 24h tail instead of a 7d one. So yes: **the future staff-management UI still needs the recorded gate.** R6 / §4-item-2 capture it correctly as friction (recorded decision), not veto, owned by Product + Architect at design time. I confirm that placement. One sharpening for whoever builds that UI: the honest copy is not "Removed" but "Access ending — signs out within 24 hours" until a future per-request layer (DEFERRED) can make it instant. The gate should check the *words*, not just the wiring.

**ES-2 (success copy must not out-run the guarantee): RESOLVED, and improved by the pivot.** In rev 1 this was hard because the guarantee itself was fuzzy (fail-open ≤30s, unbounded under correlated store failure per Breaker C2). The lean design **deletes the fuzz**: the synchronous family-delete is a hard, store-health-independent guarantee, and the access tail is a hard ≤24h deterministic bound. So the honest sentence is now *writable* — "Signed out on this device; other devices sign out within 24 hours" (proposal §9, ES-2). That is a true sentence, which is the whole ask. The runbook requirement (§9: "the only synchronously-guaranteed kill is the refresh-family delete; the access token expires within ≤24h, not instantly") is the operator-facing half and it is correct. ES-2 is downgraded from a human-gated stop to a fix (copy + runbook), which is the right call — there is no longer a fail-open window whose existence a human must consciously accept; there is only a deterministic bound to describe accurately. I confirm.

**Net: both stops resolved. Neither survives into rev 2 as an active ETHICAL-STOP. One recorded human gate (ES-1's descendant) correctly waits at the door of a UI that doesn't exist yet. That is governance working as intended — the friction moved to exactly where the risk will actually materialize, and nowhere else.**

## R2.2 — Is the DEFERRED ≤24h accepted-risk honestly framed and right-sized?

**Yes, on both counts, and this is the part the rewrite gets most right.**

*Honestly framed.* The framing-error correction (rev 1's "rolls forward indefinitely" → "bounded by the 7d refresh family, membership already re-read on refresh") is propagated into proposal §1.2, ADR Context.2, and resolution §5. This matters more than it looks: the residual risk is now stated *against a corrected baseline*. A reader is told the truth — today's window is ≤7d (access) bounded by a 7d refresh family, the change makes it ≤24h, and the thing *not* bought is sub-hour kill of an in-flight, not-yet-refreshed access token. There is no inflated "we closed an infinite hole" claim anywhere. The accepted-risk row (R1) names the exact uncovered sub-case in one sentence and does not hide behind "best practice." That is honest risk accounting.

*Right-sized.* For a pre-launch ~50-location delivery SaaS with no stated compliance mandate, ≤24h is proportionate. The reasoning is sound and I'll add the stakeholder check the proposal implies but doesn't spell out: *who bears the residual ≤24h risk, and is that distribution fair?* The bearer is an owner whose access token both (a) leaked and (b) the operator wants killed *before its ≤24h exp* and (c) the owner hasn't triggered a refresh that would re-derive their (now-revoked) membership. That is a narrow conjunction. And the acute case — known compromise — has a *synchronous* answer today (`DELETE` the family + the ≤24h tail), so the residual is specifically "leaked access token, owner not yet refreshing, operator wants sub-24h kill." For 50 locations of operators managing their own accounts, with no regulator and no PII-broker threat model on the owner surface, paying a permanent four-part runtime auth surface to close that specific 23.99-hour sub-case would be the *less* defensible choice. The promotion triggers (real incident / compliance mandate / observed abuse at volume) are the right re-open conditions, and crucially they are *observable* — not "when we feel nervous." That keeps the deferral from becoming permanent neglect: there is a named tripwire, and the Breaker's C1/C2/C3 are pre-loaded as hard gates on the day it fires. Deferral with a recorded, observable trigger and pre-attached gates is the mature shape of "not now." I confirm the sizing.

## R2.3 — Is the lean design the right answer to "build this at all now?" — and should P-c be trimmed?

**The lean design is the right answer to the open question, yes.** My rev-1 §5 question was "has a token leaked, or are we hardening a finding?" The rewrite does not pretend to answer it (it can't — it's a human fact) but it does the next-best thing: it **routes both answers correctly.** R5 / §4-item-1 records the question for the human and gives a branch — if a real incident is on the table, the synchronous `DELETE` + ≤24h cut is available *today* with existing tools and the full layer promotes per trigger 1; if it's finding-hardening, accept ≤24h and defer. That is exactly the disposition my open question was asking for: don't build the elegant layer speculatively, and don't be helpless if an incident is real. The question stays open *on the record* with an owner. I'm satisfied — an unanswered human question that is correctly recorded and branch-planned is resolved *as far as Counsel can resolve it.*

**Should even P-a/b/c be trimmed — just P-a 24h + P-b logout, defer P-c?** I considered this seriously because "minimum viable" is my own bias and the question rightly presses it. **My answer: no — keep all three. Trimming P-c would be a false economy, and here is the steel-man of trimming it and why it fails.**

*Steel-man of "defer P-c too":* P-c is a behaviour change on the refresh hot-ish path; the membership-revoke it enforces only matters once the staff-management UI (which writes those membership changes) exists, and that UI doesn't exist yet; therefore P-c hardens a path nothing currently drives, and YAGNI says defer it with the UI. This is a coherent argument and it is why the question is worth asking.

*Why it fails:* P-c does **not** only serve the not-yet-built UI. It closes **H4, a present-tense correctness bug**, verified live at `auth.ts:294`: the refresh handler hard-codes `refreshedOwnerClaims(...)` (role owner), so *today*, a user whose owner membership is gone but who still holds any active membership re-mints a fresh **owner** token on refresh. The membership *status* re-read at `auth.ts:289` already exists and already runs — but its result only steers `activeLocationId`, never the role. So the role is stale-by-construction right now. That is a live privilege-roll-forward, independent of any future UI: any path that today downgrades or removes an owner's owner-membership (and `status` transitions already exist in the schema) leaves the refresh able to re-mint owner. P-c is **0 new queries** (the SELECT runs regardless), it is the single most elegant move in the whole design (the "borrowed line" I praised in rev-1 §1 Aesthetics), and it *tightens tenant isolation today* by re-deriving `activeLocationId` from a live membership instead of carrying a baked one. Deferring it would leave a known privilege-escalation-on-refresh in place to save a handler tightening that costs nothing. **Keep P-c.** If anything, P-c is the highest-value-per-byte change in the set and P-a is the headline; P-b is the one with the most *new* surface (a route + client wiring) — but it closes the 404-logout-button dishonesty (H3), which is itself an honesty defect worth fixing, so I would not trim it either. **All three earn their place. The set is correctly minimal — none of the three is polish, each closes a distinct real defect (leak window / lying logout button / privilege roll-forward).** That is the rare case where "minimum viable" and the proposed scope coincide. I confirm the scope.

## R2.4 — New ethical/strategic concern introduced by the rewrite

One genuine new footgun, **non-blocking** (not an ETHICAL-STOP — no grounded red line is crossed), and one smaller honesty note.

**R2-new (footgun) — dev-bypass at 24h with no refresh family = staging re-login friction every 24h.** The question anticipated this correctly, and live source confirms the mechanism. The dev-bypass path (`local.ts:59-69`) mints `signDevToken(payload, '7d')` and returns **only** `{ access_token, userId, activeLocationId }` — it issues **no refresh token** at all. So the dev-bypass session has **no silent-refresh fallback**: when its access token expires, there is nothing to refresh with, and the operator re-logs-in. Today that's a 7-day cadence (rarely felt). After P-a cuts the dev-bypass mint to 24h (`local.ts:68` is in P-a's 5-site list), **the staging/dev operator gets bounced to the login screen every 24 hours**, with no silent recovery, because the decoupled-TTL trick P-a relies on (*access short, refresh long, silent-refresh covers the gap*) **does not apply to a path that issues no refresh token.** This is a real ergonomic regression for the operator who lives on staging, and it is invisible in the proposal's "silent refresh ⇒ zero relogin prompts" claim — that claim is true for the *real* login paths and *false* for the dev-bypass, precisely because dev-bypass has no family. It is not a security or ethics red line (it's a dev-only convenience path that runs under the dev keypair a prod verifier rejects, per ADR-0003), so I do **not** raise it as a stop. But it is a footgun that will annoy the one person who tests this design the most, and it's the kind of friction that quietly erodes the operator (a Physician concern — operator overload / friction-fatigue). **Recommendation (non-blocking, pick one):** (a) leave the dev-bypass mint at 7d — exclude `local.ts:68` from P-a, since the dev token is already non-prod-valid so its leak-window is irrelevant to the threat model P-a addresses; or (b) accept the 24h re-login on staging as a deliberate cost and *note it in the proposal* so it doesn't surprise. I lean (a): cutting the dev token's TTL buys **no security** (a prod verifier already rejects it) and costs **real daily friction**, so it's pure downside. The proposal should either exclude it or justify including it; right now it includes it silently, and "silently includes a change with only downside" is exactly the kind of thing a re-examine pass exists to catch. *This is the one place the rewrite reached for uniformity ("all 5 mint sites") at the cost of a real ergonomic regression with no offsetting benefit — uniformity is an aesthetic value, but here it's serving consistency-on-paper over the operator's lived experience.*

**Honesty note (minor, non-blocking) — "≤85% of the gap" is a rhetorical number.** The proposal/ADR repeat that the lean design "closes ~85% of the same gap." There is no derivation behind 85% — it's a vibe quantification. It happens to point the right direction (most of the value, near-none of the cost) and I don't contest the *conclusion*, but a made-up percent on an AUTH ADR is the kind of false-precision that a future reader could cite as if it were measured. Recommend softening to "closes the dominant share of the gap (window 7d→≤24h, plus the two correctness bugs) at near-zero surface area" — qualitative and *true* — rather than a fabricated 85%. Small thing; flagging because auth docs get cited verbatim.

**F-N strategic check (confirm, not contest):** normalizing the OAuth/Telegram 30d idle families to 7d (R4) is the right call and *improves honesty* — without it, the headline "bounded by 7d" is false on two login paths. The accepted cost (an owner idle >7d on OAuth/TG re-logs-in) matches the password path already and touches no red line. Confirmed.

## R2.5 — Verdict

The design is now **ethically clean and well-sized.** I say so plainly. The pivot did three things that matter to this office: it **removed** the substrate of both ETHICAL-STOPs rather than mitigating them (the strongest resolution), it **corrected an overstated threat** so the residual risk is accounted honestly against a true baseline, and it **moved the one surviving human gate to exactly where the risk will materialize** (a future UI) instead of carrying speculative friction now.
