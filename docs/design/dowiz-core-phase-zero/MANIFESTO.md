# Dowiz Architecture Manifesto: The Immutable Core

> Persisted verbatim as the INPUT artifact for `PHASE-ZERO.md` (operator-supplied, 2026-07-05).
> The blueprint next to this file is the executable interpretation; where they diverge, the
> blueprint states why.

## 1. Концептуальний стек

Ми відмовляємося від "швидкої розробки" на користь **інженерної безкомпромісності**.

* **Мовна уніфікація (Universal Rust):** Backend (Server-side) + Frontend Logic (WASM) + Logic Core (Shared Crate).
* **Логічний двигун:** Immutable State Machine. Жодної мутації даних. Кожен стан замовлення — це новий об'єкт, породжений функцією переходу.
* **Інтерфейс (Dual-Layer):**
  * *Base:* Astro + Svelte 5 (Signals). Максимальна надійність та SEO.
  * *Intent:* Canvas-based UI (GPU-accelerated via vello або аналог). Для візуалізації потоків та намірів.
* **Комунікація:** Binary serialization (Protobuf або rkyv). Ніякого роздутого JSON там, де потрібна швидкість.
* **Голосовий вхід:** Whisper (Local/Edge). Перетворює звуковий потік на Intent, який "споживає" CoreEngine.

## 2. Архітектурна схема (Core Engine)

* **Input Layer (Intents):** Агенти або Люди генерують Command.
* **The "Law" (Kernel):** Функція transition(State, Command) -> Result<State, Error>. Це єдине місце, де народжується правда.
* **Storage (Immutable Log):** Список усіх переходів (Event Sourcing). Ми не зберігаємо "поточний стан" як основний, ми зберігаємо *історію переходів*. Поточний стан — це лише результат "прокручування" історії.
* **Output Layer (Visuals):** React/Svelte/Canvas підписуються на зміни в CoreEngine і миттєво відмальовують результат.

## 3. Ключові особливості (Security & Reliability)

* **Детермінізм:** На вхід подаємо ту саму послідовність намірів — отримуємо той самий фінальний стан. Ніяких "галюцинацій" через мережеві затримки.
* **Запобіжники (Safety Corridors):** У кожному переході try_advance вбудована перевірка: час приготування, наявність кур'єра, ціна. Якщо перехід порушує "Коридор безпеки" — система блокується EngineError.
* **PQC-Ready:** Архітектура передбачає впровадження Kyber/Dilithium на рівні "Command Layer", щоб кожна дія була підписана криптографічно.

## 4. Етапи розвитку

1. **Phase Zero: Dowiz-Core.** Створення Shared Crate. Опис OrderStatus, UserCommand та EngineError. Написання юніт-тестів на всі "заборонені переходи".
2. **Phase One: The Hybrid UI.** Реалізація класичного фронтенду, який використовує dowiz-core. Кожна кнопка "Замовити" викликає функцію переходу стану.
3. **Phase Two: The Canvas Terminal.** Візуалізація станів через Canvas. Перетворення "статусів" на "рухомі об'єкти".
4. **Phase Three: Agent Integration.** Підключення Whisper та автономних агентів. Вони стають "джерелами подій", які рівноправні з людськими командами.

## 5. Головні правила (Hard Rules)

* **Ніяких side-effects у Core:** Функція переходу стану *не повинна* дзвонити в базу даних, надсилати імейли чи писати в логи. Вона має бути "математично чистою".
* **UI — це просто дзеркало:** Інтерфейс ніколи не має власних "істин". Він лише відображає те, що йому каже CoreEngine. Якщо на екрані щось не те — значить, у ядрі помилка.
* **Fail-Fast:** Система повинна "вмирати" миттєво при найменшій логічній невідповідності. Це краще, ніж працювати в "стані корупції даних".

## Grand Design (levels)

* **Рівень 1 (MVP):** Детермінований Rust-ядро, гібридний UI (Web/Canvas).
* **Рівень 2 (Scaling):** Криптографічна ідентичність вузлів (PQC/Dilithium).
* **Рівень 3 (Autonomy):** Mesh-протоколи; формальна верифікація (Coq) інваріантів ядра.

## Конституційний фільтр для будь-якого нового рішення

* Чи це рішення порушує імутабельність?
* Чи це рішення можна перевірити детерміновано?
* Чи це рішення веде до "галюцинацій" агента?

Якщо відповіді "Ні" — рішення правильне.
