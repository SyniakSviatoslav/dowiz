## 1. Bridge / cross-chain failures — the master pattern is *concentrated verification authority*

Five canonical bridge losses, each root-caused:

- **Ronin (Mar 2022, ~$624M).** Ronin's bridge validated withdrawals with a 5-of-9 validator multisig. Lazarus-group phishing compromised 4 Sky Mavis keys; the 5th was the Axie DAO key that Sky Mavis had been temporarily authorized to sign with during a load spike **and that authorization was never revoked**. Root cause: *validator key compromise of a small set*, compounded by a stale privileged grant. The forged withdrawals were cryptographically "valid." ([Halborn](https://www.halborn.com/blog/post/explained-the-ronin-hack-march-2022), [Roninchain post-mortem](https://roninchain.com/blog/posts/back-to-building-ronin-security-breach-6513cc78a5edc1001b03c364), [Elliptic](https://www.elliptic.co/blog/540-million-stolen-from-the-ronin-defi-bridge))
- **Wormhole (Feb 2022, ~$326M / 120k wETH).** The Solana verifier used the deprecated `load_instruction_at`, which did **not** check that the account it read was the real `Sysvar1nstructions`. The attacker passed a forged sysvar account, so the "was Secp256k1 called first?" check passed while **no guardian signature was ever actually verified**. Root cause: *signature-verification bypass* via unchecked account identity. ([Halborn](https://www.halborn.com/blog/post/explained-the-wormhole-hack-february-2022), [Kudelski](https://kudelskisecurity.com/research/quick-analysis-of-the-wormhole-attack))
- **Nomad (Aug 2022, ~$190M).** A routine upgrade seeded the trusted-root mapping with `0x00`. Solidity returns `0x00` for any absent mapping key, so *every* message proved valid against the zero root. Root cause: *acceptance of an uninitialized/default trust anchor*. Because the exploit needed no skill — copy the tx, swap the recipient — it became a "frenzied free-for-all" of hundreds of copycats. ([Immunefi](https://medium.com/immunefi/hack-analysis-nomad-bridge-august-2022-5aa63d53814a), [Mandiant](https://cloud.google.com/blog/topics/threat-intelligence/dissecting-nomad-bridge-hack))
- **Poly Network (Aug 2021, ~$611M).** The privileged `EthCrossChainData` contract (which holds the keeper public keys) could be called *through* `EthCrossChainManager`'s `verifyHeaderAndExecuteTx`. The attacker crafted a cross-chain payload that called `putCurEpochConPubKeyBytes`, **replacing the keeper set with their own key** — no private key was leaked. Root cause: *a verification path that could reach and rewrite its own trust root*. ([SlowMist](https://slowmist.medium.com/the-root-cause-of-poly-network-being-hacked-ec2ee1b0c68f), [Mudit Gupta](https://mudit.blog/poly-network-largest-crypto-hack/))
- **Multichain (Jul 2023, ~$126M+, then $103M more).** Nominally an MPC key-sharding system; in reality **all shards were controlled by the CEO**. When Chinese police detained him and seized his devices, the "decentralized" keys moved funds to unknown addresses. Root cause: *centralized custody dressed as MPC decentralization*. ([Chainalysis](https://www.chainalysis.com/blog/multichain-exploit-july-2023/), [DL News](https://www.dlnews.com/articles/defi/multichain-halts-use-of-crypto-bridge-saying-ceo-arrested/))

**The precise common structural pattern:** every cross-chain bridge collapses trust into a **single verification authority** — a small validator quorum (Ronin), one code path that stands in for signature checking (Wormhole), one trust-anchor value (Nomad), one privileged contract reachable from user input (Poly), or one human holding all key material (Multichain). Cross-chain messaging has *no shared source of truth*, so the bridge invents one, and that invented authority becomes the whole attack surface. Bridges account for the largest single category of crypto theft precisely because they minted this pattern at scale.

**(a) Structural prevention.** Do not let any actor mint on the strength of a *quorum you can compromise* or a *code path you can spoof*. Require the message itself to carry an unforgeable, independently-verifiable authorization (a real signature over canonical bytes) and verify it against a trust anchor that **no message can rewrite**. Keep the trust root out of the mutation path (Poly), reject default/zero anchors (Nomad), verify the identity of the thing you verify against (Wormhole), and never let a temporary grant outlive its purpose (Ronin).

**(b) dowiz/bebop2 posture.** *Largely defended, by construction, but one thing to hold.* There is **no bridge mint semantics** here — the mesh is not wrapping assets across heterogeneous chains, so the Ronin/Wormhole/Nomad "forge a valid-looking mint" class does not exist as such. Capability-based trust means authority travels as a **signed capability the holder presents**, not a validator-set vote an operator can seize — which structurally removes the Ronin quorum-capture and Multichain custody-concentration classes. ML-DSA-65 over canonical TLV (already shipped) is the "real signature over canonical bytes" defense against Wormhole-style verification bypass — *provided the kernel verifies the signature itself and never accepts a "was-it-signed?" proxy flag*. **New explicit decision needed:** the Poly Network lesson — *a capability must never be able to authorize rewriting the capability-issuance root*. When the mesh adds cross-mesh or bridge "flexible bridges," the delegation graph must be acyclic with respect to trust-anchor mutation. Write that as an invariant now.

## 2. Consensus / finality failures — "fast finality" is a promise that breaks under partition or cartel

- **51% / deep reorg (PoW).** Ethereum Classic was 51%-attacked **three times in one month (Aug 2020)**; one reorg rewrote **>7,000 blocks (~two days)**, enabling exchange double-spends of ~$5.6M and ~$1.68M. The economics were trivial — renting hashpower to out-mine a minority chain cost as little as ~$5,332/hour. Root cause: *finality was probabilistic and cheap to overturn* for a chain not securing the majority of an algorithm's hashpower. ([CoinDesk](https://www.coindesk.com/markets/2020/08/29/ethereum-classic-hit-by-third-51-attack-in-a-month))
- **PoS reorg under incomplete rollout.** Ethereum's beacon chain saw a **7-block reorg (May 2022)** when "proposer boost" was rolled out as a soft-fork at each validator's own pace: a late block plus a split view over who had the new fork-choice rule reorganized 7 blocks. It did *not* lose finality, but it showed fast-finality fork-choice is fragile to **heterogeneous client behavior**. ([Barnabé Monnot](https://barnabe.substack.com/p/pos-ethereum-reorg))
- **Loss of finality under load.** On **11–12 May 2023**, Ethereum mainnet failed to finalize for nine consecutive epochs — long enough to trigger the inactivity leak. Cause: a flood of attestations for old epochs forced Prysm nodes (a super-majority client) into expensive state replay, causing OOM/CPU exhaustion — a **client-monoculture correlated failure**, not a cryptographic break. Arbitrum's own sequencer had a finality post-mortem the same incident. ([Offchain Labs post-mortem](https://medium.com/offchainlabs/post-mortem-report-ethereum-mainnet-finality-05-11-2023-95e271dfd8b2), [EtherWorld](https://etherworld.co/2023/05/13/non-finality-inactivity-leak-but-ethereum-didnt-stop/))
- **Liveness monoculture (non-BFT).** Solana **halted repeatedly in 2022** — Candy-Machine spam (6M req/s, OOM crashes, ~1.5h restart), the June durable-nonce bug (>4h), the Sept duplicate-block fork-selection bug — each resolved only by **validators coordinating an off-chain restart**. Root cause: *a single client implementation and a design that traded partition-tolerance for throughput*. ([Helius](https://www.helius.dev/blog/solana-outages-complete-history))
- **Validator cartelization.** Lido crossed **~1/3 of all staked ETH** — the threshold at which a pool can affect liveness/finality — prompting Buterin and others to argue single protocols should **self-limit** (proposed caps of 15–25%), because pooled MEV/timing/censorship advantages create a rich-get-richer cartel. Root cause: *economies of scale in staking pull toward concentration*, and concentration re-creates the trusted-quorum risk of §1 inside "decentralized" consensus. ([Lido research](https://research.lido.fi/t/should-lido-on-ethereum-be-limited-to-some-fixed-of-stake/2225), [GSR](https://www.gsr.io/reports/the-great-staking-debate-lido-dominance-and-ethereum/))

The "nothing at stake" and **long-range** problems are the theoretical backdrop: in naive PoS, signing every fork is costless, and an attacker who acquires old keys can rewrite deep history — which is why real PoS chains bolt on slashing and **weak subjectivity checkpoints** rather than relying on the protocol alone.

**(a) Structural prevention.** Do not promise a single global "fast finality." Make finality *local and explicit*: an event is final for a participant when they hold the signatures they require, under a checkpoint they trust. Avoid client monoculture (the 2023 finality stall and every Solana halt are correlated-implementation failures). Cap any single operator's share of consensus-relevant weight.

**(b) dowiz/bebop2 posture.** *Structurally strong, with a caveat to make explicit.* The mesh is **local-first / event-sourced**, not a single global ledger competing for one canonical head — so global-reorg and global-halt classes largely do not apply; **fork-freedom (M11)** treats divergence as legitimate rather than an emergency to reconcile, which is the correct inversion of the ETC/Solana model. **Zero-dependency single kernel is a double-edged fact:** it eliminates the *dependency* monoculture but means the mesh *is* a client monoculture — a consensus-relevant bug affects every node identically (the exact shape of the 2023 Prysm stall and Solana halts). **New explicit decision needed:** because global consensus is avoided, the residual risk is *local* — define what "final" means for a mesh event (which signatures, under which weak-subjectivity/checkpoint assumption) so no component silently treats an unfinalized event as settled, and keep degrade-closed as the partition response.

## 3. Oracle manipulation — naive on-chain price feeds are atomically forgeable

The mechanism is identical across incidents: **borrow a flash loan → move a thin-liquidity price the protocol reads → transact against the moved price → repay — all in one atomic transaction**, so there is no inventory risk and no time for arbitrage to correct.

- **bZx (Feb 2020, ~$954k across two attacks).** Borrowed ETH on dYdX, manipulated a Uniswap WBTC/ETH price that bZx used as an oracle (tripling the effective rate), and borrowed against the distorted collateral value. First real flash-loan oracle attack. ([QuadrigaInitiative](https://www.quadrigainitiative.com/casestudy/bzxdecentralizedlendingplatformattack.php), [CoinDesk](https://www.coindesk.com/tech/2020/02/19/everything-you-ever-wanted-to-know-about-the-defi-flash-loan-attack))
- **Harvest Finance (Oct 2020, ~$24–34M; triggered a ~$570M bank run).** 32 loops in 7 minutes on a $50M flash loan: swap USDC→USDT on Curve to inflate USDT's *apparent* price, deposit into the vault at the inflated exchange rate, reverse the swap, withdraw at the restored rate — ~$500k/cycle. The vault used **the live Curve pool balance as its price source**. ([SlowMist](https://slowmist.medium.com/slow-mist-analysis-of-harvest-finances-hacked-event-63450b49e6a5))
- **Mango Markets (Oct 2022, ~$116M).** Not even a flash loan — Eisenberg opened a large MNGO-perp position, then spent ~$4M spot-buying MNGO across the three exchanges feeding Mango's oracle, spiking the oracle price ~13x, and **withdrew $116M against the unrealized "profit."** The thin spot market *was* the oracle. (His criminal convictions were later vacated in May 2025 — the manipulation is undisputed; the legal characterization is what changed.) ([CFTC](https://www.cftc.gov/PressRoom/PressReleases/8647-23), [Chainalysis](https://www.chainalysis.com/blog/oracle-manipulation-attacks-rising/))

**Why naive on-chain oracles are structurally vulnerable:** a spot price on a single venue reflects only that venue's *current* liquidity depth, and within one atomic transaction an attacker with borrowed capital can make depth arbitrarily thin relative to their trade. The oracle reports a *manipulable instantaneous quantity* as if it were a market truth.

**(a) Structural prevention.** Never read a manipulable instantaneous price. Use **time-weighted averages (TWAP)** to raise the capital cost across many blocks, **medianize independent sources**, and cap per-block moves — but the deeper lesson is *don't put a manipulable market price on the critical path of a value decision at all* unless you must.

**(b) dowiz/bebop2 posture.** *Depends entirely on whether the mesh introduces price feeds.* The current architecture has **no on-chain price oracle** — nothing reads a manipulable market number to authorize an irreversible action, so this class is absent *today*. **This becomes a live risk the moment any "resource market," "trading bridge," or "market-based micro-negotiation" (see §5) is added.** Explicit decision required *before* any pricing enters the mesh: any price a decision depends on must be non-atomically-manipulable (TWAP/medianized/externally-attested), or the decision must not depend on a market price. Flag this as a gating precondition for the trading-bridge feature.

## 4. Reputation / scoring failures — independent confirmation that NO-COURIER-SCORING is correct

This section is *not* reopening a decision — it is documenting why the existing **capability-based-trust, never-reputation** rule (and its CI-enforced NO-COURIER-SCORING gate, which rejects reputation as an "echo chamber" risk) is well-founded. The academic record on adversarial decentralized reputation is unusually clear, and it says: reputation scores are gameable in ways that are *provably* unfixable by tuning.

- **Sybil attack (Douceur, IPTPS 2002).** One adversary forges many pseudonymous identities to gain disproportionate influence over any reputation aggregate. In a permissionless setting identities are cheap, so the attacker simply out-numbers honest raters. ([Douceur 2002](https://cnitarot.github.io/papers/p2p-reputation-survey.pdf))
- **Impossibility result (Cheng & Friedman, 2005).** **No *symmetric* reputation function — the class that includes PageRank/EigenTrust-style scores — can be Sybil-proof.** Every such mechanism admits a beneficial Sybil attack. This is a theorem, not an engineering gap: you cannot tune your way out of it. ([survey](https://cnitarot.github.io/papers/p2p-reputation-survey.pdf))
- **Whitewashing / cheap pseudonyms (Friedman & Resnick, 2001).** When identities are cheap and replaceable, a bad actor simply **abandons a soured identity and rejoins fresh**. Friedman & Resnick prove the *best achievable* equilibrium is one where newcomers "pay dues" (are distrusted by default) — meaning **any open reputation system must punish all newcomers to defend against whitewashers**, an inherent social cost. Their only clean fixes are **entry fees** (which exclude legitimate low-resource newcomers) or **free-but-unreplaceable pseudonyms** (i.e., cryptographic identity — not a score). ([Friedman & Resnick 2001](https://onlinelibrary.wiley.com/doi/abs/10.1111/j.1430-9134.2001.00173.x))
- **Collusive inflation.** Beyond Sybils, malicious *collectives* mutually up-rate to manufacture standing (ballot-stuffing / self-promotion), a known-open problem even in EigenTrust-class systems. ([survey](https://cnitarot.github.io/papers/p2p-reputation-survey.pdf))

**The synthesis:** reputation systems fail in adversarial decentralized settings via **(1)** Sybil numeric override, **(2)** whitewashing to erase history, **(3)** collusive inflation — and the impossibility result means (1) is unsolvable for the whole symmetric-scoring family. The honest options collapse to either *exclude newcomers* or *use unforgeable cryptographic identity instead of a score*. That second option **is** capability-based trust.

**(a) Structural prevention.** Replace "how much do others rate this actor?" (a gameable score) with "does this actor hold a valid, unforgeable capability for this action?" (a checkable fact). Capabilities are Sybil-immune because minting a thousand identities does not mint a thousand capabilities; whitewashing gains nothing because a fresh identity holds no capabilities; collusion cannot inflate a boolean grant.

**(b) dowiz/bebop2 posture.** *Already structurally decided and correct — this research is confirmation, not a new debate.* Capability-based trust + per-actor PQ identity is precisely the "unforgeable, unreplaceable identity + explicit grant" that Friedman & Resnick point to as the escape from cheap-pseudonym gaming, and it sidesteps the Cheng–Friedman impossibility entirely because it is *not a symmetric scoring function*. **Cite this section wherever the NO-COURIER-SCORING gate needs external justification.** The only vigilance: never let a "convenience" feature (recommendations, sorting, ETA-weighting by past behavior) smuggle a reputation score in through the back door — the gate exists to catch exactly that drift.

## 5. MEV & auction/negotiation failures — transparency + adversaries + latency = extraction

*Directly relevant because a prior (unadopted) draft proposed "market-based micro-negotiation" and "rapid-fire auctions" for mesh agent coordination.* The real-world record on putting auctions/bidding into a transparent adversarial low-latency setting is bad.

- **Front-running / sandwiching (Flash Boys 2.0, Daian et al. 2019).** When pending transactions are visible before they finalize, an adversary reorders around them: buy ahead of a victim's swap, let the victim move the price, sell into it. The paper formalized MEV and **priority-gas auctions**, and warned these ordering games cause **consensus instability**. ([Flash Boys 2.0](https://www.semanticscholar.org/paper/Flash-Boys-2.0:-Frontrunning,-Transaction-and-in-Daian-Goldfeder/393ab84a86631d5fda128c3aac0bf5476da07791))
- **Scale.** In 2022 mid-year, ~2,000 sandwich attacks/day extracted >$2M/month; **72,351 victim addresses lost ~$87.7M** in a single half-year window to 168 attacker addresses. This is not a tail risk; it is the steady-state of transparent on-chain auctions. ([eigenphi](https://eigenphi.substack.com/p/flash-boys-gain-everybodys-pain))
- **Governance-as-auction failure (Beanstalk, Apr 2022, ~$182M).** A market-based governance mechanism let voting power be *bought instantly*: the attacker flash-loaned >$1B, acquired a super-majority of governance tokens, and used the **`emergencyCommit`** path to execute a malicious proposal draining the protocol in one transaction. Root cause: *a market/auction mechanism (here, token-weighted voting) with no defense against instantaneous, borrowed, transient participation*. ([Immunefi](https://medium.com/immunefi/hack-analysis-beanstalk-governance-attack-april-2022-f42788fc821e))
- **Censorship as auction manipulation (OFAC/MEV-Boost).** After the Tornado Cash sanctions, **~86% of MEV-Boost blocks (Sept 2022)** flowed through relays intending to censor certain transactions — the block-production "auction" became a **censorship chokepoint**. Whoever controls ordering controls inclusion. ([MEV Watch](https://www.mevwatch.info/), [Wahrstätter](https://blog.toniwahrstaetter.com/builder-censorship.html))

**What actually goes wrong:** an open auction in an adversarial low-latency setting is *not* a price-discovery mechanism, it is an **information-leak + reordering game**. Seeing others' pending bids/actions before finalization lets the fastest/best-connected participant extract value deterministically (sandwiching), buy transient control (Beanstalk), or selectively exclude (censorship). "Everyone bids their true value and the market clears efficiently" is false the moment participants can observe and reorder each other.

**(a) Structural prevention.** If coordination must be market-like: **hide bids until commit** (commit-reveal or threshold/encrypted mempools), use **sealed-bid / batch-uniform-clearing** auctions (no in-flight visibility, no ordering advantage), and forbid **instantaneous transient stake** from carrying decision weight (time-locks, non-flash-loanable participation). Better still, prefer mechanisms that **don't require an auction at all** — direct capability-authorized assignment over speculative bidding.

**(b) dowiz/bebop2 posture.** *Needs a new explicit decision if the micro-negotiation/auction idea advances — do not adopt it as drafted.* Nothing in capability-trust or PQ-crypto prevents an auction mechanism from leaking bids or being sandwiched; those defenses operate at the *mechanism-design* layer, which the draft did not specify. The Beanstalk lesson maps almost exactly onto "market-based micro-negotiation": **any mesh mechanism where transient, acquirable weight decides an outcome is Beanstalk-shaped.** Recommendation for the synthesis: treat "rapid-fire auctions" as **rejected-unless-sealed** — require commit-reveal or sealed-bid batch clearing and non-transient participation weight as a *precondition of even prototyping it*, and prefer capability-authorized direct assignment. The low-latency, observable, adversarial setting is the worst case for auctions, and the mesh's real-time ambition puts it squarely there.

## 6. Optimistic execution / fraud-proof failures — "speculate then challenge" has a weak real-world record

*Directly relevant because a prior draft proposed "speculative execution with local challenges."* Here is how optimistic rollups actually behave, and where they break.

**Mechanism.** A sequencer posts state assumed valid; anyone may, within a **challenge window (~7 days)**, submit a fraud proof to revert it. Safety rests on **at least one honest, funded, uncensored challenger being online**.

**Real failure modes:**

- **Fraud proofs mostly did not exist / were permissioned.** For years after Arbitrum launched (May 2021), **no optimistic rollup ran a permissionless fraud-proof system on mainnet**; challengers were a **whitelist of a few addresses**. The security story was aspirational. ([Gate research](https://www.gate.com/learn/articles/the-state-of-fraud-proofs-in-ethereum-l2s/4402))
- **The first-ever successful mainnet fraud proof was April 2024.** Kroma's challenge (12 challengers contested a bad output; one called `proveFault` and deleted it) is documented as **the first successful fraud-proof challenge in the history of Ethereum rollups — ~3 years after the first optimistic rollup shipped.** That the *first* real use took three years is the finding: the challenge mechanism was almost never exercised in anger. ([Gate research](https://www.gate.io/post/status/7328415))
- **Censorship of challengers.** If the sequencer/builder can refuse to include the fraud-proof transaction, the challenge window can be run out. Academic work ("Economic Censorship Games in Fraud Proofs") shows the 7-day window is calibrated against censorship that a hard-fork social response can survive — i.e., *safety depends on an off-chain social backstop*, not the protocol alone. ([arXiv 2502.20334](https://arxiv.org/pdf/2502.20334))
- **Economic non-viability.** "Hollow Victory" shows malicious proposers can exploit dispute-game **validator incentives** so that the **cost of proving fraud exceeds the reward** — a rational challenger stays home, and invalid state finalizes. ([arXiv 2504.05094](https://arxiv.org/html/2504.05094v1))
- **The liveness assumption is the whole game.** If every verifier is offline or colluding, invalid transactions finalize. The challenge-window length is a raw **safety-vs-liveness tradeoff**: long windows are safer but freeze funds/decisions; short windows finalize fast but give challengers less time.

**(a) Structural prevention.** Do not rely on "someone will challenge." Make the honest-verifier assumption *explicit and funded*: guaranteed challenger inclusion (censorship-resistant submission), a **reward strictly greater than proving cost**, permissionless (not whitelisted) challengers, and — where feasible — **validity proofs instead of fraud proofs** so correctness is enforced *before* finalization rather than hopefully-after. Size the challenge window to the actual cost of an unchallenged bad state.

**(b) dowiz/bebop2 posture.** *Needs a new explicit decision if "speculative execution with local challenges" advances.* Capability-trust and PQ-crypto do not supply a challenger, an inclusion guarantee, or a reward-vs-cost calculus — those are the exact things every optimistic system got wrong. The mesh's **degrade-closed** posture is actually in tension with optimistic execution (which is *degrade-open*: assume-valid-until-challenged). The honest reading of the record: **optimistic/challenge-based patterns have a poor track record — three years to a first real challenge, documented censorship and economic-non-viability failure modes.** For the synthesis, recommend the mesh either (i) prefer **verify-before-commit** (the kernel already has a drift-gate that rejects unstable events *pre-persist* — that is validity-first, the opposite of optimistic) or (ii) if speculation is truly needed for latency, specify funded/incentivized/censorship-resistant challenges up front and treat the honest-verifier assumption as a first-class, budgeted requirement — not a footnote.

## 7. Tokenomics / incentive collapses — "everyone acts rationally, therefore stable" fails under stress

- **Terra/Luna (May 2022, >$40B destroyed).** UST held its peg via a mint/burn arbitrage: 1 UST always redeemable for $1 of newly-minted LUNA and vice-versa. Under a coordinated withdrawal (large Anchor exits + Curve sell-offs), UST de-pegged, arbitrageurs burned UST to mint LUNA, **LUNA supply exploded from ~340M to >6.5T tokens in days**, crashing LUNA, which destroyed the collateral value backing UST — a self-reinforcing **death spiral**. Root cause: *the stabilizing mechanism was reflexive — the thing meant to defend the peg (minting LUNA) was exactly what destroyed the peg under a run.* The whole edifice rested on **confidence**, subsidized by Anchor's unsustainable ~20% yield. ([Harvard/Corpgov](https://corpgov.law.harvard.edu/2023/05/22/anatomy-of-a-run-the-terra-luna-crash/), [Fed FEDS Note](https://www.federalreserve.gov/econres/notes/feds-notes/runs-on-algorithmic-stablecoins-evidence-from-iron-titan-and-steel-20220602.html))
- **Iron Finance / TITAN (Jun 2021, ~$2B — "crypto's first large-scale bank run").** IRON was 75% USDC + 25% TITAN. When whales pulled liquidity and dumped TITAN, the partial-collateral arbitrage minted TITAN into a falling market, the same reflexive spiral ran, and a contract issue temporarily blocked redemptions — accelerating the panic. Root cause: *partial-algorithmic collateral + a run + a redemption mechanism that fed the fire.* ([Fed FEDS Note](https://www.federalreserve.gov/econres/notes/feds-notes/runs-on-algorithmic-stablecoins-evidence-from-iron-titan-and-steel-20220602.html), [Finematics](https://finematics.com/bank-run-in-defi-iron-finance-explained/))

**The shared failure:** the design assumed **rational actors under normal conditions** and had **no defense against reflexivity** — the equilibrium was stable only while everyone believed it was stable, and the corrective mechanism amplified the shock instead of damping it. "Rational agents ⇒ stable" ignores that a bank run *is* the rational individual response once confidence tips.

**(a) Structural prevention.** Any incentive/resource mechanism must be **stress-tested for reflexivity and runs**, not just steady-state rationality: does the stabilizer amplify or damp a shock? Avoid mechanisms whose only backstop is confidence. Prefer **fully-collateralized / capability-gated** resource claims over reflexive mint/burn. Model the mechanism under an *adversary who wants it to fail*, and under correlated exit.

**(b) dowiz/bebop2 posture.** *Absent today; becomes a live risk only if a token/resource-market/incentive component is added.* The mesh has no algorithmic stablecoin, no reflexive mint/burn, no confidence-backed peg — so this class does not exist in the current design. If any resource-market or incentive layer is introduced later, this is a **new explicit decision**: no reflexive stabilizer, no confidence-only backstop, mandatory run/reflexivity stress-testing. The existing **capability-based** stance is again the safer primitive — a capability is a fully-backed grant, not a confidence-backed claim.

## Cross-cutting synthesis

Two meta-patterns unify all seven sections:

1. **Concentrated authority masquerading as decentralization** (bridge quorums, Multichain's fake MPC, Lido cartelization, whitelisted rollup challengers, censoring MEV relays). The failure is always: trust that *looks* distributed collapses to a small, seizable, or spoofable point.
2. **Manipulable instantaneous state on a critical path** (oracle spot prices, transparent pending auctions, uninitialized trust roots, reflexive token supply). The failure is always: a value the system reads as *truth* can be moved cheaply within the window that matters.

dowiz/bebop2's already-decided architecture — **capability-based trust, per-actor PQ identity, zero-dep native kernel, local-first fork-freedom, degrade-closed** — structurally defends against the *first* meta-pattern very well (authority is a held, unforgeable capability, not a seizable quorum or a gameable score) and against several concrete classes outright (§1 bridge-mint, §4 reputation, and — because they're simply absent — §3 oracles and §7 tokenomics *today*). It is **weakest where new features would introduce the second meta-pattern**: any market/auction (§5), any price feed (§3), any speculative-execution/challenge scheme (§6), and any incentive/resource token (§7) would each import a manipulable-instantaneous-state failure class that current primitives do *not* cover, and would each require an explicit, up-front design decision — sealed-bid/commit-reveal, non-atomically-manipulable pricing, funded-censorship-resistant challenges, and reflexivity stress-testing respectively — before they can be safely built.
