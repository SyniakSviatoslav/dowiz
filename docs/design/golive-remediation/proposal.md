# Go/No-Go Pre-Launch Remediation — Design Proposal

> Status: DRAFT for review. Design-only. No production code in this document.
> Axis: engineering truth — does it work, does it hold under the live failure modes we already proved.
> Scope: 6 fixes + 1 data task. Forward-only. Boring > novel. Schema rich, runtime minimal.

Companion ADR: `docs/adr/ADR-golive-remediation.md`.

## 0. Live-audit corrections (ground truth before design)

Two audit claims were imprecise against the source. Stating them correctly so the fixes target the real defect, not the headline.

- **#1 "order.timeout has NO consumer."** A consumer *does* exist: `apps/worker/src/handlers.ts:15` registers `queue.work(QUEUE_NAMES.ORDER_TIMEOUT, …)` and runs the guarded `UPDATE … WHERE status='PENDING'`. That worker is a real deployed process group: `fly.toml:9` defines `worker = "dist/worker/index.cjs"` with its own VM (`fly.toml:29-31`). So the precise defects are:
  1. **Notification gap** — the handler publishes to the live MessageBus (`order:{id}`, `location:{id}:dashboard`) but never enqueues the `order.timeout_cancelled` notification. The event type is fully wired downstream (`event-registry.ts:59`, `render.ts:102`, `locales.ts:95/169/243`, `notifications/workers/index.ts:550`) yet nothing emits it. The owner gets no "auto-cancelled" Telegram message.
  2. **Enqueue-reliability doubt** — `courier/assignments.ts:349-355` records as known-debt that enqueue to an *unregistered* pg-boss queue silently no-ops, and asserts "order.timeout has the same fate." This is **contradicted** by `1790000000011_pgboss-bootstrap-schema.ts:62`, which pre-creates `order.timeout` under the migration superuser. So on any DB migrated past 0011 the queue IS registered and enqueue works. The debt comment is stale for this queue. **Verification step required** (see #1 operability) rather than a blind rewrite.
  3. **RLS posture of the worker UPDATE (verified — `app.current_tenant` would be a no-op).** The worker UPDATE works today **because role `deliveryos_api_user` has `BYPASSRLS`** (`packages/db/migrations/1780691681296_ops-location-alerts-policy.ts:8` — `ALTER ROLE deliveryos_api_user BYPASSRLS`). The `orders` policy keys on **`app.user_id`** (via `app_member_location_ids()`: `1780310074262_orders.ts:82-84` → `1780310071220_core-identity.ts:76-79` selects memberships `WHERE user_id = app_current_user()`), **NOT** on `app.current_tenant`. **No policy in the tree reads `app.current_tenant`.** Therefore the earlier "set `app.current_tenant` in the worker" idea is a **verified no-op** and is DROPPED: it changes no row visibility, and worse it would falsely imply the worker is RLS-correct. The real latent exposure: a future `NOBYPASSRLS` flip (the stated security direction) would zero-match the sweep — a background job has **no member `user_id`**, so `app_member_location_ids()` returns empty and `UPDATE … WHERE status='PENDING'` silently affects 0 rows. That flip-point is documented as a re-validation gate (R4), not patched with the no-op GUC.

  Net: #1 is **not** "write a missing worker," and **not** "add a tenant GUC." It is "close the notification gap + keep the guarded WHERE-UPDATE as the cancel authority (NOT an imported domain guard — see below) + add a reconciliation sweep as the real safety net + make a lost job detectable < 1 min via a `pgboss.job` throughput query (NOT the heartbeat)." Smaller in scope, larger in robustness, forward-only.

  **Round-3 corrections (breaker RC1/RC2/RH1 — they REMOVE both round-2 migrations and re-home detection):**
  - **RC1 — drop the sweep index.** `CREATE INDEX CONCURRENTLY` is incompatible with the single-transaction migration runner (`1790000000011:18-30`); a non-concurrent build locks `orders`. The index was only to make a perf claim true; the pilot seq-scans cheaply (§5). Removed → "Schema changes: NONE."
  - **RC2 — drop the notification UNIQUE.** `notification_outbox_audit` is an attempt-log (bare `INSERT`, N rows per target, NULLs — `audit.ts:31-44`, `workers/index.ts:387-444`); a UNIQUE there crashes the handler on `23505`. Removed. Dedup stays at the notification layer (in-process `Set` + `singletonKey`); cross-restart duplicate accepted (R12). The zero-notification hole stays closed by emitting on *observing* CANCELLED-by-timeout (the R2 design change, kept).
  - **RH1 — detection co-located with recovery.** The `pgboss.job` overdue query had no executor (`LivenessChecker` reads only the heartbeat; `ReconciliationWorker` is removed). It now runs inside the SAME 1-min standalone sweep job in `apps/api` — one live executor for both recover and detect.

  **Round-2 corrections (breaker C2/C3/H2 — these reshape #1's mechanism, not its goal):**
  - **C2 — detection cannot ride the heartbeat.** The worker writes ONE heartbeat row per VM keyed by `FLY_MACHINE_ID`, `job_name` NULL (`apps/worker/src/heartbeat.ts:9,28-31`); `CRITICAL_WORKERS` matches by string name (`liveness-checker.ts:11`). No row will ever carry `worker_id='order.timeout'`, so "add order.timeout to the heartbeat-watched set" is unimplementable. Lost-job detection is therefore a **`pgboss.job` overdue query** (§3 #1 detectability), confirmed SELECT-able under the runtime role (`reconciliation.ts:237-246` already queries `pgboss.job`). Heartbeat stays as coarse VM-liveness only.
  - **C3 — the reconciliation worker is REMOVED in prod** (`server.ts:382`). The sweep CANNOT ride it. It mounts as its OWN pg-boss schedule in `apps/api` (beside the live `NotificationWorker`, `server.ts:347,352-354`). Re-enabling `ReconciliationWorker` (which also owns the now-dead O1/O3 detectors, `reconciliation.ts:182-195,237-253`) is tracked separately as R9, not on the launch path.
  - **H2 — `apps/worker` cannot import `@deliveryos/domain`** (`apps/worker/package.json:11-16` deps lack it; `assertTransition` is in `packages/domain`). The timeout-cancel path's authority is the **guarded `WHERE status='PENDING'` UPDATE itself** — `PENDING→CANCELLED` on timeout is one fixed edge; the WHERE *is* the legality check. The earlier "route through `assertTransition`, single transition authority, no logic fork" claim is **dropped as structurally false**; what is enforced single-source is the *identical SQL + emit contract* in handler and sweep, test-asserted (§9), not a shared import.

- **#3 "/health maps pg timeout to degraded."** Confirmed at `health.ts:31-36`: `withTimeout` returns `status:'degraded'` when `err.message === 'timeout'`. A *hung/unreachable* Postgres (the live failure mode — connection-timeout, not a fast error) therefore yields `degraded` + HTTP 200, so external monitors stay green during an outage. The "down → 503" path (`health.ts:64-70`, `300-302`) already exists and is correct; it just never fires for the timeout case. Also confirmed: the response serializes raw pg result objects (`pgResult` carries `data: result`, i.e. node-postgres rows incl. type-catalog internals) and the rich `/health` is unauthenticated (`config: { rateLimit: false }`, no auth). So #3 is two narrow changes, not a rewrite.

These corrections do not reduce severity; they retarget the fixes.

## 1. Problem statement + non-goals

**Problem.** Six defects gate launch; one data task blocks realistic staging validation. The two CRITICAL items (order auto-cancel reliability, DB-pool wedge) are correctness/availability risks proven live. The HIGH items (health-mask, checkout fallback + dev-mock leak) and the medium/quick items (privacy notice, ESM test-collection) are launch-blockers for trust, safety, and CI.

**Non-goals.**
- No raw `CREATE` on the `pgboss` schema (migration 0009 revokes schema-level DDL). **But pg-boss's own managed `boss.createQueue` + `boss.schedule` DO work at runtime** — proven live by `anonymizer-retention.ts:26` and `reconciliation.ts:35`. So #1's reconciliation sweep can have its own scheduled queue without a forbidden raw `CREATE`. The earlier "no new queues at all" framing was too strong.
- No move off the monolith, no service split, no Redis-for-idempotency (idempotency stays in Postgres per existing `idempotency_keys`).
- No rewrite of the order state machine, the money model, or the RLS model. We *preserve* integer-money, RLS FORCE, JWT RS256.
- No new customer-facing data collection. The privacy notice (#5) describes existing collection; it adds no field.
- Data task copies **menu/location/branding only** — zero customer PII crosses prod→staging.

## 2. Back-of-envelope (pilot scale + the #2 pool math)

**Pilot scale.** 1 active pilot tenant (`sushi-durres` — the live slug per `packages/db/migrations/1790000000021_rename-slug-pizza-roma-to-sushi-durres.ts:5`; NOT `durres-sushi`, which exists nowhere in source — breaker C1), launch target ~3-10 locations. Realistic peak: **~2-5 orders/min per location** at dinner rush; say **10 locations × 3 orders/min = 30 orders/min ≈ 0.5 orders/sec** system-wide at the optimistic ceiling. This is small. The pool wedge is therefore NOT a throughput problem — it is a **connection-hold / head-of-line** problem: a handful of concurrent slow POSTs starve every other tenant's reads.

**Connection budget (the real constraint).** Operational pool `max: 8` (`packages/db/src/index.ts:20`), `connectionTimeoutMillis: 5000`. This 8 is shared by *all* hot-path API work: menu reads, location/info, `/health`'s ~11 queries, order writes, OTP, etc. Workers use a separate session pool (`max: 3`) + pg-boss (`max: 4`); migrations use the session/migration URL. So the operational ceiling is **8 concurrent checked-out connections** before `connectionTimeoutMillis` starts rejecting with 500s.

**Reads stay inside the transaction (corrected — moving them out breaks two invariants).** The original design moved validation reads outside `BEGIN`. That is DROPPED: (1) `menu_version` is folded into `request_hash` (`orders.ts:194,197`), the idempotency key — reading it on a different snapshot than the write decouples the key from the order it identifies; (2) the price snapshot rests on the **in-tx MVCC snapshot**, not a lock — the `:376` "FOR UPDATE … authoritative lock" comment is **false** (the query at `:378-382` has no `FOR UPDATE`), so moving the price read out re-introduces the very TOCTOU the in-tx read avoids. We shorten the hold by **cutting round-trips**, not by relocating reads.

**#2 — connection-hold time, before:**
POST /orders opens `db.connect()` then a single `BEGIN…COMMIT` spanning the full validation+write path. Inside the transaction: product availability/price (`:378-382`, already set-based `= ANY($1)`), modifier reads (`:410-416`, already set-based `= ANY($1)`), **per-item modifier-group reads in a loop (`:430-436`, one round-trip *per item* — the only remaining N-fan-out)**, delivery-tier read, customer upsert, order INSERT, velocity INSERT, N×(order_items INSERT + M×modifier INSERT), idempotency INSERT (`:627`), track-grant INSERT (`:641`), 2× transactional enqueue (`:650`, `:660`). For a 3-item cart that is ~7 writes + **N=3 group-reads** + the set-based reads held inside one transaction occupying **1 of 8** connections.

- Per round-trip RTT to Supabase pooler from `fra`: assume ~3-8 ms. ~15-18 round-trips ⇒ **~55-145 ms of pure RTT**, plus query exec. Under any DB latency spike (the proven trigger) each round-trip inflates and the *whole* transaction holds longer. The per-item group loop makes the held window scale with cart size.
- **Saturation point, before:** at hold-time `H` and arrival rate `λ`, concurrent held connections ≈ `λ × H` (Little's law). With `H ≈ 0.12 s` and a burst of 8 simultaneous checkouts, all 8 connections are held → menu/info/health get `connectionTimeoutMillis` → cross-tenant 500s. The audit's "~8 concurrent orders wedge the pool" is exactly `8 = max`. Non-self-healing because new POSTs keep arriving and re-occupying freed slots.

**#2 — connection-hold time, after (chosen design — batch the per-item loop, keep all reads in-tx):**
The only N-fan-out inside the held window is the `modifier_groups` query per item (`:430-436`). Replace it with **one** set-based query over all `product_id`s (`… JOIN product_modifier_groups WHERE pmg.product_id = ANY($1)`), into a `Map<product_id, groups[]>`. Products and modifiers are already batched. Net: the held-tx round-trip count drops from ~(7 writes + N group-reads + 2 set-reads) to ~(7 writes + 1 group-read + 2 set-reads) — **constant in cart size**, not linear.

> **Correctness guard (breaker H3 — this is NOT just a round-trip change).** The current loop validates min/max-select **per line-item** against that line's `groupCounts` (`orders.ts:438,456,466-476`), and the SAME `product_id` can appear in multiple cart lines (e.g. "1 Margherita no cheese" + "1 Margherita extra cheese"). The batched `Map<product_id, groups[]>` MUST be **broadcast — re-read by every line-item that references the product**, never consumed/drained once. If the partition were consumed once, the 2nd+ line of a repeated product would skip group validation and an invalid modifier selection would pass. So each line-item, inside its existing `for (const gRow of …)` validation, looks up `partition.get(item.product_id)` independently. A DoD test (§9) covers a 2-line same-product cart with one invalid line — it MUST 422.

- Held round-trips, after: ~10 regardless of item count (vs ~7 + N before). For a 3-item cart, ~15 → ~10 round-trips; the *linear* term is gone, so the win grows with cart size — a 6-item cart goes ~18 → ~10.
- Hold-time `H' ≈ 0.04-0.08 s` for a typical cart — roughly **35-55% of before**, and crucially **flat** as carts get larger. By Little's law the same burst of 8 checkouts now holds ~half as long, so concurrent-orders-to-saturation roughly **doubles** (8 → ~16) for the same `connectionTimeoutMillis`, and the held window no longer balloons with a big order.
- Idempotency + money invariants **preserved**: the `menu_version`-in-hash and the price snapshot stay on the *same* in-tx MVCC snapshot as the write — batching changes round-trip count, not snapshot coherence.
- **Plus `statement_timeout` (bound revised — breaker M2):** operational pool already sets `SET statement_timeout = '10s'` per connection (`index.ts:30`). Keep it. ADD a tighter per-transaction `SET LOCAL statement_timeout` at `BEGIN` of the write path, sized **~4.5s — just inside `connectionTimeoutMillis=5000` (`index.ts:22`) with margin**, NOT the round-1 ≈4s. Back-of-envelope sizing input (M2): a legitimate worst-case cart (~15 line-items × ~3 modifiers ≈ 60 inserts × ~50ms under a spike ≈ 3s) must complete *with* headroom — a 4s bound had none and would abort the valid large order. The bound exists to kill a *wedged* write (one stuck ≫5s), not a slow-but-progressing one. Turning a wedge into a fast 5xx for that one order, not a pool-wide stall. Failure-first lever. **Double-order is closed at the idempotency layer regardless:** a retry of the same cart re-derives the same content+`menu_version` `request_hash` (`orders.ts:194,197`) → 23505 collision, no second order — provided the client does NOT inject a random nonce into the key (made an explicit invariant + test, §6 / §9).
- **Optional real lock + comment fix:** if a hard guarantee against in-window availability flips is wanted, add a genuine `SELECT id FROM products WHERE id = ANY($1) AND location_id=$2 FOR SHARE` at the top of the write tx (one round-trip). Regardless of that choice, **correct the lying `:376` comment** — a comment claiming a lock that does not exist is a latent trap.

**Pool sizing note.** Do NOT bump `max` blindly — Supavisor transaction-mode pooler has its own ceiling and the free-tier connection budget is monitored (`free_tier_snapshots.connections_pct`). The fix is shorter holds, not more connections. If a bump is ever needed it must be back-of-enveloped against `max(operational 8 + session 3 + pgboss 4 + worker pool + migrations)` vs the Supabase connection cap, and gated behind the free-tier watch. **Accepted for launch: keep `max: 8`, win via hold-time.**

## 3. Options (≥2) for the CRITICAL items

### #1 Order auto-cancel reliability

**Decision: A + B together (B is NOT deferred — it is the real safety net).**

**Part A — close the gap on the existing `apps/worker` consumer.**
Concept: *transactional outbox already exists* (the per-order `order.timeout` job is enqueued inside the order tx via the `db:` option, `orders.ts:650-654`); the consumer exists (`handlers.ts:15`); we (a) make the handler enqueue `order.timeout_cancelled` when it observes the order in CANCELLED-by-timeout state (NOT gated on its own `rowCount>0` — breaker C4), reusing the `NOTIFY_TELEGRAM_SEND` outbox pattern (`orders.ts:660-668`) with `singletonKey: order.timeout_cancelled:{orderId}` + the notification layer's in-process LRU `Set` for dedup (breaker RC2: NO UNIQUE on the attempt-log; residual cross-restart duplicate is accepted pilot risk R12); (b) **keep the guarded `WHERE status='PENDING'` UPDATE (`handlers.ts:26-30`) as the cancel authority** — `PENDING→CANCELLED` on timeout is one fixed edge, so the WHERE *is* the legality check; **NOT** routed through `assertTransition`, because `apps/worker` cannot import `@deliveryos/domain` (breaker H2 — `apps/worker/package.json:11-16` lacks the dep). The audit row (`order_status_history`, actor `system:timeout`, `handlers.ts:40-43`) is kept. **No `app.current_tenant` GUC** — verified no-op (see §0).
- Tradeoffs: + smallest diff, reuses proven machinery. + worker isolation keeps sweeps off the API VM. + no new cross-package dep-edge (H2). + no schema change (RC1/RC2 removed both round-2 migrations). − the notification emit is *post-COMMIT* (R2): a worker crash between COMMIT and emit loses *that* attempt — recovered by Part B's sweep, which re-attempts the emit (deduped in-process by the LRU `Set`).
- Concept name: **transactional outbox + at-least-once consumer + guarded WHERE-UPDATE as transition authority + observe-CANCELLED emit (notification-layer dedup)**.

**Part B — reconciliation sweep (the actual safety net, MANDATORY).**
Concept: a short-interval scheduled job (`boss.createQueue` + `boss.schedule`, proven runnable — `anonymizer-retention.ts:26`) running the **same SQL contract** guarded UPDATE + the same observe-CANCELLED notification emit as Part A: `UPDATE orders SET status='CANCELLED', timeout_at=NULL WHERE status='PENDING' AND timeout_at < now() RETURNING id, location_id`. Catches orders whose per-order `order.timeout` job was lost or whose consumer died — without depending on per-order delivery. **1-minute cadence, mounted as its OWN pg-boss schedule in `apps/api`** (beside the live `NotificationWorker`, `server.ts:347,352-354`) — **NOT** as a sub-check on the `ReconciliationWorker`, which is **removed in prod** (`server.ts:382`, breaker C3) and **NOT** the nightly `0 3 * * *` slot (too slow for the detectability gate). **No backing index — breaker RC1: the round-2 `orders_pending_timeout_idx` is removed** (a `CONCURRENTLY` build is incompatible with the single-transaction migration runner; a non-concurrent build locks `orders`). At pilot scale (≈150 PENDING rows worst-case, low-thousands total `orders`) the 60s `WHERE status='PENDING' AND timeout_at<now()` is a sub-millisecond seq scan off the hot path — H1 is a non-issue at pilot (§5). Runs under `BYPASSRLS` today; R4 marks it as the re-validation point for any future `NOBYPASSRLS` flip.
- Tradeoffs: + self-healing, kills the "lost job" fear directly. + same SQL+emit contract as A (single *specified* contract, test-enforced — not a shared import, H2). + rides only live, deployed machinery (no dependency on the removed recon worker, C3). + no schema change (RC1 removed the index). − a periodic seq scan (negligible at pilot scale — re-index at 10× volume, R1).
- Concept name: **reconciliation sweep / poller-as-safety-net (standalone schedule)**.

**Lost-job detectability < 1 min — the SAME standalone sweep job ALSO runs a `pgboss.job` overdue count (breakers C2 + RH1).** `ops_worker_heartbeat` proves a VM breathes; it is keyed by `FLY_MACHINE_ID` with `job_name` NULL (`heartbeat.ts:9,28-31`) and `CRITICAL_WORKERS` matches by string name (`liveness-checker.ts:11`) — so it can NEVER bind to the `order.timeout` queue (C2). The round-2 fix moved detection to a `pgboss.job` overdue query but left it **without an executor** — `LivenessChecker` reads only `ops_worker_heartbeat` (`liveness-checker.ts:46-54`) and the `ReconciliationWorker` that owned the SELECT is removed (`server.ts:382`, RH1). **Fix: co-locate detection in the recovery sweep.** The same 1-minute standalone sweep job in `apps/api`, on each tick, ALSO runs `SELECT count(*) FROM pgboss.job WHERE name='order.timeout' AND state IN ('created','active') AND start_after < now()` (confirmed SELECT-able under the runtime role — `reconciliation.ts:237-246`) and logs/emits it as an ops counter, warning when overdue > 0. Detection and recovery share one live executor — the detector cannot lose its host because it rides the same job that fixes the problem. The heartbeat stays only as coarse "is the worker VM up" signal. (O1/O3 detectors remain deferred — R9.)

**Decision rationale:** Counsel's steel-man is adopted — the #5 retention promise already commits us to trusting one reliable scheduled sweep, so the marginal cost of trusting a second (cancel) sweep is low, and it removes the dependence on per-order job delivery (the exact failure the stale `assignments.ts:349-355` debt comment feared). A+B ship together; the Go gate verifies both.

### #2 DB pool / transaction hold

**Option A — Batch the per-item reads inside the transaction + bound the hold (CHOSEN).**
Concept: **fewer round-trips, same snapshot.** Keep all reads inside `BEGIN…COMMIT` (idempotency + money snapshot demand it — see §2 corrected) but collapse the only N-fan-out — the per-item `modifier_groups` loop (`orders.ts:430-436`) — into one set-based query (`WHERE pmg.product_id = ANY($1)`) into a `Map<product_id, groups[]>` **broadcast per line-item, never consumed once** (breaker H3 — see §2 correctness guard). Products (`:378-382`) and modifiers (`:410-416`) are already batched. Add `SET LOCAL statement_timeout` (~4.5s, just inside `connectionTimeoutMillis=5000`, sized for the ~60-insert worst case — breaker M2) on the write path. Optionally add a real `FOR SHARE` on product rows and fix the lying `:376` comment.
- Tradeoffs: + cuts held round-trips from ~7+N to ~10-flat, so the held window stops scaling with cart size (~halved for a 3-item cart, more for larger). + **preserves idempotency** — `menu_version`-in-`request_hash` (`:194,197`) and the `idempotency_keys` INSERT + 23505 race handling stay on the same in-tx snapshot, unchanged. + preserves integer-money (arithmetic byte-identical) and the price snapshot's MVCC coherence. + RLS FORCE / tenant scoping untouched. − the in-window TOCTOU shrinks to intra-tx only (R1), already covered by the `is_available` re-check (`:386-388`) on the same snapshot; optional `FOR SHARE` closes it hard.
- Concept name: **short transaction via round-trip batching + bounded hold + server-authoritative MVCC snapshot**.

**Option B (REJECTED as primary) — move reads outside `BEGIN`.**
Concept: read-then-write on the pool, write set in a tiny tx.
- Tradeoffs: + would also shorten the hold. − **breaks idempotency**: `menu_version` is in the hash, so reading it on a different snapshot decouples the key from the order. − **breaks the money snapshot**: the price read would land on a different MVCC snapshot than the INSERT (there is no lock — the `:376` "FOR UPDATE" comment is false), widening TOCTOU. Rejected: it trades two correctness invariants for a hold-time win we get more cheaply via batching.

**Option C (REJECTED) — raise `max`.** Spends the scarce Supavisor/free-tier connection budget and makes a wedge *bigger* (more connections to exhaust). Treats the symptom.

**Decision:** **A.** Win via fewer round-trips on the same snapshot, plus the failure-first `SET LOCAL statement_timeout`. No reads moved out, no pool bump.

## 4. Per-item designs (the four non-CRITICAL fixes)

### #3 Health truthfulness + no driver-internal leak
- **down vs degraded (exact change):** the bug is at `health.ts:31` — `if (err.message === 'timeout' || treatErrorAsDegraded)` returns `degraded` **unconditionally on timeout, ignoring the flag**. A hung Postgres (connection-timeout, not a fast error) therefore yields `degraded`+200 and monitors stay green during an outage. Fix narrowly: the **pg check specifically must classify a timeout as `down`**. Do NOT change the generic path for the soft checks — give the pg invocation (`:60-63`) an explicit `treatTimeoutAsDown` flag (or a dedicated pg classifier) that maps *both* timeout and fast-error to `down`, evaluated *before* the `err.message==='timeout'` short-circuit. Soft checks (telegram/r2/messageBus) keep `degraded` on failure so a blip never 503s the app or triggers a Fly restart (the `/livez` vs `/health` split, `health.ts:54-56`, preserved). Once the pg check yields `down`, the existing `pgResult.status==='down' → 503` path (`:64-70`) fires correctly — no change there.
- **No raw driver internals — drop `data` AND `detail` (breaker L2).** `withTimeout` stuffs `data: result` (`health.ts:28`) — the node-postgres result object (rows incl. `oid`/`dataTypeID`/type-catalog) — AND, on the down/degraded paths, `detail: \`${label}: ${err.message}\`` (`:35,38`) which surfaces raw pg driver text (host, role, SQLSTATE fragments) — into the response, and `/health` is unauthenticated (`config:{rateLimit:false}`). The public shape becomes `{ status, timestamp, checks: { <name>: { status, latencyMs } } }`. Drop BOTH `data` and `detail` from the public payload (round-1 dropped only `data`, leaving the L2 leak). The internal richness (worker entries, free-tier %s, `detail` diagnostic text) stays behind auth or moves to a separate authenticated `/health/detail`; the unauthenticated `/health` returns only name/status/latency.
- **#3's 503 is for an EXTERNAL monitor, not Fly (breaker M1).** Fly reads `/livez` (`fly.toml:21`), so the `/health` 503 changes nothing Fly does. Its production observability value depends on an external uptime monitor hitting `/health` — see §7, launch act R10 (Owner). The leak-fix above stands regardless of whether a monitor is wired.
- Verifiable: E2E asserts `/health` returns 503 + `status:"unhealthy"` when pg is unreachable (simulated), and that the JSON body contains no `oid`/`dataTypeID`/raw-row keys AND no `detail` key / `err.message`-shaped substring.

### #4 Checkout failure fallback + dev-mock containment
- **Restaurant phone fallback on ALL non-422 errors (breaker H4):** on order-place failure (the `catch` at `CheckoutPage.tsx:437-450`), surface a "Call the restaurant" affordance using `location.phone` for **any error that is not a 422 business-validation error** — i.e. 5xx, network, timeout. Round-1 special-cased only `422 MIN_ORDER_NOT_MET` (`:440`); but under #2's `statement_timeout` abort the *most likely* failure is a fast 5xx, which would otherwise fall to the bare toast at `:448` with no out-of-band path. 422s (min-order, modifier) carry an actionable message the customer can fix → no phone CTA needed. Source: the **existing** public route `GET /api/public/locations/:slug/fallback-config` (`routes/public/fallback-config.ts`) which returns `{ phone, showPhoneOnError, showPhoneOnOffline }`. Show the phone only when `showPhoneOnError !== false`. **Fetch this config once on page MOUNT and cache it in component state (breaker H4)** — the catch reads cached state, so the CTA does NOT depend on an additional network call succeeding under the same DB/load pressure that caused the failure. If the mount fetch failed, the catch fail-softs to the generic toast. Cart is already preserved (`clearCart()` only runs on success / dev-mock) — keep it; the fallback must NOT clear the cart.

## 5. Data / migrations (forward-only, RLS FORCE preserved)

**Schema changes required: NONE (round-2's two additive indexes are BOTH REMOVED — breakers RC1 + RC2).** Round-2 added two migrations (M-1 partial index on `orders`, M-2 partial UNIQUE on `notification_outbox_audit`). Both proved broken against the live tree and are **dropped**; "Schema changes: NONE" is true again — and is the simpler, more honest state.
- **(M-1 REMOVED — breaker RC1.)** The proposed `CREATE INDEX CONCURRENTLY orders_pending_timeout_idx` is structurally incompatible with the migration runner: node-pg-migrate runs ALL migrations in ONE transaction (`1790000000011_pgboss-bootstrap-schema.ts:18-30` documents this verbatim; the only escape is 0011's own `pgm.noTransaction()`+manual COMMIT), and `CREATE INDEX CONCURRENTLY` throws `25001` inside a transaction → boot fails. A *non*-concurrent `CREATE INDEX` would instead take an `AccessExclusiveLock` on `orders` for the whole batch — exactly the write-lock the round-2 "online-safe" claim denied. The index was only ever shipped to make round-1's "indexed on `(status,timeout_at)`" claim *true*; the pilot does not need it. Back-of-envelope: PENDING rows at any instant ≈ `0.5 orders/s × ~5 min ≈ 150`; the whole `orders` table is low-thousands for months. A 60s seq-scan `UPDATE … WHERE status='PENDING' AND timeout_at<now()` over a few-thousand-row table is sub-millisecond and runs off the hot path (standalone schedule). H1 is a **non-issue at pilot**, not a CRITICAL — there is no correctness/availability break from the seq scan, only a perf claim we now simply do not make (R11). Re-add via its own `pgm.noTransaction()` migration if/when volume hits 10× (R1 trigger).
- **(M-2 REMOVED — breaker RC2.)** The proposed partial UNIQUE on `notification_outbox_audit (event,target_id)` was a mistake: that table is a **delivery-attempt journal**, not an event ledger. `writeAudit` (`apps/api/src/notifications/audit.ts:31-44`) is a **bare `INSERT` with NO `ON CONFLICT`**, and one logical send writes `sending`→`delivered`/`failed` (2+ rows per target per attempt, `workers/index.ts:387-444`). A UNIQUE there makes the 2nd legit row throw `23505` and crash the handler — failing the notification it meant to dedup. `target_id` is nullable and written NULL (`audit.ts:37`, `workers/index.ts:470-473`) so NULL rows never dedup, and `target_id` = `owner_notification_targets` id (`workers/index.ts:391`), not the order id — the dedup key was semantically wrong. Removed entirely. Dedup for `order.timeout_cancelled` is handled at the notification layer for the pilot (in-process LRU `Set` `notifications/workers/index.ts:57` + `singletonKey`), which works within the single web VM (`fly.toml:15` `auto_stop_machines=false`); the only residual is a duplicate owner Telegram if an api **restart** lands in the ~60s emit-gap — an accepted pilot risk (R12). If a 2nd web instance is ever added, the durable path is a SEPARATE dedup **table** (`event_id PK`), NOT a UNIQUE on the attempt-log.

Existing columns already present (no migration): `locations.retention_days` (`1780421100060`), `locations.fallback_config` + `locations.phone` (`1780421100062`, exposed by `routes/public/fallback-config.ts`); `order.timeout` queue registered (`1790000000011:62`); `order.timeout_cancelled` event wired (`event-registry.ts:59`, render/locales); `idempotency_keys`, `order_status_history`, `orders.timeout_at` all present.

With both migrations removed, the launch carries **no schema change at all** — RLS `ENABLE`+`FORCE` untouched, integer-money arithmetic untouched (no money code moves in #2; reads are batched, arithmetic is byte-identical), JWT RS256 untouched. Forward-only is satisfied by construction (there is no migration to apply). The round-1 "no migrations" claim — briefly false in round 2 — is true again (R11).

**The one design decision with a data contract:** #1's worker UPDATE RLS posture — **resolved, no schema change at all** (the round-2 perf/dedup indexes are removed; this was never an RLS change either). The worker + reconciliation sweep run under `deliveryos_api_user`/`BYPASSRLS` (`1780691681296:8`) and are correct *today*. ## 6. Consistency + idempotency

- **POST /orders idempotency unchanged — and reads STAY in-tx to keep it that way.** The `idempotency_keys` INSERT (`orders.ts:627`) + 23505-race handling stay inside the write tx; the key is written atomically with the order. Critically, `menu_version` is folded into `request_hash` (`:194,197`), so the key's meaning is bound to the menu version observed *on the order's snapshot* — that read is NOT moved out (Option B rejected). Integer-money arithmetic untouched. **Retry double-order guard (breaker M2):** because the key is derived purely from cart content + `menu_version`, a retry of the same cart re-derives the same `request_hash` → 23505 collision → the prior order is returned, no second row. This holds ONLY if the client does NOT mix a random nonce into the key — made an explicit invariant + DoD test (§9 #4). This closes the M2 "statement_timeout aborts a valid large order → client retries → double-order" vector at the idempotency layer, independent of the timeout bound.
- **#1 idempotent cancel — guarded WHERE is the authority; emit on observing-CANCELLED, dedup at the notification layer (breakers C4 + H2 + RC2).** Both the handler (`handlers.ts:26-30`) and the standalone reconciliation sweep run the **same SQL contract** `UPDATE … WHERE id=$1 AND status='PENDING'` (rowCount 0 ⇒ no-op) — the guarded WHERE *is* the single transition authority for this one fixed `PENDING→CANCELLED` edge. **NOT** routed through `assertTransition`: `apps/worker` has no `@deliveryos/domain` dep (H2), and asserting one hardcoded edge adds a cross-package dependency for zero added safety. The `order.timeout_cancelled` emit is **NOT** gated on the emitter's own `rowCount>0` — that made the sweep silent in the crash-recovery case (handler commits then dies pre-emit → sweep's UPDATE finds CANCELLED → rowCount 0 → zero notification, C4). Instead, **whichever path observes the order in CANCELLED-by-timeout state attempts the emit** — this closes the zero-notification hole regardless of dedup substrate. Dedup is at the notification layer: `singletonKey: order.timeout_cancelled:{orderId}` (dedups in-flight pg-boss jobs) + the per-process in-memory LRU `Set` (`notifications/workers/index.ts:57`), which dedups correctly **within the single web VM** (`fly.toml:15` `auto_stop_machines=false`). The round-2 partial UNIQUE on `notification_outbox_audit` is **REMOVED** (breaker RC2 — that table is an attempt-log written N rows per target with NULLs; a UNIQUE there crashes the handler on `23505`). Net at pilot: exactly-once **within the process** (handler+sweep both run in `apps/api`'s NotificationWorker), never zero. The only residual is a duplicate owner Telegram if an api restart lands in the ~60s gap between two emits — an accepted single-instance pilot risk (R12); the durable upgrade, if a 2nd web instance is ever added, is a SEPARATE dedup **table** keyed on `event_id`, never a UNIQUE on the attempt-log.
- **#2 TOCTOU — intra-tx only (smaller than before).** Because reads stay in-tx, the only window is intra-transaction on a single MVCC snapshot. Server stays authoritative: prices snapshotted into `order_items` from the in-tx read; `is_available` re-checked on the same snapshot (`:386-388`); order records `menu_version` + `request_hash`. For a hard guarantee add a single `SELECT id FROM products WHERE id = ANY($1) AND location_id=$2 FOR SHARE` at the top of the write tx (one round-trip, preserves the batched short-hold). Default: snapshot + re-check; window is intra-tx (R1).

## 10-DATA. Data task — seed staging `demo` from prod `sushi-durres` (no PII)

**Goal:** make staging (`dowiz-staging`, Fly Postgres `dowiz-staging-db`) carry a realistic `demo` tenant mirroring prod `sushi-durres`'s **menu + location config + branding** — and nothing else. (Source slug is `sushi-durres` per `1790000000021:5` — breaker C1; the earlier `durres-sushi` was a typo that would 404 every read.)

**Boundary (hard):** prod = Supabase (read source); staging demo DB = Fly Postgres reachable via `flyctl proxy`. **No customer rows, no orders, no PII** cross the boundary.

**Mechanism — read/transform/load via the public surface (CHOSEN):**
1. **Read (prod, no DB creds needed):** pull `sushi-durres`'s menu + branding from the **public** APIs — the SSR public menu (`/s/sushi-durres` data / `/api/public/...`), `GET /api/public/theme/sushi-durres` (branding colors + logo), `GET /api/public/locations/sushi-durres/fallback-config` (phone/fallback). These return menu/location/branding only — they structurally cannot return customer PII. This is the safest source: it reuses the same read path customers use, so anything it exposes is already public.
2. **Transform:** remap to a new `demo` tenant — new `location_id`, `slug='demo'`, keep categories/products/modifiers/prices/branding. Integer-money preserved on copy (no re-derivation). **Scrub ALL free-form identifying fields, not just the top-level phone (breaker M3):** the public phone → placeholder demo number; AND scan/placeholder the `fallback_config` JSON (it is owner-authored free-form — `fallback-config.ts:25-28` returns `config.phone` from it, so it can carry a personal phone/email/name); AND the branding `name`/contact fields → demo placeholders. Nothing owner-identifying may survive the transform.
3. **Load (staging):** via `flyctl proxy` to `dowiz-staging-db`, insert under the migration/session role into the `demo` tenant with RLS-correct `location_id` stamping. **Idempotent loader — conflict target is `slug='demo'` (breaker M3):** there is no stable external key across prod→staging (`location_id`s differ across DBs), so the only natural key is the demo slug. On re-run the loader `UPDATE`s the existing `demo` location and upserts categories/products/modifiers keyed by their stable in-tenant identifier (e.g. `ON CONFLICT (location_id, name)` / sort within the demo tenant) — never blind-inserting, so re-runs refresh rather than duplicate.

**Alternative — direct prod DB read (REJECTED as default):** reading prod Supabase directly with a read-only role would also work and is faster for bulk, but it (a) needs prod DB creds in the loader's environment and (b) puts the loader one query away from customer tables. The public-API path has no such blast radius. Use direct-DB only if the public API lacks a needed config field, and then read *only* `locations`/`products`/`modifiers`/branding tables with an explicit column allow-list — never `customers`/`orders`.

**Verification (four assertions — breaker C1/M3):** after load, `/s/demo` on staging renders the `sushi-durres`-shaped menu + branding; and SQL checks on staging assert: (1) `SELECT count(*) FROM customers WHERE location_id = <demo>` = 0; (2) `orders` = 0; (3) `staging demo.phone ≠ prod sushi-durres.phone` (the scrub at .2 is *proven*, not assumed); (4) the `demo` location's `fallback_config` JSON and branding `name`/contact fields contain **no** prod phone/email/owner-name string (the free-form-JSON PII vector M3 raised). All four green = no PII landed.
