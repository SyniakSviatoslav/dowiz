## CRITICAL

### C1 — B-DATA / B-OPS · The prod slug is `sushi-durres`, not `durres-sushi`. The entire data task reads a 404.
- The proposal and ADR every time name the source tenant `durres-sushi` (proposal §2, §10-DATA.1-3; ADR Decision 7; resolution #5 data-task). The live tree shows the slug was renamed to **`sushi-durres`**: `packages/db/migrations/1790000000021_rename-slug-pizza-roma-to-sushi-durres.ts:5` — `UPDATE locations SET slug = 'sushi-durres' WHERE slug = 'pizza-roma';` and `1790000000024_update-sushi-durres-settings.ts:13,22` keys on `WHERE slug = 'sushi-durres'`. There is **no** `durres-sushi` literal anywhere in real source (grep of `packages/db/migrations apps/api/src apps/web/src e2e` = empty).
- Break: the chosen read mechanism is the public API path — `GET /s/durres-sushi`, `GET /api/public/theme/durres-sushi`, `GET /api/public/locations/durres-sushi/fallback-config`. `fallback-config.ts:18` does `SELECT … WHERE slug = $1` and `:24` returns 404 on `rowCount === 0`. So every read in the data task returns 404. The loader produces an empty `demo` tenant; the §10-DATA verification (`/s/demo` renders the durres-sushi menu) fails not because of a bug but because the source name is wrong.
- Violated invariant: DoD/verification must reference a real target. The whole §10-DATA mechanism is specified against a non-existent slug.

### C2 — B-OPS · #1's "add the timeout consumer to the heartbeat-watched set" is unimplementable as written: the worker heartbeat has no per-queue identity.
- Proposal §7 #1 and resolution #1(detectability) both say "add `order.timeout` to the heartbeat-watched set so a dead timeout-consumer is observable," mirrored by the `CRITICAL_WORKERS` mechanism. But `CRITICAL_WORKERS` matches by `worker_id` string (`liveness-checker.ts:11` — `'dispatcher,settlement-cron,dwell-monitor,anonymizer-retention'`), and the `apps/worker` process writes its heartbeat with **`worker_id = process.env.FLY_MACHINE_ID || 'local-worker'`** and `job_name` left NULL (`apps/worker/src/heartbeat.ts:9,28-31`). There is exactly one heartbeat row for the whole worker VM, keyed by a random machine id — never the string `order.timeout`, never any queue name.
- Break: you cannot add `order.timeout` to `CRITICAL_WORKERS` — no row will ever have that `worker_id`, so it would report permanently dead (false alarm) or be ignored. The reconciliation `EXPECTED_WORKERS` list (`reconciliation.ts:217`) has the same defect: it lists named in-process workers (`dispatcher`, `anonymizer-retention`, …) but the entire `apps/worker` VM heartbeats under a machine id, so a dead timeout-consumer in `apps/worker` is invisible to name-based liveness. The heartbeat proves "some machine breathed," not "the timeout queue drained" — exactly the gap the proposal claims to close, left open.
- Violated invariant: lost-job detectability < 1 min. The stated detection mechanism does not bind to the thing it claims to detect.

### C3 — B-FAIL / B-OPS · The reconciliation worker — the "safety net" the design leans on — is NOT running in prod. It was removed.
- Resolution #1(e) offers the cancel sweep as "its own pg-boss schedule OR as a sub-check on the reconciliation worker." `server.ts:382` reads: **"Nightly Reconciliation Worker — temporarily removed (esbuild bundle issue). Re-add in separate deploy."** The `ReconciliationWorker` is constructed nowhere; `new ReconciliationWorker` / `reconWorker.start()` appear only inside the file's own doc comment (`reconciliation.ts:8-10`).
- Break #1: the "sub-check on the reconciliation worker" option is a fork into dead infrastructure — that process does not exist in prod. Only the standalone-queue option is real, and that needs new queue machinery (see C5).
- Break #2: the recon worker also owns O1 (stale PENDING > 1h, tolerance 5 — `reconciliation.ts:182-195`) and O3 (failed pg-boss jobs > 10 — `:237-253`). Because it is removed, there is **today no detector at all** for piled-up PENDING orders or a failed `order.timeout` queue. The design's detectability story silently assumes a worker that isn't deployed.
- Violated invariant: failure-first / observability < 1 min. The recovery+detection both ride a process that has been disabled since an unrelated bundle issue.

### C4 — B-CONSIST · The `order.timeout_cancelled` dedup substrate does not deduplicate across processes/restarts → owner gets double "auto-cancelled" Telegram when sweep races the per-order handler.
- The design's anti-duplication claim is layered: `rowCount > 0` guard (only the cancelling worker emits) + `singletonKey: order.timeout_cancelled:{orderId}` + "dedup at the notification layer" (proposal §6, resolution #1b). Each layer leaks:
  1. **`rowCount > 0` does not prevent a double notify across the two cancellers.** Handler and sweep run the *same* guarded `UPDATE … WHERE status='PENDING'`. Under a race both can read PENDING and one wins the row; but if the handler cancels, COMMITs, then dies *before* its post-COMMIT enqueue (the explicitly accepted R2 gap), the sweep's later UPDATE finds status already CANCELLED → `rowCount 0` → **no emit from the sweep either**. Net: zero notification, not one — R2 says "sweep recovers" but the guard makes the sweep silent precisely in the recovery case. Conversely if both manage `rowCount>0` against different snapshots before either commits, both emit.
  2. **`singletonKey` only dedups jobs in `created/active/retry` state.** pg-boss accepts a new job with the same singletonKey once the prior one **completes**. The handler's notify can complete in milliseconds; the sweep's enqueue up to ~60s later is then a fresh accepted job, not a dedup.
  3. **The cross-process dedup is an in-memory LRU `Set` scoped to one process lifetime** (`apps/api/src/notifications/workers/index.ts:56-57` "NX-5: Dedup set for idempotency (in-memory LRU)", key `${event}:${entity_id}:${location_id}` at `:298`, `dedupCache.add` at `:432`). `notification_outbox_audit` does **not** back it: the `ON CONFLICT DO NOTHING` inserts (`:388,416,440`) have nothing to conflict on — the only constraints are the auto `id` PK and **non-unique** indexes on `location_id`, `status`, `(event,target_id)` (`1790000000007_notification-outbox-audit.ts:29-31`). No UNIQUE on `(event,target_id)`. So the audit row never suppresses a duplicate.
- Break: the NotificationWorker runs only in `apps/api` (`server.ts:347-354`). Two api instances, or one api restart between the handler's notify and the sweep's notify, defeats the in-memory `Set` → **two identical "order auto-cancelled" Telegram messages to the owner.** At pilot scale with a single api machine and `auto_stop_machines=false` (`fly.toml:15`) this is mostly masked — but any deploy/restart during a timeout window, or the moment a second instance is ever added, breaks it. The design asserts dedup "at the notification layer"; the live notification layer's dedup is process-local and unconstrained at the DB.
- Violated invariant: idempotent cancel + at-most-once owner notification. The stated dedup is not durable.

## HIGH

### H1 — B-ANTIPATTERN / B-DATA · "Schema changes required: NONE" is false — the sweep's own performance claim requires a new index.
- Proposal §5 states unconditionally "Schema changes required: NONE." Resolution #1(e) and proposal §1 #1 both claim the sweep is "Indexed on `(status, timeout_at)`; cheap at pilot scale." That index does not exist: `orders` has `orders_location_status_idx (location_id, status)` and `orders_location_created_idx (location_id, created_at DESC)` (`1780310074262_orders.ts:44-45`), `timeout_at` is a bare column (`:38`). No `(status, timeout_at)` index anywhere.
- Break: the sweep `UPDATE … WHERE status='PENDING' AND timeout_at < now()` runs **cross-tenant with no `location_id` predicate**, so `(location_id, status)` is not a usable leading-column match — Postgres falls to a seq scan / partial-index miss on the `orders` table every minute. Cheap at pilot scale (a few thousand rows) — but the claim "indexed" is false, and making it true is a `CREATE INDEX`, contradicting "no schema change," which the whole forward-only narrative rests on. Either the index is missing (claim false) or the no-migration invariant is broken (claim false). Both can't hold.
- Violated invariant: design honesty (back-of-envelope claims must match the schema) + the stated "no new migrations" constraint.

### H2 — B-CONSIST / B-ANTIPATTERN · #1 claims "single transition authority, no logic fork" but the handler and the sweep live in two different packages and the worker can't import the domain guard.
- Resolution #1(c) and proposal §3 #1A insist the cancel routes through `assertTransition('PENDING','CANCELLED')` reusing "the single transition authority" and #1(e) says the sweep "reuses the SAME guarded UPDATE + machine routing + notification emit as the handler (no logic fork)." But the per-order handler is in `apps/worker` (`apps/worker/src/handlers.ts`) whose `package.json` deps are exactly `@deliveryos/{config,db,platform,shared-types}` — **no `@deliveryos/domain`**. `assertTransition` lives in `packages/domain/src/order-machine.ts:38`. The worker cannot call it without a new dependency the design treats as free.
- Break: three call sites now need the "same" cancel logic — the existing `apps/worker` handler, the new `apps/api` sweep, and (for the guard) the domain package. They are in three packages with non-overlapping dep graphs. "No logic fork" is contradicted by the module boundaries; in practice the cancel SQL + notify emit gets copy-pasted (the handler's UPDATE at `handlers.ts:26-30` is already a hand-written duplicate of the sweep SQL in the proposal, not a shared function). The "single authority" is aspirational, not structural.
- Violated invariant: single transition authority / DRY guard. The design's central robustness claim assumes a shared code path that the package layout forbids without a new dependency edge.

### H3 — B-SCALE / B-CONSIST · #2 modifier-group batching changes correctness, not just round-trips: the per-item loop and the batched query select different columns and partition differently.
- The per-item query at `orders.ts:430-436` selects `mg.id, mg.min_select, mg.max_select, mg.required` for one `product_id` and validates min/max-select **per item** at `:466-476`. The proposal §2/§3 #2A replaces it with one set-based query over all `product_id`s "partitioned by product in memory." Two break vectors:
  1. **Per-item vs per-product collapse loses cardinality.** The same `product_id` can appear in multiple cart line-items (e.g. two separate "Pizza" lines with different modifiers). The current loop runs the group-validation once **per line-item** against that line's `groupCounts` (`:438,456,466`). A single batched fetch keyed by `product_id` is fine for the *rules* (rules are per product), but the in-memory partition must re-broadcast the product's groups to **every** line-item that references it. If the partition is built as a `Map<product_id, groups>` and consumed once, repeated products silently skip validation on the 2nd+ line. The design says "partition by product" — that is correct only if each line-item re-reads the shared partition; the proposal does not state that, and the naive partition breaks multi-line carts.
  2. **MVCC coherence is preserved (the proposal is right there)** — but that is not the risk; the risk is the partition logic, which is presented as a pure round-trip optimization ("changes round-trip count, not snapshot coherence," §2). It changes the validation control flow, which is where modifier min/max bugs hide. No DoD test in resolution covers a cart with two line-items of the same product after batching.
- Break number: a 2-line same-product cart (common: "1 Margherita no cheese" + "1 Margherita extra cheese") can pass an invalid modifier-group selection if the partition is consumed once. Money/validation invariant at risk, not just latency.
- Violated invariant: server-authoritative modifier-group validation must hold per line-item.

### H4 — B-SEC / B-CONSIST · #4's `import.meta.env.DEV` gate does not remove the deeper risk: `o_mock_123` is a *navigation to a fabricated order id*, and the real failure path still swallows the error.
- The gate `if (import.meta.env.DEV && isDevMode())` does dead-strip the branch in a prod build (Vite statically evaluates `import.meta.env.DEV` to `false` for `vite build`, which `apps/web/package.json:8` runs). That part holds. But the design frames this as fully resolving ETHICAL-STOP-2. It does not:
  1. The post-gate failure path (`CheckoutPage.tsx:440-449`) only special-cases `MIN_ORDER_NOT_MET`; **every other failure** (network, 500, pool-wedge 5xx from #2's `SET LOCAL statement_timeout` abort) falls to a generic toast at `:448`. With #2 deliberately converting a wedged write into a fast 5xx (proposal §7 #2), the **most likely** failure under the exact load scenario #2 targets is a 5xx — which lands in the generic-toast branch with no phone CTA unless the fallback-config fetch (an *additional* network call, itself fragile under the same DB/load pressure) succeeds. The fallback's own fail-soft (§7 #4) then drops back to the bare toast. So under pool pressure the customer most often sees a plain "try again," the cart is preserved (good) but no out-of-band path — the degradation story is weaker than stated precisely when it matters.
  2. The dev-mock removal is verified by "grep prod bundle for `o_mock_123` → absent" (resolution threat-item 9). That proves the string is gone, not that no other code path fabricates success. It is a necessary, not sufficient, proof — fine, but the design overstates it as closing the ethical stop.
- Violated invariant: a failed real order must never present as success AND must offer a real fallback under the failure mode the system most expects. The first holds; the second is thin under #2's own 5xx.

## MEDIUM

### M1 — B-OPS · #3 down→503 has zero effect on the live failure it cites; Fly never reads `/health`.
- #3's stated value (proposal §0, §7 #3) is "a hung Postgres now correctly pages monitors instead of staying green." True for *external* monitors only. The design repeatedly reassures that the `/livez` vs `/health` split prevents a Fly restart loop — and it does, because `fly.toml:21` health check path is **`/livez`**, full stop. So: the 503 change is invisible to Fly (correct, no restart loop — but also the restart-loop worry is moot from the start), and it only matters if an external uptime monitor is actually pointed at `/health`. The proposal does not establish that any external monitor hits `/health` rather than `/livez` or `/`. If none does, #3's down→503 is a no-op in production observability — a correctness fix to a code path nothing reads.
- Also: `withTimeout`'s soft-check path is unchanged, but note the **worker** check (`health.ts:86-94`) reads `ops_worker_heartbeat WHERE last_seen_at >= now() - 60s` — combined with C2, this surfaces the worker VM's machine-id row as "ok," giving a green worker signal that says nothing about whether `order.timeout` drains. #3's "richer detail behind auth" inherits C2's blind spot.
- Violated invariant: observability changes must be wired to a consumer. Unproven that anything consumes `/health`'s status code.

### M2 — B-CONSIST · #2's `SET LOCAL statement_timeout` ≈4s can abort a legitimate large/slow order mid-write, and the catch path then shows the customer a failure for an order that may have partially executed.
- Proposal §3 #2A/b adds `SET LOCAL statement_timeout` (~4s) on the write tx so a wedged write self-aborts inside `connectionTimeoutMillis=5000`. The write path holds ~7 writes + N×(order_items + M×modifier inserts) inside one tx (`orders.ts:609-624`). At a large cart (say 15 line-items × 3 modifiers = ~60 inserts) under a DB latency spike — the exact trigger the fix targets — a *legitimate* order can exceed 4s and be aborted by `statement_timeout`. The tx rolls back cleanly (good, no partial commit), but the customer sees the generic failure toast (H4) for an order that was valid; on retry the idempotency key collides only if the client resends the same key. If the client regenerates the key (common on a fresh attempt), the customer can double-order.
- Break number: 4s budget vs ~60 round-trips at, say, 50ms each under spike = 3s baseline, no headroom. The timeout meant to protect the pool will fire on the legitimate large order during the spike — converting a slow-but-correct order into a customer-visible failure.
- Violated invariant: a valid order under load should complete, not be self-aborted; the bound trades pool safety for legitimate-order loss without a stated threshold check.

### M3 — B-DATA · A `demo` slug may already exist on staging (loader idempotency unverified) and the scrub proof only checks phone equality, not other PII vectors.
- §10-DATA.3 claims an "idempotent loader: upsert by stable external key." No stable external key exists across prod→staging (different `location_id`s, different DBs); the only natural key is `slug='demo'`, and the loader stamps a *new* `location_id`. Re-running without an upsert key duplicates categories/products. The design asserts idempotency without identifying the conflict target.
- The scrub proof (resolution #5 data-task: `staging demo.phone ≠ prod sushi-durres phone`) checks the phone column only. The public theme/menu payload can carry other identifying strings (owner name in branding, a contact email in `fallback_config`, address). The three-assertion gate does not cover those. PII-by-construction is asserted ("public APIs structurally can't return PII") but `fallback_config` is owner-authored free-form JSON (`fallback-config.ts:24` returns `row.fallback_config`) and could contain a personal phone/name.
- Violated invariant: zero PII crosses prod→staging; the proof is narrower than the claim.

## LOW

### L1 — B-OPS · #5 retention Go-gate scope value is correct, but the gate's "in-window" freshness check has no defined window against a nightly cron.
- Good news first (regression check, design is right): `anonymization_audit_log.scope` exists with `CHECK (scope IN ('retention','gdpr'))` (`1780421100060_anonymization-seam.ts:34`) and the retention sweep **does** write `scope='retention'` (`anonymizer-retention.ts:61` passes `scope:'retention'` → `anonymizer/index.ts:164` → `insertAuditLog` at `:286-288`). So the §5(c) gate "an `anonymization_audit_log` row with `scope='retention'`" is satisfiable. **However**: the sweep only inserts an audit row when it actually anonymizes something (`anonymizeCustomer`/`anonymizeOrder` insert per-subject at `:163,234`); a run that finds **nothing expired writes zero audit rows**. At launch (fresh tenant, no customer older than `retention_days=365`) the sweep runs nightly and produces **no** `scope='retention'` rows for ~a year. The Go gate "row in-window" is therefore unmeetable at launch by the audit-log path; only the heartbeat-OR branch can be green, and that heartbeat (C2 caveat) proves the api-side `anonymizer-retention` worker breathes, not that retention is enforced.
- Violated invariant: an honest retention promise needs a *positive* proof of enforcement; at launch there is no positive proof available, only "worker alive + nothing to do."

### L2 — B-SEC · #3 strips `data:` from `/health`, but `detail` still leaks driver error text on the soft-check/down paths.
- `withTimeout` returns `detail: \`${label}: ${err.message}\`` on the down path and `detail: \`${label} check timed out or failed: ${err.message}\`` on degraded (`health.ts:35,38`). #3 drops `data: result` (the row objects) but the design does not mention `detail`, which on a pg error surfaces the raw driver message (host, role, sometimes SQLSTATE/connection string fragments) on the unauthenticated `/health`. Dropping `data` alone leaves a smaller but real recon string on the exact endpoint the fix is hardening.
- Violated invariant: unauthenticated surface should not leak internal error text.

## Re-attack раунд 2 — регресія фіксів C1–L2

> Атаковано НОВЕ, що внесли фікси раунду 2: дві additive міграції (M-1 partial index, M-2 partial UNIQUE), C2-detection через pgboss.job overdue-query, standalone sweep у `apps/api`, broadcast-partition (H3), 4.5s statement_timeout (M2). Кожна знахідка re-verified проти живого дерева. Дві нові поломки — CRITICAL. Решта старих фіксів held (нижче — що саме перевірено).

### RC1 — B-DATA / B-OPS · M-2 partial UNIQUE `CONCURRENTLY` ЗЛАМАЄ міграцію: node-pg-migrate жене все в ОДНІЙ транзакції, а `CREATE INDEX CONCURRENTLY` заборонений у транзакції. **NEW CRITICAL.**
- Проп §5 / ADR Decision 1(a) / resolution C4a/H1 специфікують обидва нові індекси як `CREATE … INDEX CONCURRENTLY`. Але міграційний раннер у цьому репо явно документує, що **node-pg-migrate жене ВСІ pending-міграції в ОДНІЙ транзакції** (`--single-transaction` defaults to true) — `1790000000011_pgboss-bootstrap-schema.ts:18-30`: *"node-pg-migrate runs ALL pending migrations inside ONE transaction"*. Postgres кидає `25001: CREATE INDEX CONCURRENTLY cannot run inside a transaction block`.
- Доказ, що це реальна перешкода саме тут: ЄДИНА міграція в дереві, яка вирвалась із single-transaction — це та сама bootstrap (`grep -rln "noTransaction\|CONCURRENTLY" packages/db/migrations` → лише `1790000000011`), і вона зробила це ЯВНО через `pgm.noTransaction()` + ручний COMMIT (`:25-30`). Жоден `pgm.createIndex(... { unique:true })` у дереві (`1780338982025:22-23`, `1780338982026:11-16`) НЕ використовує `CONCURRENTLY` — усі вони звичайні (блокувальні) build всередині транзакції. Тобто `CONCURRENTLY` у звичайній up-міграції тут ще ніколи не проходив, і за поточним раннером пройти не може.
- Break: міграція впаде на старті → **`server.ts` не завантажиться** (той самий клас відмови, що описаний у `1790000000011:9` — "the API cannot boot"). C4-фікс (дедуплікація) і H1-фікс (sweep index) обидва залежать від цих міграцій → весь Go-path блокується на самій міграції. Без `CONCURRENTLY` (звичайний `CREATE INDEX`) — це AccessExclusiveLock на `orders` (H1 index) на час build; на pilot це мс, але це вже НЕ "online-safe `CONCURRENTLY`", як стверджує проп §5/R11.
- Violated invariant: forward-only міграція має застосовуватись. Специфікація `CONCURRENTLY` структурно несумісна з раннером цього репо; "online-safe `CONCURRENTLY`, no write-lock" (R11, проп §5) — **FALSE** проти раннера.

### RC2 — B-CONSIST / B-DATA · M-2 partial UNIQUE на `(event, target_id)` ЗЛАМАЄ ВСІ легітимні notification-аудити: код пише БАГАТО рядків на один (event,target_id) і безліч рядків з `target_id = NULL`. **NEW CRITICAL.**
- Фікс C4a (проп §5 M-2, resolution C4a) додає `UNIQUE … ON notification_outbox_audit (event, target_id) WHERE event='order.timeout_cancelled'` і будує emit на `INSERT … ON CONFLICT (event,target_id) DO NOTHING`. Але `notification_outbox_audit` — це **журнал спроб доставки, не журнал подій**: код пише туди НЕ одну фінальну позначку, а рядок на КОЖНУ фазу КОЖНОГО target-а:
  - `handleTelegramSend` для одного `(event, entity_id)` ітерує по ВСІХ active targets (`workers/index.ts:332`) і пише окремий audit-рядок `'sending'` (`:387-392`), потім `'failed'` (`:415-420`) АБО `'delivered'` (`:439-444`) — тобто **2 рядки на target на спробу**, і ще `'prefs_disabled'`/`'circuit_open'` гілки (`:338,350`).
  - `writeAudit` (`notifications/audit.ts:31-44`) — звичайний `INSERT` БЕЗ `ON CONFLICT`, пише `no_target`/`dedup`/`prefs_disabled`/`sending`/`delivered`/`failed`. Якщо два legit рядки `order.timeout_cancelled` з тим самим `target_id` зіткнуться з новим UNIQUE — цей `INSERT` кине `23505` і **впаде весь handler** (немає `ON CONFLICT`), що завалить доставку нотифікації, заради якої все робилось.
  - `target_id` у схемі **nullable** (`1790000000007:15`), і код реально пише NULL у dead-letter гілці (`workers/index.ts:470-473` — без `target_id`) та через `writeAudit` з `targetId || null` (`audit.ts:37`). У Postgres `NULL` у UNIQUE **не конфліктує сам із собою** — тож `ON CONFLICT DO NOTHING` для рядків з `target_id=NULL` нічого НЕ дедуплікує (дірка лишається), а partial UNIQUE на цих рядках просто безсилий.
- Break #1 (дедуп не працює як треба): "exactly-once" будується на `(event, target_id)`, але legit-потік пише `delivered`+`failed`+`sending` для того ж `(order.timeout_cancelled, target_id)` → ці legit-рядки самі конфліктнуть між собою на UNIQUE. Тобто або UNIQUE ламає аудит, або, щоб його не ламати, треба переписати ВСІ `notification_outbox_audit` вставки для цього event — чого фікс не специфікує.
- Break #2 (target_id семантика не визначена): проп ніде не каже, ЩО таке `target_id` для `order.timeout_cancelled`. У наявному коді `target_id` = id рядка `owner_notification_targets` (`workers/index.ts:391`), а НЕ order id. Якщо emit ставить туди order id — він в іншому домені id, ніж усі решта рядків того ж event, і дедуп по `(event, target_id)` стає семантично неузгодженим із рештою аудиту. Питання breaker-а "target_id це order id чи щось інше?" у дизайні **не вирішене** — а від відповіді залежить, чи UNIQUE взагалі коректний.
- Honest-caveat прелому: resolution C4a(*) каже "at pilot there are none (the emit doesn't exist yet), so the index builds clean". Це TRUE для рядків `order.timeout_cancelled` (їх ще нема). АЛЕ проблема не в build — проблема в тому, що ПІСЛЯ того як emit + звичайний legit-аудит почнуть писати для цього event, partial UNIQUE конфліктує з власним legit-патерном многорядкового аудиту. Build clean ≠ runtime safe.
- Violated invariant: at-most-once owner notification + не ламати наявний journaling-патерн. Новий UNIQUE накладається на таблицю, чий код пише N рядків на (event,target_id) і NULL-и — конструкт несумісний без переписування аудиту, чого фікс не передбачає.

### RH1 — B-OPS · C2-detection (pgboss.job overdue-query) НЕ має виконавця: LivenessChecker його не запускає, а ReconciliationWorker (де такий SELECT живе) ВИМКНЕНО. Детекція знов висить на мертвій/неіснуючій інфрі. **HIGH (регресія C2/C3).**
- C2-фікс (проп §3 #1 detectability, §7 #1, §9, resolution C2) перебудовує lost-job detection на `SELECT count(*) FROM pgboss.job WHERE name='order.timeout' AND state IN ('created','active') AND start_after < now()`, alert коли `>0` >60s. Питання: ХТО цей SELECT виконує і алертить?
- Доказ, що ніхто: ЄДИНИЙ scheduled-checker у `apps/api`, що зараз біжить — `LivenessChecker` (`server.ts:430-431`). Його `run()` робить ЛИШЕ `SELECT … FROM ops_worker_heartbeat` (`liveness-checker.ts:46-54`) і матчить по `CRITICAL_WORKERS` іменах (`:57`). Він **не торкається `pgboss.job`** і не має overdue-логіки. Той самий overdue/failed-job SELECT, на який посилається дизайн (`reconciliation.ts:237-246`), живе ВИКЛЮЧНО в `ReconciliationWorker`, який **вимкнено в prod** (`server.ts:382`, підтверджено round-1 C3).
- Break: фікс перебудував детекцію з heartbeat (C2: не біндиться) на pgboss.job-query — але цей query НЕ мешкає в жодному живому процесі. Дизайн каже "surfaced in `/health` detail or an ops counter" (проп §3 #1) — але `health.ts` worker-check читає `ops_worker_heartbeat` (M1 нота), а не pgboss.job overdue. Тобто детекція переїхала з "механізму, що не біндиться" (C2) на "механізм, якого ніхто не викликає" — тот самий клас провалу, що C3. Lost-job <1 min лишається недетектованим у prod.
- Violated invariant: lost-job detectability < 1 min. Жоден deployed-процес не виконує overdue-query; "surfaced in /health detail" не реалізовано в `health.ts`.

### Перевірено й HELD (фікси раунду 2, що тримаються)

- **Standalone sweep host реальний (C3-фікс держить).** `apps/api` справді має повну `boss.schedule`/`createQueue`/`work` інфру поза NotificationWorker: `anonymizer-retention.ts:22,26,27`, `liveness-checker.ts:32,37,38`, `server.ts:435,443` (FREE_TIER_WATCH `boss.work`+`boss.schedule`). Тож sweep як власний `boss.schedule` у `apps/api` — runnable, не залежить від вимкненого ReconciliationWorker. HELD.
- **Подвійне виконання sweep — НЕ ризик на pilot.** `fly.toml:15` `auto_stop_machines=false` + єдина web-VM (`fly.toml:25-27`, без `min_machines_running>1`/multi-count). Один api-інстанс → sweep не біжить двічі. Якщо колись додадуть 2-й web-інстанс — `singletonKey` на schedule (як у `anonymizer-retention.ts:22,27`, `liveness-checker.ts:32,39`) дедуплікує запуск; дизайн має це застосувати, але на pilot це не ламається. HELD (з нотою: 2-й інстанс без singletonKey → подвійний sweep, тоді ж і RC2-double-emit; покрито RC2).
- **apps/api НЕ спить (sweep не пропуститься через сон).** `auto_stop_machines=false` (`fly.toml:15`) → web-машина не зупиняється, scheduled sweep тикає. HELD. (`order.timeout` consumer — у `apps/worker`, окрема always-on VM `fly.toml:30`; теж не спить.)
- **Broadcast-partition спека (H3-фікс) тепер коректна — за умови дотримання.** Жива петля валідації справді per-line-item (`orders.ts:426` `for (item of items)`, `:438` свіжий `groupCounts` на лінію, `:466-476` min/max на лінію). Якщо §2/§3 #2A будує `Map<product_id, groups[]>` і КОЖНА лінія re-read-ить (broadcast, не drain) — семантика збігається з поточним per-line `groupRes.rows`. Специфікація тепер це вимагає явно + DoD-тест (2-лінійний same-product cart). Коректна як специфікація; залишковий ризик — імплементація, не дизайн. HELD-as-specified.
- **4.5s statement_timeout (M2-фікс) — headroom реальний, але тонкий.** Baseline ~60 inserts × ~50ms ≈ 3s (вхідне число breaker-а, проп §3 #2b). 4.5s vs 4s додає +0.5s = ~17% headroom над 3s-baseline (1.0s→1.5s запасу). Дрібно, але це НЕ головний захист від double-order — реальний замок на idempotency-layer: retry того ж cart → той самий content+menu_version `request_hash` → 23505 (проп §6, `orders.ts:194,197`), за умови без client-nonce. M2 закрито на idempotency-рівні незалежно від bound; bound лишається тонким, але це accepted-risk, не break. HELD (з нотою: 0.5s — косметичний headroom; справжній замок — content-derived hash).

**Підсумок раунду 2:** дві НОВІ CRITICAL від самих фіксів — RC1 (`CONCURRENTLY` несумісний із single-transaction раннером → міграція впаде, API не стартує) і RC2 (partial UNIQUE на `(event,target_id)` ламає наявний многорядковий/NULL-аудит-патерн і має невизначену `target_id`-семантику). Плюс RH1 (HIGH-регресія): C2-detection-query не має виконавця в prod. Решта фіксів раунду 2 (standalone-host, no-sleep, no-double-run на pilot, broadcast-spec, timeout-bound) — held, з нотами.

## Final convergence — раунд 3 held

> KROK 6 hard-exit. Цільова регресія: чи round-3 (прибрав обидві міграції M-1/M-2, accept-risk R12 на at-most-once notification, detection co-located у sweep-job) не лишив діри. Кожен пункт re-verified проти живого дерева. **Нуль нових CRITICAL/HIGH. Дизайн готовий до GO.**

### 1. RC2-fix: подвійний емітер (handler + sweep) — задушений у межах single VM, R12 чесний. HELD.
- Перевірено живе: обидва шляхи РЕАЛЬНО спостерігають-і-емітять для того ж order. Handler (`apps/worker/src/handlers.ts:24-30`) кенселить, потім (round-3 spec) емітить на спостереженні CANCELLED; sweep пізніше бачить вже-CANCELLED і теж намагається емітити. Це справді ДВА емітери на той самий order — питання поставлене коректно.
- Але дедуп **реально душить у межах одного процесу**: dedupKey = `` `${event}:${entity_id || ''}:${location_id}` `` (`apps/api/src/notifications/workers/index.ts:298`), тобто ключ на **entity_id (order id)**, НЕ на `target_id`. Другий еміт того ж `(order.timeout_cancelled, orderId, locationId)` потрапляє в `dedupCache.has(dedupKey)` → skip (`:301-307`). Handler + sweep обидва біжать у тому самому `apps/api`-процесі (NotificationWorker, `server.ts:347`), `auto_stop_machines=false` + єдина web-VM (`fly.toml:15`) → один процес → Set покриває. `singletonKey: order.timeout_cancelled:{orderId}` додатково душить in-flight-джоби.
- R12 чесний: він НЕ стверджує exactly-once глобально — він явно каже "exactly-once **in-process**, residual = дубль owner-Telegram якщо api **restart** влучає у ~60s emit-gap" (proposal §10 R12, resolution R12). Це точний опис справжньої залишкової діри, не applied-makeup. Числа в resolution:336-339 (`P(restart) × P(restart у 60s-вікні timeout-order)` ≈ near-zero/міс, ціна = один зайвий Telegram) — це чесний accept-risk, не severity-dodge.
- Чому НЕ regress до CRITICAL: справжні дефекти C4 були (i) zero-notification на crash-recovery і (ii) double-send. (i) закрито незалежно від substrate через "emit on observing CANCELLED" (R2 design-change kept) — whichever path acts emits, ніколи нуль. (ii) лишає тільки cross-restart-дубль — benign, near-zero, owner-accepted. **HELD.**

### 2. RC1-fix: прибрання індексу — seq-scan sweep дешевий, cross-tenant lock-scope прийнятний. HELD.
- Back-of-envelope sweep тримається: PENDING-rows у будь-який момент ≈ `0.5 orders/s × ~5 min ≈ 150` (proposal §5:152), вся таблиця `orders` — low-thousands на місяці. Seq-scan `UPDATE … WHERE status='PENDING' AND timeout_at<now()` раз/60s по few-thousand-row таблиці — sub-ms, поза hot-path (standalone schedule, не request). Sub-ms твердження реалістичне для такого N.
- Cross-tenant UPDATE без `location_id`: бере row-locks ЛИШЕ на матчені рядки (`status='PENDING' AND timeout_at<now()` ≈ ті ж ~150, реально жменя справді-overdue), не table-lock. Це НЕ забагато локів на pilot — seq-scan читає всю таблицю, але lock тільки на overdue-PENDING. Прийнятно.
- Чесність прибрання: round-3 обрав drop (option b), не `pgm.noTransaction()`+non-concurrent (option a, що взяв би AccessExclusiveLock). Це уникає і `25001` (CONCURRENTLY у single-tx раннері, `1790000000011:18-30`), і table-lock. H1 коректно знижено з "false claim" до "perf-claim просто не робимо" (R11). **HELD.**
