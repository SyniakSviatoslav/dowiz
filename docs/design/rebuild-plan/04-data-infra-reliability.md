## 1. Postgres connection management — the high-value fix

### What is actually right (keep it)

The two-pool split is the textbook Supavisor pattern and should not be replaced: hot-path API traffic on **transaction mode :6543** (multiplexed, cheap, no session state), and LISTEN/NOTIFY + advisory-lock + DDL work on **session mode :5432** ([Supabase: connecting to Postgres](https://supabase.com/docs/guides/database/connecting-to-postgres), [Supavisor pool modes](https://supabase.github.io/supavisor/configuration/pool_modes/)). The `withTenant` helper (`tenant.ts:3-21`) is exactly the transaction-pooling-safe GUC pattern: `set_config(..., is_local=true)` inside an explicit `BEGIN…COMMIT` pins the whole unit to one backend and auto-clears the GUC at COMMIT/ROLLBACK. The superuser guard and per-pool `statement_timeout`s are genuinely good hygiene.

### 🔴 The bug class: session-level GUC over transaction pooling (P0, council-gated — RLS red-line)

`spa-proxy.ts:771` and `onboarding.ts:75` run `set_config('app.user_id', $1, **false**)` on an **autocommit** client from the **transaction-mode** pool. Under Supavisor transaction pooling, every autocommit statement is its own transaction and may be routed to a **different backend connection**; session state is not supported in this mode ([Supavisor pool modes](https://supabase.github.io/supavisor/configuration/pool_modes/), [Supavisor FAQ](https://supabase.com/docs/guides/troubleshooting/supavisor-faq-YyP5tI)). Two concrete failure modes:

1. **Flaky RLS denial**: the follow-up `UPDATE locations …` (`onboarding.ts:85-88`) can land on a backend that never saw the GUC → FORCE-RLS silently filters/denies → intermittent onboarding failures that no local test reproduces (locally, one client = one backend, so it "works").
2. **Cross-request GUC pollution**: the backend that *did* execute the `set_config(..., false)` keeps `app.user_id` set for whatever transaction lands on it next — a latent **cross-tenant RLS-context leak**. The in-code comment ("set_config persists across statements on a single client", `spa-proxy.ts:766`) is true only for direct/session connections; it is false through the transaction pooler.

**Fix** (small, but 🔴 council-gated because it changes RLS-context semantics on an owner-bootstrap path): wrap each block in `BEGIN … COMMIT` with `set_config(..., true)` — i.e. route through `withTenant` (or a sibling `withTenantLocation` for `app.current_tenant`). The `bootstrap_owner` SECURITY DEFINER call composes fine inside the transaction. Effort **S** (2 sites). Then **ratchet it**: a `tools/eslint-plugin-local` rule (or grep gate) banning `set_config(` + `, false)` in `apps/api/src` and requiring `set_config` only inside the sanctioned helpers, proven red→green per the harness rules. That converts today's ~30 hand-rolled `pool.connect()` GUC sites from "each one is a review burden" into "the pattern is mechanically enforced."

### Starvation-proofing (the incident class)

The prod starvation incident was txn-hold × pool-size arithmetic, and both factors are addressable without new infrastructure:

- **Shrink the hold** — the dominant lever. Worst-case create-order holds a pooled connection ~3s (60 sequential inserts; the code's own comment, `orders.ts:113-117`). §2a cuts that ~10–30×.
- **Alert on the gauge that already exists** — `pg_pool_operational_waiting` (`server.ts:262`) was built as "the early-warning for the operational-pool starvation incident class" but nothing watches it. Wire a threshold (waiting > 0 for > 30s → Sentry event) into the existing free-tier-watch/reconciliation cadence. Effort S.
- **Connection budget math** (Supabase Nano/Micro: **60 direct / 200 pooler clients** — [compute & disk](https://supabase.com/docs/guides/platform/compute-and-disk)): op pool 20 are transaction-pooler clients (200-cap — safe). Session-side per deploy: backup 2 + session 3 + API pg-boss 4 + worker pg-boss 4 = **13 session connections**, each holding a dedicated backend — comfortably inside 60, but only because the pools are small; document the budget and keep it in free-tier-watch (its "15 active" cap already approximates this).
- **Hygiene**: fix the "max:8" comment drift (`orders.ts:114`); have pg-boss use `DATABASE_URL_SESSION` instead of the hand-port-swapped operational URL (`server.ts:243-245`) so one env var means one thing.

**Verdict: no pooler/topology change.** Supavisor is already managed and correctly split; the fragility was never "multi-pool" — it was GUC discipline + txn hold. PgBouncer/self-hosted pooling would add ops surface for zero gain.

## 2. Query performance & data-access reliability

Replace with two multi-row statements (`INSERT … SELECT * FROM unnest($1::uuid[], $2::text[], …) RETURNING id`, then map modifier rows against the returned ids). Same rows, same RLS, same txn — just ~10–30× less hold time on the exact path that starved the pool. Proof: existing order E2E + a unit asserting one `order_items` statement per order.
- **2b. Trivial: drop the redundant OTP column re-query** (`orders.ts:162-168` — the value is already in `locRes`). One fewer round-trip per order.
- **2c. 🔴 RLS initplan wrapping (M, staged, council-gated).** All 48 policy references use the bare per-row form (`location_id = current_setting('app.current_tenant')::uuid`, e.g. `migrations/1790000000036_order-routes.ts:28`; membership `EXISTS` subqueries e.g. `1780338982025_import_sessions.ts:31`). Postgres re-evaluates these per row; wrapping as `(SELECT current_setting(...))` hoists to a once-per-query InitPlan — Supabase documents >100× improvements on large tables and ships a dedicated linter for exactly this ([RLS performance & best practices](https://supabase.com/docs/guides/troubleshooting/rls-performance-and-best-practices-Z5Jjwv), [discussion #14576](https://github.com/orgs/supabase/discussions/14576), [database advisor 0003_auth_rls_initplan](https://supabase.com/docs/guides/database/database-advisors?queryGroups=lint&lint=0003_auth_rls_initplan)). - **2d. Slow-query surfacing: use what Supabase already runs.** There is no OTel in the API (verified) and adding the SDK to a 512 MB VM for this is YAGNI. `pg_stat_statements` is enabled on every Supabase project — a monthly ritual of "top 10 by `total_exec_time` + index advisor + unused-index check (`pg_stat_user_indexes`)" costs nothing and finds real wins ([Supabase database advisors](https://supabase.com/docs/guides/database/database-advisors)). Pair with the existing `/metrics` route-latency counters for the app side.

- **2f. Indexing approach**: 98 indexes already exist; don't add blind. The 2d ritual (advisor + unused-index sweep) is the whole recommendation — quarterly, 30 minutes, zero risk.

Rename `RedisMessageBus` → `PgMessageBus` and fix the `REDIS_URL … pub/sub only` config comment **now** (S, pure hygiene) even if 3c waits.
- **3d. CDN/ETag: DEFER.** Public menu already ships `max-age=60, stale-while-revalidate=300` + `X-Menu-Version`; at current traffic a CDN layer buys latency Albania→fra doesn't need yet and adds an invalidation surface. Revisit if egress or TTFB becomes a measured problem.

## 4. Job queue — keep pg-boss, fix the config debt

pg-boss stays (lane 2 concurs; one datastore, and current pg-boss is actively maintained — v12.24.x as of mid-2026, with DLQ/retry semantics native since v10: [pg-boss](https://github.com/timgit/pg-boss), [v10 release notes](https://github.com/timgit/pg-boss/releases/tag/10.0.0), [npm](https://www.npmjs.com/package/pg-boss)). The reliability gap is not the engine — it's five config/ordering defects:

1. **P0 — worker on the wrong pooler** (`apps/worker/src/index.ts:11` → `:6543` fallback): LISTEN/NOTIFY and session advisory locks don't work through transaction mode, so the worker VM's queue behavior (job pickup latency, maintenance, graceful drain) is degraded/broken. One-line fix: construct with `env.DATABASE_URL_SESSION`.
2. **Bare-create defeats reliability options**: `server.ts:251-254` creates all 30 queues optionless at boot, before the backup worker's `createQueueWithDefaults` — and v10 `createQueue` won't update an existing queue, so even the 6 "hardened" queues may run on defaults. Fix: one create path (`createQueueWithDefaults` in the boot loop), verified by querying `pgboss.queue` options in a test.
3. **Defaults are wrong for side-effect queues**: retry 2 / 0s / no backoff / no DLQ means a transient Telegram/Push outage burns all retries in milliseconds, then the job dies invisibly (`state='failed'`; the only watcher is a nightly >10-per-queue reconciliation check, `reconciliation.ts:232-248`). Apply retry 3/30s/backoff + `deadLetter` per queue class; add `.dlq` + `failed` counts to `/health` and the `pgboss_jobs_pending` gauge's neighborhood. Also reconcile `handleTelegramSend`'s app-level `MAX_RETRIES=10` (`notifications/workers/index.ts:516-534`) with the queue's actual limit — today that code path is unreachable.
4. **Idempotency under at-least-once**: `notify.customer_status` is enqueued with no `singletonKey` (`bootstrap/messaging.ts:92,138`) and Telegram dedup is an in-memory Set lost on every deploy (`index.ts:70-71`). Fix: singletonKey on enqueue + a durable delivery claim — `INSERT INTO notification_outbox_audit … ON CONFLICT DO NOTHING RETURNING id` *as the send gate*, not just an audit. The money-adjacent paths (order.timeout, courier.dispatch) are already correctly idempotent (singletonKey + status-guarded UPDATE + unique constraints) — preserve as the reference pattern.
5. **Version skew**: v10.4.2 runtime under v12 types, with casts (`bootstrap/workers.ts:137-168`). Align `@deliveryos/platform` to ^12 (schema migration between majors is operator-run via `scripts/bootstrap-pgboss.ts` with `DATABASE_URL_MIGRATIONS` — 🔴 treat the pgboss-schema DDL as gated, staged on staging first). Add the missing maintenance options (`deleteAfterDays`, `monitorStateIntervalSeconds`) while there — the 1800-job pileup note at `messaging.ts:33-36` says retention defaults have already bitten once.

## 5. WebSocket / realtime — keep the architecture, add the missing floor

The hypothesis "raw ws is fragile at multi-instance" is **falsified by the code**: fan-out rides Postgres LISTEN/NOTIFY (`PgMessageBus`), every machine LISTENs, rooms are per-process by design — multi-instance-correct today. The layer also carries substantial paid-for hardening: tri-state courier/owner relay guards with fail-safe ceilings (ADR-0013, `courier-relay-guard.ts`, `websocket.ts:102-171`), heartbeat/zombie reaping, subscription-leak fix (P1-WSDUP), claim-check for the 8 KB NOTIFY cap, indefinite bus reconnect.

- **REJECT Supabase Realtime migration.** It exists and its RLS-on-`realtime.messages` authorization model is real ([Realtime authorization](https://supabase.com/docs/guides/realtime/authorization)) — but migrating means re-expressing the council-gated tri-state courier-binding and owner-membership guards as RLS policies on a Supabase-owned table, re-proving the C1 leak closure, and accepting a 200-concurrent-connection free-tier cap ([pricing](https://supabase.com/pricing)). That is L effort and high regression risk on a hardened security surface, to replace something that works. The LISTEN/NOTIFY scaling ceiling (single-Pg fan-out) is orders of magnitude away at current scale.
- **ADOPT the backpressure floor (S)**: `new WebSocketServer({ server, maxPayload: 64 * 1024 })` + a `bufferedAmount` ceiling check before relaying (skip-or-terminate a consumer buffering > ~1 MB). Today inbound frames and slow consumers are unbounded (`websocket.ts:192`, no checks anywhere) — the one genuine DoS gap in the layer.
- **ADOPT (S)**: FE refetch on `_truncated` claim-check frames (currently the delta merge can silently drop a >7800-byte update to `{id}`); a `bus_degraded` gauge/flag so a machine whose listener is mid-reconnect isn't silently stale behind a passing `/livez`; finish the `?token=` deprecation once its usage counter hits zero.

## 6. Infra & Docker — the quantified wall-clock wins

- **6a. Fix the layer order (P0, S).** `Dockerfile:10-16` copies all of `packages/ apps/ scripts/` before `pnpm install --frozen-lockfile`, so **any** source edit invalidates the install layer → full dependency reinstall in every one of the 4–5 min builds. Canonical fix ([pnpm docker guide](https://pnpm.io/docker)): copy lockfile+manifests → `pnpm fetch` (only lockfile-sensitive) → copy source → `pnpm install -r --offline`, plus a BuildKit cache mount for the pnpm store. Expected: warm rebuilds drop to the `pnpm -r build` cost (~1–2 min); dep layer rebuilt only on lockfile change. In CI, buildx with `cache-to/from: type=gha,mode=max` gives the same across runs (80%+ typical cut on unchanged-deps builds — [Docker GHA cache backend](https://docs.docker.com/build/cache/backends/gha/)).
- **6b. Move the Docker build out of pre-commit (P0, S — the standing P1).** `.husky/pre-commit:40-58` runs a full image build per commit and drags `docker-disk-guard.sh` (which exists because per-commit orphaned layers once filled the disk and crashed local Postgres into WAL recovery). CI is the right home: a `docker-build` job on PR with the gha cache; deploys already build remotely (`flyctl deploy --remote-only`). Saved: 4–5 min × every commit, plus the disk-crash exposure retired. Pre-commit keeps lint/typecheck (fast, high-signal).
- **6c. Determinism (S)**: `Dockerfile:7` `corepack prepare pnpm@latest` silently overrides the repo's `pnpm@9.4.0` pin (`package.json:8`) — use `corepack enable && corepack prepare --activate` (honors `packageManager`). Runtime-stage `npm install argon2 sharp @aws-sdk/client-s3 @aws-sdk/lib-storage` (`Dockerfile:53`) is unpinned and lockfile-less — pin exact versions matching the workspace lockfile.
- **6d. Image size/base**: 590 MB uncompressed is dominated by node + sharp/aws-sdk; distroless is **rejected** (glibc-native modules, loses shell debuggability, saves ~50–80 MB that costs nothing on Fly's rootfs pulls after first deploy). Go to `node:24-slim` per lane 2, add `npm cache clean --force` after the runtime install, done.
- **6e. Fly sizing/cost**: web shared-cpu-1x 512 MB ≈ **$3.19/mo**, worker 256 MB ≈ **$1.94/mo** ([Fly pricing](https://fly.io/docs/about/pricing/), [calculator](https://fly.io/calculator/)) — the floor is already reached; autoscaling machinery for a ~$5/mo footprint is YAGNI. `auto_stop_machines=false` is correct while WebSockets are load-bearing. Single-region fra suits an Albania user base (~25–35 ms). The one thing to document: there is no `min_machines_running`, so redundancy is one machine per process — an accepted SPOF at this stage (see §7).

## 7. Reliability / resource — close the small gaps, name the SPOFs

**Adopt (all S):**
1. **Fastify timeouts**: set `requestTimeout` (~30s) and `keepAliveTimeout` above the upstream proxy's idle timeout; today all are defaults (`server.ts:82-101`) — no slow-loris/idle-socket floor.
2. **Global shutdown deadline**: wrap the whole `setupShutdown` sequence in `Promise.race(…, 25_000)` + `process.exit(1)` fallback — today only the queue drain is boxed (10s) and `fastify.close()`/`pool.end()` are unbounded against Fly's 30s SIGKILL (`shutdown.ts:44-80`, `fly.toml:44`). Also drop the duplicate `queue.stop()` (called in both `server.ts:870` and `shutdown.ts:57`).
3. **Heap ceilings**: `--max-old-space-size=384` (web) / `192` (worker) in the Fly process commands, so memory pressure becomes GC work instead of a hard OOM kill on the 512/256 MB VMs.
4. **`redis.on('error')` handler** on the decorated client (`server.ts:352`) — today errors fall through to the process-level guards. (Moot if §3c lands.)

**Keep as-is (verified good):** honest `/health` vs cheap `/livez` split with the WS-preserving rationale (`fly.toml:27-34`, `routes/health.ts:53-63`); release-command migrations with fail-safe abort; boot guards (env preflight throw, schema-head `exit(1)`, 3s-budget non-fatal worker boot); upload DoS hardening (10 MB + truncation→413 + 5/min); free-tier watch thresholds; real-client-IP rate-limit keying. In-memory rate-limit store is correct at one web machine — **revisit only at scale-out** (move keyed limiters OTP/auth to a shared store then).

**SPOF ledger (accepted, documented):**
- **Postgres is the everything-SPOF** — data + realtime bus + job queue. This is a deliberate one-datastore trade; mitigations already exist (bus reconnect, worker-boot resilience, backup/restore drills in `/health`). The rebuild plan should *strengthen* this bet (§3c removes Redis), not dilute it.
- **One web machine, one region** — a deploy or host failure is a brief outage; acceptable at current revenue, revisit with `min_machines_running=2` + shared rate-limit store when uptime is worth >$3.19/mo.
- **Redis is explicitly NOT a core SPOF** (routing fails open, `routing.ts:146,160`; only anonymous import 503s) — a good property to preserve until it's removed.
