## PART 1 — Did v2 kill v1's CRITICALs?

### C1 (fleet-wide sweep wedge) → CONFIRMED-FIXED (money-coupling), with a pre-existing caveat
The v1 wedge was a *throwing* `payment_events` INSERT folded into the atomic `LANGUAGE sql` sweep (`1790000000078_phase2-sweep-fns.ts:13-22`). v2 **drops that fold entirely** (M-1 no longer touches the sweep — verified against §3.2/§5 and the scoreboard) and records the obligation via a non-throwing `AFTER UPDATE OF status` trigger + the L-D reconciler. Verified sound:

- **Can a poisoned row halt the fleet-wide sweep (P6b)?** No. The sweep body is byte-unchanged; it still does one CTE `UPDATE orders SET status='CANCELLED' … RETURNING` (`:16-22`). The trigger fires as a queued AFTER-ROW action; if its INSERT throws, the throw is contained in the trigger's own `BEGIN/EXCEPTION` subtransaction and the outer sweep statement completes. One poisoned row cannot wedge cancellation for anyone. **P6b holds.**
- **Does the per-row swallow actually catch an RLS `WITH CHECK` rejection, or does a trigger error still abort the statement?** It genuinely catches it. An RLS `WITH CHECK` failure raises SQLSTATE `42501` (`insufficient_privilege`) — an ordinary ERROR, caught by `WHEN OTHERS`, which rolls back the plpgsql subtransaction and lets the statement proceed. This is **already the proven pattern in this repo**: `updateOrderStatus` wraps its `order_status_history` insert in a SAVEPOINT precisely "so a failed insert (e.g. RLS denial) cannot poison the caller's transaction" (`orderStatusService.ts:150-163`). Non-throwing property is real.
- **Does the trigger fire on ALL terminal-status paths?** Yes, and it misses none of the four asked:
  - sweep → `UPDATE orders SET status='CANCELLED'` (`:16`) — status in SET → fires.
  - manual / owner PATCH / completeDelivery-refused-tail / customer-cancel → all funnel through `updateOrderStatus` (`orderStatusService.ts:113-118`) `UPDATE orders SET status=…` → fires.
  - webhook is NOT a terminal-status setter (it writes `payments.status` + `orders.payment_status`, never `orders.status` — `payments-webhook.ts:60-82`); the pay-after-cancel direction is L-B's job, so the trigger *correctly* does not need to fire there — no miss.
  - Terminal status lives only on `orders.status`; every transition is an `UPDATE orders` — there is no terminal-setting path that bypasses it. Confirmed no gap.
- **Recursion?** The trigger body inserts into `payment_events` only; it never `UPDATE`s `orders`, so `AFTER UPDATE OF status` cannot re-fire itself. Safe.

**Caveat (pre-existing, already flagged as DEP-2 — not a v2 regression):** the sweep's own `hist` CTE INSERT into `order_status_history` (`1790000000078:19-21`) is un-swallowed, and that table's policy is member-only with no GUC arm (`1780338982015:19`). Post-B3-flip that INSERT throws and aborts the **whole sweep** — the identical fleet-wide wedge, now caused by the history insert rather than money. v2 correctly leaves the sweep untouched and carries this on the B3-flip checklist (DEP-2), but the honest statement is: **C1's money-coupling is fully removed; the sweep's post-B3 liveness is still blocked on DEP-2.** Both crypto-flip and B3-flip are gated, so this is consistent — just don't read "C1 fixed" as "the sweep is B3-safe."

### C2 (settlement backfill double-pay) → CONFIRMED-FIXED, with one MEDIUM residual (see N2)
v2 lower-bounds the catch-up scan by a literal deploy watermark and moves pre-watermark rows behind an operator-gated `app_backfill_historical_settlements()` with a pre-count and a `backfilled` flag. Verified against the current fn (`1790000000078:160-197`, item scan `:173-179`):

- **Is there a window where a pre-watermark row still gets auto-swept (P9b)?** In steady state, no — cron scans `delivered_at >= WATERMARK`; the operator flow scans `< WATERMARK`; the ranges are disjoint. **But the watermark's *value* is the whole game — see N2**, which is a real reopening for the authoring→deploy gap.
- **Double-count at the cron/backfill boundary?** Structurally impossible regardless of range arithmetic: `settlement_items_assignment_uniq` (`1780421100045:15`) is a hard unique on `assignment_id`, so even if cron and backfill both selected the same boundary row, the second `INSERT … ON CONFLICT (assignment_id) DO NOTHING` (`:181-183`) is a no-op. No double-settle.
- **Race across a redeploy?** `CREATE OR REPLACE` re-running the *same* migration re-bakes the *same* literal → deterministic, no race. (A *future* migration that moves the watermark forward would orphan the gap — out of scope, worth a checklist note.)

### DEP-1 (customer-cancel through updateOrderStatus) → CONFIRMED-FIXED / safe
- **Premise verified:** `orders` genuinely has **no** `cancelled_at`/`cancellation_reason` column (create table `1780310074262:21-43`; those columns exist only on `courier_assignments` `1780421100041:16`), so today's raw `UPDATE orders SET … cancelled_at=now(), cancellation_reason=$1` (`customer/orders.ts:308-312`) 500s on a phantom column. LC3 is real; rerouting through `updateOrderStatus` (which sets only `status`/`timeout_at`, `orderStatusService.ts:113-118`) removes the phantom write.
- **Does it work under a FORCE-RLS non-bypass role (P4b)?** Yes — and this is the load-bearing fact I verified: `orders` **has a GUC UPDATE arm**. `courier_tenant_update` (`1790000000077:50-53`) is `FOR UPDATE USING/WITH CHECK location_id = current_setting('app.current_tenant', true)`. So once DEP-1 does `set_config('app.current_tenant', $loc, true)`, `updateOrderStatus`'s `UPDATE orders SET status='CANCELLED'` passes post-B3, and the L-C trigger's `payment_events` insert passes via the dual policy's GUC arm (`1790000000083:77-81`). This is the *exact* precedent the webhook already relies on: `payments-webhook.ts:41` sets the GUC and its own `UPDATE orders SET payment_status='paid'` (`:66-70`) leans on the same `courier_tenant_update` policy. Precedent is genuine, not a hand-wave.
- **New authz hole — can a customer set a tenant context they shouldn't?** No exploitable hole. `location_id` is resolved from an order the customer provably owns (`WHERE o.id=$1 AND o.customer_id=$2`), so the customer can only ever set the tenant to a location where they hold an order, and the tx runs only ownership-scoped statements. One **LOW** defense-in-depth note carried to N5.

## PART 2 — New flaws in the v2 mechanisms

### N2 — MEDIUM — Watermark-literal-choice reopens C2 double-pay for the authoring→deploy gap
The proposal specifies the catch-up floor as "a literal deploy-watermark timestamp baked into the fn body **at migration-authoring time**" (§4.1(1), §5 M-2). Authoring time and prod-deploy time are not the same instant. Under Ship Discipline this migration goes staging-first, then prod after approval — **hours-to-days** later. If the baked literal is an authoring-time (past) value `< actual prod deploy`, then every row delivered in `[literal, prod_deploy]` is (a) processed by the OLD buggy fn while it is still live (SKIP-LOCKED-droppable, and plausibly reconciled in person — the exact C2 hazard) *and* (b) `>= WATERMARK`, so the NEW fn **auto-sweeps it into a fresh `pending` payout** → double-pay reopens for that window. The safe value is `≥ prod_deploy` (a future timestamp), which a compile-time literal chosen at authoring cannot guarantee; the phrase "at authoring time" points implementers at the wrong side of the boundary. Erroring late (literal after deploy) only defers rows to the operator backfill (safe); erroring early (literal before deploy) pays cash twice. The fix's correctness is entirely contingent on this one literal being chosen `≥` the actual prod deploy, and the spec does not say so. Grounded in the live scan bound `delivered_at >= p_period_start` at `1790000000078:166,177` that M-2 replaces.

### N3 — MEDIUM — L-C trigger adds a per-row subtransaction to the fleet-wide sweep for EVERY cancel
The trigger body "wraps the insert in `BEGIN … EXCEPTION WHEN OTHERS`" (§3.2). A plpgsql block with an EXCEPTION clause establishes a **subtransaction on every entry**, whether or not an error occurs (Postgres docs call this out explicitly). The `WHEN status IN ('CANCELLED','REJECTED')` clause cannot filter on "has a paid payment" (a subquery isn't allowed in `WHEN`), so the exception-wrapped block is entered for **every** sweep-cancelled order — overwhelmingly cash orders with zero paid payments today. The sweep cancels all overdue PENDING orders in one statement/one transaction (`1790000000078:16-22`); a backlog flush (e.g. after an outage, hundreds of orders time out at once) → hundreds of subtransactions in one top-level xact → past the 64-subxid cache the `pg_subtrans` SLRU is consulted, which under concurrency causes the well-known subtrans SLRU-contention stall — on a live 1-minute cron that is also the stuck-queue detector. Today's sweep is a single statement with **zero** subtransactions, so this is a genuine new cost introduced by v2. Cheap mitigation the spec omits: guard the exception block behind `IF EXISTS (SELECT 1 FROM payments WHERE order_id=NEW.id AND status='paid')` so only (dark, ~zero-today) crypto orders pay the subxact cost. Latent until crypto volume + a timeout backlog coincide, but the per-row subtransaction is incurred for all cancels regardless.

### N4 — MEDIUM — M4 ratchet is oversold: an import-path allowlist cannot block inline laundering through the module-under-test
`resolution.md` labels M4 **FIXED**: vector files must be zero-import + literals-only, and composition test files "may import **only** the module under test + the vector file." That kills alias/hoist/`estimateOrderTotal`-from-another-module evasions. It does **not** kill the core disease. The module under test must be importable (that is the whole point of the test), so a test can still write `const expected = composeOrderTotal(sub, opts); assert.equal(actual, expected)` — `composeOrderTotal` is an allowed import, and after Option B `estimateOrderTotal` lives in the *same* module path, so laundering through it is also an allowed import. An import-*path* allowlist has no export-level granularity to forbid "use the implementation to build the expectation." The genuinely oracle-free guard is the property test P2 (`inclusive ⇒ total===subtotal+fee`, which references no implementation) — that one holds. So coverage is saved by P2, but the **lint/ratchet (P12) is a speed-bump, not the guarantee the "FIXED" label implies**; the recurrence class M4 targeted is only partially closed. (The separately-declared ACCEPT-RISK is about *offline* generation; this inline-call path is a distinct, un-flagged residual.)

### N5 — LOW — `payment_events_idem_unique` NULL-defeat undercuts the "idempotency is free" claim
The proposal says the shared refund_due idempotency is "free, by the existing `payment_events_idem_unique (provider, provider_payment_id, type)` … at most one `refund_due` per payment, replay-safe" (§3.2). But `provider_payment_id` is nullable (`1790000000083:54`) and a UNIQUE constraint treats NULLs as distinct — so if any `status='paid'` payment ever carries a NULL `provider_payment_id`, **all three of L-A / L-C / L-D can each insert a `refund_due` row** (no conflict fires) → a duplicated, human-payable obligation in `owner/refunds.ts`. In practice a Plisio-paid payment has a non-null invoice id (the webhook matches on it, `:61-62`), so this shouldn't arise — but v2 now leans three concurrent writers on this one unique where v1's `deliveryCompletion.ts:137-143` had a single writer, so the invariant deserves an explicit non-null guard (or a partial unique) rather than the stated "free/guaranteed."

### N6 — LOW — Trigger GUC save/restore must correctly restore an UNSET tenant
The L-C/L-D per-row dance is `save current_setting('app.current_tenant', true)` → `set_config(...NEW.location_id...)` → insert → **restore**. On the sweep path the saved value is NULL/unset; restoring it via `set_config('app.current_tenant', <null>, true)` must land back on unset, not the empty string, or a *subsequent* GUC-armed statement in the same session/tx could mis-evaluate `nullif(current_setting(...),'')::uuid`. The proposal explicitly invites this attack ("restore is mandatory … breaker should attack this"); it is a real implementation footgun but bounded and testable. Flagging so the implementation pins it (save into a `text` var, restore with `set_config(..., true)` and verify `current_setting('app.current_tenant', true)` reads back NULL on the sweep path).

### N7 — LOW — DEP-1 transiently grants an authenticated *customer* connection tenant-wide scope
Setting `app.current_tenant = loc` on the customer's cancel tx makes every GUC-armed policy (orders `courier_tenant_update`, `payment_events`/`payments` `tenant_dual`, `courier_cash_ledger`, `delivery_trace`, `courier_shifts`) admit any row in that location for the tx duration. Not exploitable as specified (the tx runs only ownership-scoped statements), but it is a defense-in-depth downgrade: a customer principal briefly holds member-equivalent tenant authority, and any future statement added to that route/tx inherits it. Mitigation: keep the GUC window minimal (set immediately before, reset immediately after `updateOrderStatus`), never run a non-ownership-scoped query while it is set.

### N8 — INFO — L-C inverts the repo's trigger convention (throwing → non-throwing)
Every existing trigger on money-adjacent tables **throws** to enforce (`prevent_cash_mutation` `RAISE EXCEPTION`, `1780421100052:24-30`; the existing `orders` BEFORE-UPDATE trigger `1790000000066:38`). L-C must be the deliberate opposite — a swallowing AFTER trigger whose entire safety rests on never propagating an error. The swallow pattern is proven in app code (SAVEPOINT in `orderStatusService.ts:152-163`) but is novel *in a trigger* here; an implementer copying the existing throwing-trigger template would silently reintroduce C1. Call it out in the migration header and the P6b proof.

## Summary (6 lines)
1. v2 genuinely kills C1: the throwing money-INSERT is removed from the atomic sweep, replaced by a non-throwing AFTER-trigger whose `WHEN OTHERS` subtransaction provably catches the RLS `42501` rejection (repo already relies on this pattern) — one poisoned row can no longer wedge fleet-wide cancellation (P6b holds); the trigger fires on every terminal-status path and misses none.
2. The only C1 caveat is pre-existing and already flagged (DEP-2): the sweep's own un-swallowed `order_status_history` insert still wedges it post-B3 — money is decoupled, sweep-liveness-post-B3 is not, and both flips are gated.
3. v2 kills C2 in steady state (disjoint watermark ranges + the `assignment_id` unique make double-count structurally impossible), but the fix's correctness hinges entirely on one literal (N2): an authoring-time watermark `< prod deploy` reopens double-pay for the authoring→deploy window.
4. DEP-1 is confirmed safe: `orders` has a real GUC `FOR UPDATE` arm (`courier_tenant_update`), so P4b works under FORCE-RLS non-bypass exactly like the webhook precedent; the customer can only set the tenant of an order they own — no exploitable authz hole (one LOW defense-in-depth note).
5. New flaws are all sub-critical: MEDIUM watermark-choice (N2), MEDIUM per-row subtransaction cost the trigger imposes on the live fleet-wide sweep for every cancel (N3), MEDIUM over-sold M4 lint that an import-path allowlist can't actually enforce against inline laundering (N4); plus LOW NULL-defeat of the idempotency unique (N5), LOW GUC-restore footgun (N6), LOW customer transient tenant-scope (N7), INFO trigger-convention inversion (N8).
6. Verdict: **C1 CONFIRMED-FIXED** (money-coupling; sweep-B3 caveat is pre-existing/DEP-2), **C2 CONFIRMED-FIXED** (with the N2 watermark residual to pin at implementation), **DEP-1 CONFIRMED-FIXED/safe** — and the v2 design introduces **no new CRITICAL**, only 3 MEDIUM + 3 LOW + 1 INFO.
