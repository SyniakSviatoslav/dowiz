## 2.B Компоненти P&ID → інженерні вимоги (нормалізовано)

1. **Вузол приймання (§4.1)** = компілятор well-posed BVP. Вимоги: (a) схема YAML еталону з typed-полями Діріхле/Нейман/Робін; (b) детермінована перевірка повноти/несуперечності; (c) «адвокат двозначності» = дешевий LLM-прохід, advisory-only. REJECT до запуску = найдешевший гейт.
2. **Резервуар еталону (§4.2)** = WORM store + дві форми (текст + вектор). Вимоги: незмінність протягом прогону (hash-lock), диференційована подача по ролях, маркування службових розділювачів.
3. **Насос (§4.3)** = тактовий генератор: sync (2–3 оберти) / async (вежа) / drip. Mutex на стан.
4. **Тріада (§4.4)**: Генератор (T=0.7–1.0) → Критик (T≈0.1, бачить лише вихід+еталон) → Сепаратор (T=0, ДЕТЕРМІНОВАНИЙ КОД: CONVERGED/ITERATE/BRANCH/ABORT/BYPASS). Порядок: спершу зовнішній verify, потім внутрішні метрики.
5. **Теплообмінник (§4.5)** = ренормалізатор: дренаж мертвого контексту кожні K=2; інваріант H(out)≤H(in) при збереженні persistent claims.
6. **Редукційний клапан (§4.6)** = T-scheduler (annealing): gen 0.9→0.6→0.3; гілки {0.5,0.8,1.0}.
7. **Арматура (§4.7)**: diode якості (best checkpoint назавжди), запобіжники (max_iter, token-fuse ДО виклику, timeout, Δ-стагнація→BRANCH, рознос→ABORT, кавітація→відкат).
8. **Бак+чекпойнти (§4.8)**: scratch-буфер + політика {best, last2, forks}.
9. **Байпас людини (§4.9)**: рівно 3 тригери (пізня персистентна ознака; невиконуваний verify; ручна кнопка).
10. **Клапан видачі (§4.10)**: тільки best + журнал + вирізані маркери.
11. **Манiфольд (§4.11)**: N гілок від чекпойнта, спільний сепаратор, бритва Оккама як тайбрейкер, архів гілок для graft.

## 2.C Слабкі місця схеми (мій незалежний аудит)

1. **k ∈ [0.5,0.8] — неперевірене припущення.** Формула k ≈ g·c·(1−λ) — евристика без розмірностей; реальний k залежить від задачі. ПЛАН: k вимірюється, не задається; коридор — це аларм-межі, калібрувальний пакет дає розподіл k по класах задач.
2. **Cosine similarity до еталону як манометр — слабкий сигнал.** Embedding-подібність насичується (все "про те саме" ≈0.85+). ПЛАН: Sn доповнюється (а) детермінованим verify (тести/схеми), (б) claim-level покриттям (частка пунктів еталону, покритих persistent claims) — це і точніше, і дешевше.
3. **Ентропійний термометр — проксі не визначений строго.** ПЛАН: H_proxy = зважена сума (довжина контексту / ліміт, частка n-gram повторів, кількість dead claims не дреновано). Всі три обчислювані детерміновано.
4. **Схема мовчить про ідемпотентність/конкурентність** поза mutex. ПЛАН: event-sourced стан (kernel fold-патерн) робить кожен оберт відтворюваним: replay журналу = детермінована реконструкція (agentic_git вже це вміє для пам'яті).
5. **Схема не специфікує ОЦІНКУ якості критика (ReflectionAccuracy)** механічно. ПЛАН: калібрувальний пакет з навмисно-правильними випадками; метрика = частка already-correct, які критик НЕ зіпсував.
6. **BRANCH-правило** (Δ мала + fail двічі) може зациклитись, якщо всі гілки повертаються в той самий басейн. ПЛАН: гілки зобов'язані відрізнятись початковими умовами (T, роль, seed) І перевіряється дисперсія результатів гілок; нульова дисперсія = сигнал некоректного еталону → BYPASS.
7. **Відсутня явна модель вартості.** Токени = integer money: точний облік у мінімальних одиницях, без float (kernel/money.rs патерн 1:1).

## 2.D Що з L5-архітектури bebop СИЛЬНІШЕ за схему (підняти в план)

- **"Advisor proposes, kernel decides"** — у схемі сепаратор детермінований, але L5-гейт іде далі: LLM-пропозиція обрізається bounded envelope (max delta per tick) + ансамблева незгода ⇒ ігнор LLM. Це строго сильніше за "критик оцінює".
- **Field veto domінує L5-оптимізм** — червоні лінії задачі перевіряються незалежним детермінованим шаром, який LLM не може переголосувати. У схемі це лише hard-constraints Діріхле; у bebop це виконано як окремий шар з тестами.
- **Fail-closed freeze** (нестабільне поле ⇒ вся адаптація заморожена) — сильніша форма запобіжника, ніж ABORT: система деградує в READ-ONLY, а не вимикається.
- **Agentic-git пам'яті** — контент-адресована ланцюг снапшотів = чекпойнт-батарея З КРИПТОГРАФІЧНОЮ цілісністю (сильніше за §4.8).
- **Error-patterns auto-learning** — макро-контур (§12) вже частково існує як код: систематичні збої персистяться і матчаться наступними сесіями.

## 2.E Попередня архітектура синтезу (чернетка, уточнюється після звітів)

Шари цільової системи (кожен = вже наявний кодовий актив + добудова):
- L0 ФІЗИКА: детермінований kernel (Rust/WASM або TS): сепаратор, запобіжники, бюджет, T-scheduler, чекпойнт-дерево. [kernel decide/fold + resonator]
- L1 ПАМ'ЯТЬ: WORM-еталон + persistence-таблиця claims + agentic-git чекпойнтів. [LivingMemory + agentic_git — з ремонтом destructive tick → ATTIC-move]
- L2 ВИМІРЮВАННЯ: панель приладів (Δn, Sn, Vn, Ln, H_proxy, k-метр, ортогонометр) + Kalman-згладжування шумних сигналів + FFT по трендах. [bebop2 kalman/fft + eval-layer metrics]
- L3 АКТУАЦІЯ: генератор/критик як пплагіни (LLM-виклики) під bounded envelope; ансамбль-незгода ⇒ ігнор. [wiring.rs L5-gate патерн]
- L4 ОРКЕСТРАЦІЯ: машина станів прогону (INTAKE→…→DELIVER) + манiфольд гілок + вежа. [loops/*.yaml схема + tier-автоматизація]
- L5 МЕТА: міжпрогонна еволюція еталонів (Робін) з людським вето + error-patterns harvest. [librarian/loop-architect патерни + error_patterns.rs]
