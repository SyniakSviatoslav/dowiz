## 1. Confirmed first divergence + every confirmed bug (Method steps 2–3)

Both sides were instrumented to print FIPS 204 intermediates in the identical `STAGE <name> <values>`
format for a fixed seed (`0x01*32`), then diffed **in FIPS 204 order**. The Rust probe is the
`mldsa_diff_probe` test (compiled + run: `cargo test -p bebop2-core --release mldsa_diff_probe -- --nocapture`).

### FIRST DIVERGENCE — `KeyGen`'s `H` input omitted the `‖k‖l` domain-separation bytes

The very first derived value, `ρ`, differed — so **everything** downstream (matrix, secrets, keys, sigs)
was wrong. FIPS 204 Alg 6 line 1: `(ρ,ρ′,K) ← H(ξ ‖ IntegerToBytes(k,1) ‖ IntegerToBytes(ℓ,1))`.
bebop2 hashed only `ξ`.

| stage | C reference (ground truth) | bebop2 HEAD (before) |
|---|---|---|
| `keygen.rho` | `c4e999a2033fb0e19d9f88d62662838f682405572ca756db8bd96285c44df213` | `dbf77c8d79c80ad6b0763d42a3fe53f0f1dc030220c2e3fedc2aa903abfcad08` |

HEAD source (`bebop2/core/src/pq_dsa.rs:483`): `shake256(seed, &mut ext);` — missing the `k=6, l=5` bytes.

### BUG 2 — `ExpandA` used CBD instead of uniform rejection sampling (the operator's prime suspect, CONFIRMED)

Feeding the **reference** `ρ` into bebop2's `expand_a` (isolated probe `iso.expandA`) still produced tiny
center-binomial values instead of uniform NTT-domain coefficients:

| stage | C reference | bebop2 HEAD (fed reference ρ) |
|---|---|---|
| `matA[0][0]` (first coeffs) | `7077740 3948226 384905 4903644 3487570 …` (uniform in [0,Q)) | `0 -1 1 -1 0 0 0 -1 -1 -1 1 -2 …` (CBD, tiny) |

HEAD source (`pq_dsa.rs:234`): `a[i][j] = sample_poly_cbd(&buf, 0);`. FIPS 204 requires
`RejNTTPoly` (uniform rejection from SHAKE128, result already in the NTT domain). This is the root cause the
operator suspected — **confirmed by compiled diff.**

### BUG 3 — `ExpandS` used CBD instead of `RejBoundedPoly`, with the wrong nonce encoding

HEAD `expand_s` (`pq_dsa.rs:210,218`) used `sample_poly_cbd` with a 66-byte `ρ′‖r‖i` seed. FIPS 204 uses
`RejBoundedPoly` (SHAKE256, reject nibbles ≥ 9, map `4 − nibble` for η=4) keyed by `ρ′ ‖ IntegerToBytes(nonce,2)`
with `nonce = 0..ℓ−1` for s1 and `ℓ..ℓ+k−1` for s2. Fed the reference `ρ′`, bebop2's `s1[0]` diverged from the
reference's `s1[0]`. Confirmed by diff (`iso.expandS`).

### BUG 4 — `power2round` off-by-one at the rounding boundary

FIPS 204 `Power2Round`: `a1 = (a + 2^{d-1} − 1) >> d`. HEAD used `(f + half) >> d` (no `−1`). Isolated probe:

```
input a = 4096, d = 13:
  C reference / FIPS:  t1 = 0, t0 = 4096
  bebop2 HEAD:         t1 = 1, t0 = -4096      (STAGE iso.power2round(4096) t1=1 t0=-4096)
```

### BUG 5 (documented in HEAD, now fixed) — c̃ length 32 instead of 48, and non-FIPS packing

HEAD's own comments admitted it: `c̃ = 32B` (FIPS cat-3 requires `λ/4 = 48B`), `pk = 3104B` (FIPS 1952),
`sig = 3896B` (FIPS 3309), 13-bit t1 packing (FIPS 10-bit), 24-bit z packing (FIPS 20-bit), custom hint layout,
and 32-byte `tr` (FIPS 64). All of these prevented byte-interop even if the math had been right. Fixed to the
FIPS 204 §7 encodings (`polyt1/t0/eta/z/w1` pack, `pack_pk/sk/sig` with the reference bit-layout, `TRBYTES=64`,
`CTILDEBYTES=48`).

### Additional structural bug — coefficient-domain schoolbook instead of the NTT domain

Even with uniform sampling, HEAD multiplied via schoolbook in the coefficient domain. FIPS 204's `Â` (from
`ExpandA`) lives in the NTT domain; `t = A·s1 + s2` and `w = A·y` are defined through `NTT`/`NTT⁻¹`. A
schoolbook path with uniformly-sampled-as-coefficients matrix produces a **different** `t` (hence different pk
bytes) than the reference. Byte-interop therefore required a faithful NTT. The fix ports the reference's exact
Montgomery NTT (zeta table, `montgomery_reduce`, `invntt_tomont` with `f = 41978`).

## 2. The fix (the actual diff)

Montgomery NTT + `invntt_tomont`; `ExpandA = RejNTTPoly` (SHAKE128, NTT domain, nonce `(i<<8)+j`);
