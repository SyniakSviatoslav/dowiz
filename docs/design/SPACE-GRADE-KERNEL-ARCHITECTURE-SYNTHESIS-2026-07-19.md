## 0. The Constraint, Stated Precisely

The operator's directive is absolute: **zero external libraries — all in kernel.** The shipped kernel binary must have zero external runtime dependencies.

This is not a new philosophy imposed from outside. It is a hardening of the kernel's own pre-existing, partially-honored discipline. GROUNDED: the kernel already hand-rolls SHA3/Keccak-f[1600] from scratch per FIPS 202 (`kernel/src/event_log.rs`, no crypto crate anywhere), and already hand-rolls JSON escaping in the markov CLI wrapper (`kernel/src/bin/markov_attractor.rs`, deliberately no serde). The constraint asks the kernel to finish becoming what it already mostly is.

The line must be drawn precisely, and it will be drawn the same way every time it comes up in this document:

- **Counts against the constraint:** anything compiled into and shipped inside the kernel binary. `[dependencies]` in `Cargo.toml` that survive into release artifacts.
- **Does NOT count:** build-time/CI-time verification tooling that never links into the shipped binary — Kani, TLA+/TLC, dudect-style test harnesses, criterion benchmarks. These are exactly the same category as `cargo test` and clippy, which the kernel already uses today without anyone calling it "not zero-dep." Every external tool recommended below is explicitly labeled "CI-time, not linked" or "must be reimplemented in-house."

### 0.1 The Constraint's Own Live Violation — the Flagship "Upgrade a Working System" Case

GROUNDED (verified via `cargo tree -p dowiz-kernel` this session): `kernel/Cargo.toml` declares three real runtime dependencies — `regex = "1"`, `tracing = "0.1"`, `tracing-subscriber = "0.3"` (with `env-filter`) — pulling **25 unique transitive crates** into the build. Thirteen come from the tracing pair alone: proc-macro2, quote, syn, unicode-ident, tracing-core, tracing-attributes, tracing-log, once_cell, smallvec, sharded-slab, lazy_static, thread_local, cfg-if, nu-ansi-term, plus log, pin-project-lite, matchers; regex brings aho-corasick, memchr, regex-automata, regex-syntax.

Nothing here is broken. `tracing` works. That is precisely why it is the right headline: the operator asked that already-working systems not be exempt from re-examination, and this is the single biggest lever between the kernel's current state and its stated goal. A crate that hand-rolls Keccak-f[1600] rather than depend on a crypto crate, but links 13 transitive crates to emit a log line, has an inconsistency of principle, not of function.

GROUNDED (honest self-correction): earlier blueprint docs this session asserted kernel zero-dep status via a narrow check (`cargo tree -e no-dev | grep -c serde → 0`). That check was true — serde is absent — but it verified one grep target, not the dependency tree, and produced false confidence. The lesson generalizes: **verify the whole tree, not a token of it.** The falsifiable form of the zero-dep claim is `cargo tree -e no-dev` producing exactly one line (the kernel itself), enforced in CI.

PROPOSED remediation, in the kernel's own established pattern:

1. **Replace `tracing`/`tracing-subscriber` with a hand-rolled minimal span/log primitive.** The kernel needs: leveled log lines, optional structured fields, a span-enter/span-exit pair with monotonic timestamps, and an env-var level filter. That is a few hundred lines of std-only Rust — `std::io::Stderr` behind a `Mutex` or a lock-free ring (see §5, where the same buffer becomes the flight recorder), `std::time::Instant`, and the already-existing hand-rolled JSON escaping as the structured-field serializer. The precedent argument is decisive: this codebase already wrote a FIPS 202 hash function from scratch; a log formatter is not the hard case. Cite `esc()` in the markov CLI as prior art — "we already know how to do this ourselves."
2. **Audit the `regex` usage and replace it with explicit parsers.** PROPOSED (usage sites not enumerated this session — enumeration is step one): kernel-internal regex use is typically a handful of fixed patterns; each can become a small hand-written matcher, which is also more auditable and more obviously constant-behavior than a backtracking-free-but-opaque regex engine.
3. **Add the CI gate**: `cargo tree -e no-dev` must equal the kernel crate alone; any new `[dependencies]` line fails the build with a message pointing at this document. This converts the constraint from ethos to invariant.

## 1. Already-Built Systems: What Is Excellent, and the Concrete Upgrade Angle for Each

The operator's instruction — nothing is exempt because it works — is applied here system by system. Every "upgrade angle" below was identified against real code this session, not invented.

### 1.1 `order_machine.rs` — the 12-State Order FSM (~1200 lines)

GROUNDED-excellent: `has_cycle`, `cyclomatic_number`, `topological_order`, `reachable`; `spectral_radius` upgraded from computed value to **proven constant** `FSM_SPECTRAL_RADIUS: f64 = 0.0` via Perron-Frobenius (DAG ⇒ nilpotent adjacency ⇒ ρ = 0), with the old power-iteration demoted to a test-only oracle cross-checked at 1e-12. A `FSM_GOLDEN_SIGNATURE` drift gate pins the entire graph shape. Zero float in the hot path, zero I/O, zero `unsafe`. This is already space-grade in spirit: a proof replacing a computation is exactly the determinism ladder this synthesis advocates.

GROUNDED upgrade angles (both already identified against source):
- `all_edges()` is recomputed on every call from three different functions, allocating a small `Vec` each time. Replace with a **`const` adjacency bitmask computed once at compile time** — 12 states fit in a `u16` row, the whole graph in `[u16; 12]`. This makes the module genuinely zero-allocation and moves graph shape fully into the type/const system, where the golden-signature gate can pin it at compile time rather than test time.
- `idx_of` is defined three times (module level plus two local closures) — a minor but real one-source-of-truth violation; collapse to the module-level definition.

PROPOSED extension: once the adjacency is `const`, `topological_order` and `reachable` become candidates for `const fn` evaluation, making FSM structural properties *compiler-checked* facts. Unmeasured effort; likely small.

### 1.2 Markov Attractor Detector (`kernel/src/markov.rs` + CLI wrapper)

GROUNDED-excellent: the analysis engine (`analyze_detailed`, `Verdict`) was moved from a standalone binary *into* the kernel — the correct direction under "maximal in-kernel computation" — leaving a thin 130-line CLI (`markov_attractor.rs`) doing stdin → hand-rolled JSON, with a `--selftest` parity harness and a deliberate no-serde stance.

GROUNDED upgrade angle: the hand-rolled `esc()` escaping is applied manually per field, so every new output field is a fresh opportunity to forget escaping. A **tiny embedded formatter** — a ~50-line `JsonWriter` that owns escaping at the write boundary (`field(name, value)` methods that cannot emit unescaped bytes) — removes the per-field repetition risk without adding any dependency. This is also the "make the illegal state unrepresentable" pattern (§1.5) applied to serialization: an unescaped write should be inexpressible, not merely avoided. PROPOSED: the same writer then serves the new logging primitive (§0.1), consolidating two hand-rolled JSON sites into one.

### 1.3 `token_bucket.rs` — the Ready, Quantified, Gated Upgrade

GROUNDED-excellent: `new`/`try_acquire`/`available` (no `release` — confirmed this session, correcting a stale citation earlier today), `std::sync::Mutex<Inner>`, monotonic clock only, poison recovery via `unwrap_or_else(|e| e.into_inner())` — sound because the inner state is POD with no cross-field drop invariant. The clock is sampled outside the lock, worth ~15% throughput at 8-way contention per its own doc comment. Careful, honest, already tuned.

GROUNDED upgrade — the clearest "upgrade the working thing" candidate in the whole kernel: a **lock-free single-`AtomicU64` GCRA** (Generic Cell Rate Algorithm) variant was already implemented and measured at **2.5–3.6× faster**, then deliberately left `NEEDS-OPERATOR-DECISION`-gated. The gate is honest and correct: this is an algorithm swap on a security primitive (rate limiting is an abuse-control surface), and such swaps deserve explicit sign-off, not silent adoption. The recommendation is therefore not "ship it" but "**put it through the §4 hardening checklist and present it for the decision**": differential oracle against the mutex implementation over a large randomized schedule, plus a loom-style or model-checked interleaving argument (Kani can bound-check the atomic transition function — CI-time, not linked). That converts an operator judgment call into an operator judgment call *with proof attached*.

### 1.4 `event_log.rs` — the Existence Proof for the Whole Constraint

GROUNDED-excellent: `MeshEvent{prev, actor_pubkey, actor_seq, payload}` with `event_id() = sha3_256(prev || actor_pubkey || actor_seq || payload)` — a genuine hash chain — implemented on an **own from-scratch Keccak-f[1600]/SHA3-256 (FIPS 202), zero crypto-crate dependency**. This is the codebase's standing proof that "hand-roll it, don't depend on it" works at real cryptographic complexity, and it is cited as precedent everywhere this document proposes removing a dependency. The typed `StoreError`/`CommitError` split — "Law rejected, never retry" versus "accepted but not durable, must alarm" — is exactly the fault taxonomy aerospace systems demand: distinguish invariant violation from resource degradation at the type level.

GROUNDED gap, flagged honestly: `MemEventStore` (non-durable) is the only **confirmed-wired** default found this session. A durable `FileEventStore` exists and is documented in `hydra.rs`, but whether any real hub binary actually constructs the durable path is **unconfirmed — PROPOSED-to-verify, not asserted as a bug**. The verification is cheap and falsifiable: grep every binary's construction site for `FileEventStore`; if absent, the "genuine hash chain" currently evaporates on process exit in production, which would be the single most consequential quiet finding in this document. Verify before concluding.

### 1.5 Layer F Decision-Import Gate — the House Standard for Illegal-State-Unrepresentability

GROUNDED-excellent (`kernel/src/decision/mod.rs` + `decision/import.rs`, commit `1e1a7db09`, 344 lines, wired, 6 tests green including 5 adversarial paths): `import_unit()` runs six ordered checks — size, sha3 integrity, instance-set-hash pin, **full independent replay of every harvested case against the local oracle** (any mismatch → `ReplayDisagreement`, nothing persisted), epoch-no-downgrade, lineage-parent-must-resolve. The replay check structurally closes the "optimistic trust" bug class that the entire P06 key_V effort existed to prevent — imported decisions are re-derived, never believed.

The detail to canonize: the money gate is enforced **structurally** — `DecisionUnit::new` *forces* `Pending` for money-gated units in the constructor, rather than exposing a checkable-and-therefore-bypassable boolean. This is "make the illegal state unrepresentable," and it should be named the house standard. PROPOSED extensions of the pattern: the JSON writer of §1.2 (unescaped output inexpressible); event-log commit paths (an event without a valid `prev` linkage should not construct); and any future breaker state machine (§2 — a tripped breaker's "allow" state should not exist as a value).

PROPOSED generalization: the six-ordered-check pipeline is a reusable shape — see §3's Sequencer parallel — and should be extracted as the kernel's template for every multi-stage admission gate, so the next gate inherits ordering, short-circuit, and typed-rejection semantics instead of reinventing them.

### 1.6 The NTT Wire-In — Today's Flagship, Already Done

GROUNDED (verified against the actual pushed commit `cf1fc90` on `openbebop/main`, `bebop2/core/src/pq_kem.rs` — not a stale local checkout): `ring_mul()` is the sole multiply call site across `keygen_internal_prod`/`kpke_encrypt`/`kpke_decrypt` (4 sites), calling the NTT with the schoolbook cross-check under `debug_assert_eq!` — compiled out of release, so production runs full-speed NTT while every debug/test run continuously oracle-checks it. A real dudect-style `ntt_ct_gate` (Welch's |t| < 4.5) covers both ring-multiply and the FO implicit-rejection decaps path, with a **self-test proving the gate catches a deliberately planted leak** — the gate is validated, not a rubber stamp. `red()` (mod-q reduction) is branch-free *because the dudect gate caught a real timing leak in the branching version*.

This is not a proposal in this document. It is the completed, today-dated instance of the exact philosophy this synthesis exists to formalize. Its job here is to serve as the template for §4 — the pattern, generalized and made mandatory, so the next algorithm swap does not depend on whoever performs it independently rediscovering this discipline.

## 2. The Largest Structural Gap: `kernel/src/breaker/` Does Not Exist

GROUNDED, stated plainly: there is **no** `kernel/src/breaker/` directory and zero references to one anywhere in the kernel. This morning's swarm-safety "Blueprint A" (Telemetry & Safety circuit breaker) is a real, detailed, reviewed, buildable design document — **and it is paper, not code**.

This matters more than any single upgrade angle above, because everything else in this synthesis assumes a fault-containment layer that currently is not there. The event log can classify failures (`CommitError` = "must alarm") but nothing *receives* the alarm and trips. The FDR of §5 records; a breaker *acts*. Building Blueprint A is the single largest structural item on the list, and — PROPOSED — it should be built under the §1.5 house standard from day one: breaker states as types where "tripped-but-permitting" is unconstructible, transitions as a small FSM that reuses the `order_machine.rs` proof machinery (golden signature, proven-DAG-or-proven-cycle-set), zero external dependencies, which Blueprint A's design does not require anyway.

## 3. Aerospace Patterns Worth Adapting (Never Adopting as Dependencies)

GROUNDED research, with the constraint applied to each:

Its component/port architecture — typed ports between active/passive/queued components, a `Health` component doing periodic watchdog pings, a `Sequencer` executing command sequences — is directly relevant. Recommendation: **reimplement the pattern in-kernel, do not link the framework.** PROPOSED concretely: (a) a `Health`-equivalent — each kernel module exposes a cheap `heartbeat()` invariant probe, polled by one supervisory loop that feeds the future breaker; (b) the `Sequencer` shape as the formal model for generalizing the decision-import gate's ordered-check pipeline (§1.5) to all multi-stage gates.
- **cFS/cFE** — real, actively maintained (Goddard). Its software bus (typed pub/sub between apps), time services, event services, and validated uploadable config tables are the canonical decomposition of a flight kernel. PROPOSED audit: check what the dowiz kernel uses for internal module-to-module messaging today — if it is ad-hoc function calls with implicit coupling, the typed-message-bus discipline (in-house, zero-dep) is the upgrade; if messaging is already minimal and direct, do not add a bus for ceremony. This audit was not performed this session; do it before deciding.
- **Open MCT / GMAT** — real and active, but telemetry visualization and mission analysis respectively; low direct relevance to kernel architecture. Noted; no connection forced.
- **meshNetwork** — real repo, but dormant (46 commits, zero releases/issues/PRs), and its "flight-tested on small-UAS swarms" claim appears only in its own README with **no supporting citation found — unverified**, however tempting for a delivery-adjacent project. Not a precedent this synthesis relies on.
- **Hermit OS** — real and actively committed, single-address-space unikernel (app links against kernel at ring 0, no context-switch boundary). Honestly research-grade: a 2024 RWTH security thesis found "faulty implementation of some of the most basic security mechanisms." **Do not adopt.** Extract only the architectural idea — minimize or eliminate kernel/user boundary crossings in the dowiz kernel's own process model — as in-house inspiration. PROPOSED, unmeasured relevance until the kernel's deployment shape (bare process vs. VM) is pinned.
- **Wasmtime fuel** — real production API (`Config::consume_fuel`, `Store::set_fuel`, trap on exhaustion), with a strong precedent: Shopify Functions run untrusted merchant code under a hard 11M-instruction budget (error literally `RunOutOfFuel`). **Do not adopt Wasmtime.** PROPOSED: reimplement the *fuel-budget pattern* natively — any kernel path that executes less-trusted or agent-supplied logic carries an explicit, pre-committed step budget with a deterministic trap on exhaustion. This also serves determinism: bounded execution is a precondition for the temporal partitioning of the next section.

### 3.1 The Genuine Differentiator: an ARINC-653-Style Two-Level Scheduler in Rust

GROUNDED: ARINC 653 is the real avionics standard — an APEX API, MMU-enforced **spatial** partitioning (one partition's fault cannot corrupt another's memory), and **temporal** partitioning via a two-level scheduler: a fixed cyclic major time frame divides guaranteed slices among partitions; within a slice, ordinary priority-preemptive scheduling runs. Open implementations exist (RTEMS+AIR-II, POK) but **all are C/C++ — no ARINC-653-compliant Rust RTOS exists anywhere** (verified this session; a real gap, not an oversight).

PROPOSED: a from-scratch, zero-dep, Rust-native two-level partitioning scheduler inside this kernel would be a **first**, not a reimplementation. And the kernel's own primitives map onto it unusually well: temporal slices are token buckets with a proven refill law (§1.3's GCRA is literally a cell-rate scheduler); partition admission is a §1.5-style structural gate; partition health is the F´-pattern heartbeat; slice-exhaustion is the fuel trap of §3. Spatial partitioning is the honest hard part — full MMU enforcement means OS-level or bare-metal work; a nearer-term Rust-native approximation is process-per-partition with the kernel as supervisor, upgrading later. Scope honestly: this is a large arc, operator-gated, sequenced after §2's breaker — but it is the one item in this document where the kernel would be creating precedent rather than following it.

## 4. Formalizing "Kernel-Level Hardening" as a Standing, Mandatory Discipline

GROUNDED: the philosophy — exhaustive-proof-over-sampling, oracle-based differential fuzzing, binary/assembly auditing after compiler bumps, dudect/PMU instrumentation — is a coherent synthesis of independently real, well-precedented techniques, not one named industry methodology. dudect and ct-fuzz are real tools; differential fuzzing against a simple reference oracle is literally BoringSSL's own practice (`BORINGSSL_UNSAFE_FUZZER_MODE`); post-compiler-bump binary auditing is a real, recently formalized problem ("Breaking Bad: How Compilers Break Constant-Time Implementations," arXiv:2410.13489, with real incidents of `-O` breaking constant-time code in NIST PQC finalists Kyber and HQC — directly relevant to this kernel's own ML-KEM work). PMU-based verification is the weakest-precedented element — more studied as an attack vector than deployed as routine CI — and is graded accordingly below.

The critical point: **this kernel does not need to adopt the philosophy — it already built the flagship instance today** (§1.6). What is missing is that it exists as heroics, not as process. PROPOSED: a standing, documented, CI-enforced checklist — any new or changed algorithmic hot path (crypto or not: graph algorithms, scheduler math, GCRA arithmetic all qualify) must ship with:

1. **An oracle** — exhaustive where the input space permits (the FSM's 12 states permit it), or a large randomized corpus differentially checked against a simple reference implementation (the NTT's schoolbook, the token bucket's mutex version), with the reference retained forever as a test-only crate-internal module.
2. **A dudect-style gate** where secret-dependent timing is conceivable — including a planted-leak self-test proving the gate itself works, per today's `ntt_ct_gate` standard. (CI-time harness, not linked.)
3. **Debug-mode differential cross-check** (`debug_assert_eq!` against the oracle) compiled out of release — continuous verification at zero production cost, per today's `ring_mul` standard.
4. **A binary/assembly spot-check on every compiler version bump** for the branch-free paths — grounded in the arXiv:2410.13489 incident class. PMU counters remain optional/experimental, honestly labeled as the least-precedented element.

The deliverable is one markdown checklist plus a CI job that fails when a diff touches a designated hot path without the corresponding artifacts. Cheap to institute; its value is that the *next* NTT-class change gets today's rigor by default.

## 5. Flight Data Recorder via the pstore/ramoops Pattern

GROUNDED precedent: Linux's **pstore/ramoops** — a reserved RAM ring buffer, written continuously, surviving reboot, read back post-crash through a pseudo-filesystem. This is the concrete, named, buildable mechanism; it replaces the vague "black box recorder" metaphor.

PROPOSED, zero-dep and buildable now: a kernel-owned ring buffer (fixed-size, preallocated, lock-free single-writer or the §0.1 log primitive writing into it) capturing the last N seconds of structured events — state transitions, breaker trips, `CommitError` alarms — with three durability tiers: (a) in-process ring (crash-within-process loses it), (b) `mmap`-backed file ring (survives process death — pure std, no dependency), (c) genuinely reserved RAM region (survives reboot; requires host cooperation — kernel cmdline `memmap` reservation or actual ramoops on Linux hosts; PROPOSED-to-scope, host-config-dependent, not pure userland). Tier (b) is the honest immediate target: on any unclean restart, the kernel's first act is reading back its own last moments and emitting a post-mortem event into the (durable — see §1.4's verification) event log. The synergy is deliberate: the hand-rolled tracing replacement of §0.1 and the FDR share one buffer — the logger *is* the flight recorder.

## 6. Software TMR / SIHFT — Honestly Scoped, PROPOSED Only

GROUNDED: software TMR against single-event upsets is real and established under the name SIHFT (Software-Implemented Hardware Fault Tolerance) — real literature (Nicolescu & Velazco 2005, IEEE/ACM), with >99% SEU tolerance demonstrated on COTS hardware in cited work.

GROUNDED caveat, carried in full and not oversold: identical-replica software TMR reliably protects only against faults confined to state **private to one replica**. If all three replicas share physical silicon — same die's cache, ALU, memory controller — a single SEU in shared hardware can corrupt all three simultaneously. Real SEU defense requires physically separate COTS units or hardware ECC/lockstep alongside software voting. Pure single-chip software TMR is a **partial** defense.

PROPOSED, narrowly: triple-compute-and-vote for the kernel's few most critical pure computations only — event-id hashing at commit, money-gated decision evaluation — where the pure-function, no-I/O shape of the existing code (§1.1's zero-I/O discipline generalizes) makes triplication nearly free to implement, and where a vote-mismatch is treated as a §2 breaker trip plus §5 FDR entry rather than a claim of SEU immunity. On ECC-RAM Hetzner hosts the residual value is modest — this is a defense-in-depth option for the operator to weigh, not a recommendation to blanket the kernel in it.

## 7. CI-Time Formal Verification: Kani and TLA+ (Zero Runtime Footprint)

Both are **CI-time, not linked** — the same category as `cargo test`, fully compatible with the constraint.

- **Kani** — GROUNDED: real, production-grade CBMC-backed bounded model checker for Rust MIR (panics, overflow, assertions; unbounded properties via contracts), run as `cargo kani` in CI by AWS for Firecracker and s2n-quic, and by **Hifitime, an aerospace time library** — direct aerospace precedent. PROPOSED wiring, cheapest-first: prove panic/overflow-freedom for the hand-rolled Keccak permutation, the `order_machine` graph algorithms, the NTT/`red()` arithmetic, and the GCRA atomic transition (§1.3). This catches the fault class the exhaustive-oracle methodology does not specifically target — arithmetic edge conditions and hidden panics — and slots directly into the §4 checklist as item 5.
- **TLA+** — GROUNDED: real, in continuous industrial use (AWS: DynamoDB/S3/EBS since 2011, still used alongside Dafny/Kani/Lean today). The closest genuine cheap win of the four verification approaches researched: a scoped spec is days-to-weeks, not months. PROPOSED target: the decision-import pipeline (`kernel/src/decision/`) and the order FSM — both hand-built state machines with real invariants — model-checked (TLC) for deadlock/livelock freedom and for the epoch-no-downgrade and replay-before-persist invariants at the abstract level, alongside the Rust that implements them.
- **Not recommended:** MIRAI — GROUNDED: archived/dead (Meta disbanded its sponsor, Aug 2024). **seL4** — GROUNDED: the genuine gold standard for spec→C→binary proof (~20 person-years, Isabelle/HOL, no Rust path), cited here as a methodology exemplar (spec-then-refine) only — and note the nuance precisely: seL4's base guarantee explicitly does **not** cover timing/side channels; seL4-style verification would not catch what the dudect gate catches. Different guarantee classes; the kernel needs both, and currently has the dudect one. **Lean4/mathlib and the AI-prover ecosystem** (LeanDojo-v2, Goedel-Prover, DeepSeek-Prover-V2, LeanInteract) — GROUNDED as real, but aimed at AI-generated mathematical proofs on competition-style benchmarks, not systems verification; honestly a large from-scratch investment with no existing leverage here. One caught citation error recorded for the discipline's sake: SorryDB's real paper is arXiv:2603.02668, not the 2602.24273 originally cited — verify, don't trust the citation.

## 14. Wave Propagation Moved Into the Kernel — One Laplacian, Two Representations, Parity-Pinned

**(a) The two implementations, both real.** GROUNDED (verified this session): `kernel/src/spectral.rs:645` exports `laplacian(adj: &[Vec<f64>]) -> Vec<Vec<f64>>` — the **dense** graph Laplacian L = D − A, materialized as a full matrix. That form is not a naïveté: `charpoly`, `eigenvalues`, `algebraic_connectivity` and the rest of the §11/§12 functional family require matrix or matrix-vector access for eigendecomposition; a one-time spectral analysis legitimately pays for the N×N object. Separately, GROUNDED: bebop-repo's `crates/bebop/src/field_physics.rs` (1148 lines, confirmed real and current this session) implements `adjacency(n, edges) -> Vec<Vec<usize>>` (line 140) and `hop_distances` (line 154) — integer neighbor/hop data, never a matrix — and `step_wave` (line 334), whose own comment (line 330) names its coupling term the "bulk graph Laplacian L_graph," computed **matrix-free**: Lu applied per node as a neighbor-tensor-mean, Σ over neighbors of w(u_j − u_i), inside the wave-equation integrator (MÜ + ΓU̇ + c²LU = S). Never materializing L is the standard, correct choice for a real-time PDE stepping loop, where forming a dense matrix every tick would be pure waste.

**(b) The verdict, per this document's own §10/P2 audit.** These are almost certainly the **same mathematical operator** — the graph Laplacian acting on a field over a shared vertex set — in two representations serving two genuinely different, stated performance needs: dense form for one-time spectral analysis, matrix-free form for per-tick stepping. P2 explicitly permits divergence for a stated, falsifiable performance reason — *but only parity-pinned*, so the two forms cannot silently drift. GROUNDED gap, phrased carefully: no such parity test was **found** this session. The search was not exhaustive, so this is "not found," not "proven absent" — but the burden now runs the P2 direction: until a parity pin is cited by file:line, the two Laplacians are an unpinned divergence, the exact failure shape §10/P2 names.

**(c) The plan — operator-confirmed, and the concrete instance of "all possible computation in the kernel" for this subsystem.** PROPOSED, zero-dep, buildable now: the dowiz kernel gains its own **matrix-free Laplacian-action primitive** — a new function in `spectral.rs` (or a sibling module), e.g. `laplacian_action(neighbors, field: &[f64]) -> Vec<f64>`, computing Lu directly without materializing L — proven correct by a parity test against the existing dense `laplacian()`: same graph, same field vector, Lu computed both ways, exhaustive over small graphs plus a large randomized corpus, matching to float epsilon. This is precisely the §4 checklist's oracle discipline (the dense form *is* the simple reference implementation, retained forever), and it gives the kernel real wave-propagation capability built on the same provably-equivalent operator as its existing spectral analysis — not a second silent implementation, and not physics left siloed in bebop-repo.

**(d) The one open choice, named and not decided here.** Bebop's `step_wave` serves as the **design template** for the matrix-free technique — not a literal dependency. The §0 zero-external-library constraint plus the different-repo boundary rule out direct code sharing without an explicit decision. Two honest options: (i) dowiz reimplements the operator independently, with its own parity proof as above; (ii) the operator explicitly decides to vendor/duplicate bebop's proven implementation into dowiz, under the **same** parity-test discipline. Both are legitimate; (i) maximizes independence, (ii) reuses code already exercised by a live integrator. Per this document's standard for algorithm-swap-class choices (§1.3), the choice is marked `NEEDS-OPERATOR-DECISION` — presented with evidence, not made unilaterally.

## 16. Closing the Loop — a Deterministic Autonomic Layer, Not Machine Learning

**(a) Sensing is excellent; action is absent.** GROUNDED: the kernel's diagnostic layer is real and strong — the Markov attractor detector (`kernel/src/markov.rs`, `analyze_detailed`, `Verdict`: HEALTHY/LIMIT_CYCLE/STRANGE_ATTRACTOR, computing entropy, escape-mass, Foster-Lyapunov drift, SLEM) and `spectral.rs`'s `DriftClass`/`classify_drift` (Damped/Resonant/Unstable), which §10/P3 already cites as literally control-theory stability classification, built. And both diagnostic layers are advisory/fail-open per §1.2 and the wider session's hook-wiring research: they inform; nothing acts. The kernel can diagnose a limit cycle in its own behavior and then continues doing exactly what it was doing.

**(b) The answer — operator-confirmed, and the ML-rejection is a conclusion, not an open question.** Under this document's own hard constraints — §0's zero-external-dependency law and §10/P6's rule that no kernel effect may be a non-reproducible function of its cause — a genuinely "autonomous" mechanism here **cannot be machine learning** or any other opaque adaptive technique. Learned weights are exactly the non-reproducible cause→effect mapping P6 outlaws, before the dependency question even arises. What fits, completely, is classical, deterministic, **bounded adaptive control** — gain-scheduling in the control-theory sense: a small set of proven control laws mapping an already-classified drift state (the existing `Verdict`/`DriftClass` outputs) to an adjustment of an already-named, already-owned rate constant — `token_bucket`'s refill rate (§1.3), a retry/backoff cadence (§10/P3's Vibration). Three properties are non-negotiable: (i) every adjustment stays within a **pre-proven stability bound** — the same CFL-style bound §10/P3 requires before any rate reaches an integrator — and an adjustment that could leave the proven-stable region must be *inexpressible*, not merely avoided, per the §1.5/§10-P4 house standard; (ii) every adjustment is logged to the §5 FDR as a first-class event — a self-tuning system that does not log its own tuning is unauditable; (iii) every control law is itself a checkable equation per §10/P1 — never learned, never opaque. This is the source material's "closing the loop" (Monitoring vs. Control) implemented with only primitives the kernel already owns: zero new dependencies, zero new nondeterminism.

**(c) The shape, PROPOSED and explicitly sequenced.** A small module — `kernel/src/autonomic.rs`, or folded into the §2 breaker's own design; that placement is an open design choice, PROPOSED, not decided — that (1) subscribes to `Verdict`/`DriftClass` outputs, (2) holds a small, explicit, auditable table of {classified-state → bounded adjustment} laws, one per tunable constant, (3) applies adjustments only through types that make out-of-bound values unconstructible where possible, (4) writes every adjustment to the FDR, and (5) is covered by the §4 checklist from day one — an adjustment law is precisely the algorithmic hot path that needs an oracle/differential test, headlined by the property test that **no sequence of adjustments, under any classification sequence, can push a rate outside its proven-stable bound**. Sequencing is strict: this lands **after** §9.9's breaker exists. The breaker is the binary emergency net; the autonomic layer is the continuous, graduated tuning layer — they compose, they do not replace each other — and the autonomic layer's most severe responses should very plausibly route *through* the breaker rather than acting unilaterally at the extreme end. Sensing (§1.2), record (§5), emergency stop (§2), graduated control (here): that is the complete closed loop, every stage deterministic.

## 20. Optical/Pixel Context Compression — a Real Technique for Living-Memory Archival, and a New Category of Dependency Question

**(a) The research, verified.** GROUNDED (verified this session via direct paper fetch, not secondhand): the render-context-as-image idea is exactly **DeepSeek-OCR: Contexts Optical Compression** (arXiv:2510.18234, Oct 2025, open-sourced) — a "DeepEncoder" renders a page into a small set of vision tokens, decoded by a 3B-MoE text decoder. The published numbers are the whole evidence package: **under 10× compression, 97% OCR decoding precision; at 20×, ~60%** — explicitly lossy, degrading predictably rather than catastrophically as the ratio climbs. A second, independently fetched paper — "Text or Pixels? It Takes Half" (arXiv:2510.18279) — confirms the claim beyond OCR: on RULER long-context retrieval and CNN/DailyMail summarization, ~50% fewer decoder tokens at 97–99% accuracy, degrading faster on smaller (7B) models and on dense text. A cluster of adjacent 2025–2026 work (SeeTok, Glyph, VIST2, LensVLM, VTCBench, VTC-R1) was surfaced but **not** independently fetched — corroborating-but-unverified signal that this is an active research direction, not one isolated paper. The honest mechanism limit: vision tokens are not universally denser than text tokens — a fixed-size image maps to a fixed vision-token budget regardless of how much text is rendered into it, so **rendering resolution/density is the compression knob**, and the technique strictly requires a vision-capable model on the receiving end; a text-only model cannot use it at all. GROUNDED local feasibility: the 3B decoder already runs locally via GGUF quantizations — llama.cpp (PR #17400) and Ollama (`ollama run mike/deepseek-ocr`) — real and confirmed, not lab-only.

**(b) The fit, PROPOSED — not built, not decided.** The natural consumer is §15's living-memory subsystem: older or lower-priority retrieval/index content — exactly what the P95 persistence layer (§15(c), item 20) will need to store durably — archived as rendered images rather than raw text, trading a bounded, *measured* accuracy loss for real storage/transmission savings. The shape composes with what exists: P95 decides durability; this decides representation for the archival tiers only.

**(c) Two tensions, named plainly, neither papered over.** First, **this is a different kind of dependency question from everything else in this document.** §0's line — drawn the same way every time — governs crates compiled into and shipped inside the kernel binary. A vision-capable model artifact is *weights*, not a Cargo dependency; §0 as written simply does not address model-weight dependencies, and this document declines to resolve the question by stretching §0's wording in either direction. Whether the constraint's spirit extends to model artifacts is a genuine scope question only the operator can rule on. Second, **the technique is lossy, and the plane boundary is absolute**: under §10/P6's plane-ranking discipline, anything on the determinism planes — idempotency keys, signatures, gate verdicts, any hash surface — is categorically excluded from optical compression; 97% fidelity on a signature is 100% failure. The only legitimate candidates are the same forensic/display/archival plane §10/P6 already assigns logger timestamps to — rankings, archival prose, display text — where bounded, measured lossiness is acceptable by construction. Both tensions make the whole item `NEEDS-OPERATOR-DECISION`, with the numbers in (a) as the evidence package.
