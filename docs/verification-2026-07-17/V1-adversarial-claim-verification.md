## 5. Degrade-closed budget — BROKE IT (real, unmitigated degrade-OPEN hole)

**What I tried:** the module docstring calls "degrade-closed" the "load-bearing word." I looked
for an input the ceiling gate must reject but does not, on the `budget.rs` spend path.

**What I found — an unchecked-`f64` slip that permanently opens the ceiling:**
- `ComputeBudget::debit` (`budget.rs:113-120`) gates on `self.spent + amount > self.ceiling`;
  `BudgetedJobPort::submit` (`:157`) gates on `b.spent() + job.estimate > b.ceiling()`. Both are
  raw `f64` compares. `Job.estimate: f64` (`:51`) is caller-supplied and **never validated** —
  `grep is_finite|is_nan|>= 0.0 budget.rs` → **NONE FOUND**.
- **NaN attack:** `estimate = f64::NAN` ⇒ `spent + NaN > ceiling` is `NaN > ceiling` ⇒ `false`
  ⇒ the gate does **not** refuse ⇒ `debit` runs `self.spent += NaN` ⇒ `spent` is NaN forever.
  Every subsequent debit then computes `NaN + x > ceiling` ⇒ `false` ⇒ **passes** — the ceiling
  is permanently bypassed. One poisoned submit converts a degrade-*closed* budget into a
  degrade-*open* one for the rest of its life.
- **Negative attack:** `estimate = -1e30` ⇒ `spent + (-1e30) > ceiling` ⇒ `false` ⇒ debited ⇒
  `spent` driven hugely negative ⇒ effectively unbounded headroom thereafter.
- **Contrast (why this is a real finding, not pedantry):** the *integer* money law is genuinely
  safe — `domain.rs:74-76,138-141` `compute_order_total` uses `checked_mul`/`checked_add`
  (BP-17), and `money.rs:269-287` `apply_tax` computes in `i128` with `i64::try_from`
  range-checks. The slip exists **only** on the `f64` compute-spend path, which has no such
  guard. P-C §2.1 lists "budget" under *genuine zero-authority* ("`debit` refuses past ceiling")
  — that classification is **false for NaN/negative inputs**: the arithmetic does not refuse.

This is a live, unmitigated degrade-open weakness in shipped code — no blueprint claims it closed,
and none closes it.

## 6. `token_bucket.rs` poison-cascade — PARTIALLY FIXED (migrated, not eliminated)

**What I tried:** confirm the `.lock().unwrap()` poison-cascade V7/P-H flagged is still in
`token_bucket.rs`, and sweep for the same pattern elsewhere in the kernel.

**What I found — fixed in the named file, alive in its sibling:**
- `token_bucket.rs:78` and `:101` now use `.lock().unwrap_or_else(|e| e.into_inner())`, with a
  detailed A6 (P-H) comment: `Inner` is two POD fields with no panic-spanning invariant, so
  recovery is sound and a poisoned bucket degrades-closed. The flagged bug is **closed here.**
- **But the identical raw pattern survives in `budget.rs`:** `self.budget.lock().unwrap()` at
  `:147` (`spent()`) and `:156` (`submit()` — the spend gate). `Mutex<ComputeBudget>` has no
  `into_inner` recovery. If any thread panics while holding that lock (e.g. the NaN path in
  claim 5 does not panic, but any future panic point does), **every** subsequent `submit`/`spent`
  cascades `PoisonError` — the exact DoS the token_bucket A6 hardening was written to prevent,
  left in place one module over, on the compute-spend surface.

Claim held on the letter (`token_bucket.rs` is fixed) but the risk was **relocated, not
eliminated** — the P-H remediation did not sweep `budget.rs`.
