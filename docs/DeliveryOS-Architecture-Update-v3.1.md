## 0. Головний зсув оптики

Принцип «надійність через мінімалізм — менше рухомих частин, що падають о 19:00» лишається аксіомою №1. Але аудит показав: окремі рішення були майже бездоганні, а **сума рішень переростала принцип** (повна horizontal-SaaS-візія замість MVP). Тому ключове рішення сесії — **різати скоуп, а не архітектуру**: відокремити MVP-ядро від платформа-візії й перенести візію в `docs/planned-features/`, лишивши схему БД повною (нуль рантайм-ваги), але не запускаючи код/воркери/security-поверхню відкладених фіч.

Друга наскрізна теза: **легкість досягається не діями зараз, а збереженням швів** (dependency-inversion інтерфейси: `MessageBus`, `QueueProvider`, `ThemeRenderer`, `GeocodingProvider`). Зміна вендора/масштабу має бути свопом за інтерфейсом, не рерайтом.

## 1. MVP-ядро vs Planned-features (рішення про скоуп)

**MVP-ядро (будуємо зараз):**
- Клієнт: Меню (`/s/:slug`) → Кошик → Checkout → Статус замовлення
- Власник: Dashboard → Замовлення → Меню → Курʼєри (база)
- Курʼєр: Завдання → Активна доставка
- Брендування: **тільки лого + колір** (базове)
- Оплата: **готівка при отриманні**
- Fallback на телефон закладу при будь-якій відмові
- Онбординг (8 кроків, окремий route)

**Відкладено в `planned-features` (схему БД лишаємо, код — ні):**
voice-ввід, AI price-intel, AI-асистент (розширений), webhooks, API-keys, recurring-замовлення, промо-конструктор (jsonb rule builder), embed-widget, custom-domain, custom-CSS, hypertables, PITR, behavioral-baselining (фрод-історія).

Критерій відрізу: **чи лежить на критичному шляху перших 50 закладів І чи додає новий стовп відмови.**

## 3. Чому лишились на Supabase (рішення-розворот)

Єдиною причиною йти з Supabase був депрекейт TimescaleDB. Оскільки **hypertables зняті зі скоупу** (§7), причина відпала. Залишатись — правильний наслідок, не sunk-cost.

Невикористання Supabase-надбудови (Realtime вимкнено, Auth свій, черга pg-boss) — це **нульова вага**, а не борг: вимкнена функція керованої платформи нічого не коштує. Натомість Supabase безкоштовно дає: керований Postgres ($25 Pro), Supavisor pooling, RLS-інтеграцію (зокрема для Storage), щоденні бекапи. Один вендор замість двох (Supabase + R2) — легше, ніж було.

**R2 лишається named-наступником за інтерфейсом сховища** (тригер — поява egress у рахунку), і **використовується вже зараз як зовнішня DR-копія файлів** (§5).

## 4. Черга: pg-boss (виправлення хибного припущення)

**ВИПРАВЛЕННЯ:** у розмові спершу хибно стверджувалось, що pg-boss залежить від `LISTEN/NOTIFY` і це конфліктує з пулером. **Це неправда.** pg-boss — **polling-based** (`SKIP LOCKED`); `pollingIntervalSeconds` керує опитуванням. NOTIFY у ядрі немає. Тож «проблеми NOTIFY через пулер» **не існує**.

**Рішення:**
- pg-boss у тому ж Postgres, **session mode (порт 5432)** через Supavisor, dedicated pool **max ~4**, `pollingIntervalSeconds` ~5.
- Session mode тут — не обхід, а **правильний режим для довгоживучого процесу** (Supabase сам це рекомендує для stateful/background). Прибирає весь клас гострих кутів (advisory locks, prepared statements) за конструкцією.
- Лінивий polling виправданий: критична джоба — скасування таймауту — **хвилинного масштабу** (5/10/15/30 хв), тож латентність підхоплення в секунди неважлива. Polling тут — перевага (менше chatter), не компроміс.
- **Транзакційний enqueue** зберігається: вставка замовлення + enqueue таймаут-джоби в **одній транзакції** (outbox-патерн) — два джерела істини не можуть розсинхронитись.

**Connection topology (Supavisor, стан з лютого 2025):**
- Порт **6543** = transaction mode (для API-трафіку)
- Порт **5432** = session mode (для воркера, аналітики, міграцій)
- Прямий конект = IPv6-only; для IPv4 — через Supavisor
- Пул-сайз **спільний** між обома портами — рахувати сукупно: `API(6543) + worker(5432) + analytics(5432) + migrations(5432) ≤ total pool size`. На 50 закладах запас великий.

## 5. DR і бекапи (виправлення хибного припущення)

**ВИПРАВЛЕННЯ:** хибно стверджувалось, що PITR входить у $25 Pro. **Неправда.** PITR — платний usage-based add-on (~$100–400/міс), вимагає мінімум Small-compute, і при ввімкненні **замінює** щоденні бекапи. Pro включає лише безкоштовні **щоденні бекапи (7 днів зберігання)**.

**Рішення MVP:**
- Покладаємось на безкоштовні щоденні бекапи, **RPO до 24 год** — свідомо прийнято (дані = замовлення/клієнти, не фінтранзакції; втрата доби болісна, але переживальна).
- PITR → `planned-features`, тригер — поява доходу / неприйнятність добового RPO.

**КРИТИЧНО — файли НЕ покриті бекапом БД:**
- Об'єкти Supabase Storage **не входять** у нативний бекап БД.
- Storage **не має версіонування** — видалені обʼєкти стираються назавжди.
- Тому **обовʼязковий окремий backup-job** (не відкладається):
  - Нічна **інкрементальна** синхронізація бакетів через S3-сумісний API (повний sync задорого по egress).
  - У **зовнішнє** сховище (R2 / інший провайдер), не в інший бакет того ж проєкту.
  - Версіонована ретенція (кілька днів назад) — компенсує відсутність версіонування Storage.
  - Запускати **не рідше** вікна бекапу БД, інакше `image_url` у відновленій БД вказуватимуть на неіснуючі файли.
  - Реалізувати як pg-boss scheduled job (той самий планувальник).

## 6. Redis: звужений до pub/sub

- Єдина роль — **pub/sub для крос-інстансного WS-broadcast**.
- Idempotency → **Postgres** (unique-constraint, §8), НЕ Redis-TTL — переживає падіння Redis.
- Кеш → in-process з коротким TTL.
- **Нюанс:** оскільки обрано WS-масштабування (кілька інстансів), Redis pub/sub стає тим, через що статус замовлення доходить до клієнтів на інших інстансах — тобто **важливіший, ніж «просто кеш»**, хоч і деградує gracefully (клієнт побачить при reconnect / на сторінці статусу, активній 7 днів).

## 7. Hypertables — знято зі скоупу

Відмовились від TimescaleDB-hypertables на MVP. Обґрунтування: для 50 закладів метрики (виручка, рейтинги, ai_usage) — це **низькооб'ємні звичайні таблиці** з btree-індексом `(location_id, ts)`. Єдина справді time-series-важка — `courier_locations`, але вона **дублює** живу позицію в Redis; персистентний слід зберігаємо як **один `delivery_trace` jsonb на DELIVERED**, а не сотні пінгів. Потреба в hypertables зникає, депрекейт TimescaleDB перестає бути проблемою. Якщо колись знадобиться сирий ping-аналіз — нативне declarative partitioning, не pg_partman, і це `planned-features`.

## 8. Stateful-борг — головне оновлення моделі

**Правило (точне формулювання):**
> У памʼяті процесу може жити лише **одноразовий, відновлюваний** стан. Жоден **авторитетний** стан (втрата якого = втрата правди) у памʼяті жити не може.
>
> **Тест:** «якщо процес зараз `kill -9` — це втрата чи незручність?» Втрата → стан у неправильному місці.
>
> **Лік один на всі:** винести авторитет у спільний backing-service; у памʼяті лишити те, що переживе `kill -9` своєю відновлюваністю (живі сокети, контекст запиту, пул — реконструюються через reconnect/retry).

Модель уже переважно правильна (JWT = stateless-сесія; pg-boss transactional enqueue = outbox; `timeout_at` = durable timer; MessageBus = backplane; idempotency-у-Postgres). Борг — **три-чотири протікання, які закриваємо ЗАРАЗ на N=1**, поки дешево:

| Протікання | Баг при N=2 | Фікс (зараз) |
|---|---|---|
| **Idempotency** у памʼяті/Redis-TTL | подвійні замовлення | unique-constraint у Postgres, у тій же транзакції зі вставкою |
| **node-cron** in-process | кожна задача фає двічі (дублі recurring/нагадувань) | pg-boss singleton scheduled jobs (`SKIP LOCKED`), викинути node-cron |
| **Локальна WS-мапа** broadcast | доставка лише частині клієнтів | broadcast через MessageBus-публікацію; сокети локальні, рішення сповістити — глобальне |
| **In-memory rate-limit** | атакуючий отримує N× ліміту | первинний rate-limit на Cloudflare; `@fastify/rate-limit` вторинний |

**Принцип «N-safe by construction»:** кодити так, ніби інстансів N, навіть запускаючи один. Тоді N=1→N — рядок конфігу (`[processes]`, RedisMessageBus), не рерайт.

**Перевірка:** crash-only мислення + у dev запускати **N=2**, щоб кожне in-memory-припущення впало явно, а не тихо легалізувалось під N=1.

## 9. Чому НЕ мікросервіси (зафіксоване рішення)

Stateful-борг — це борг **stateful-процесу**, не моноліту; мікросервіси його **розмножують** (той самий клас багів у кожному сервісі × репліки). Лік однаковий в обох (винести стан), у моноліті — раз.

- Незалежне масштабування частин → **вже маємо** через Fly `[processes]` (worker окремо від API), без мережевих меж.
- Ізоляція відмов → ілюзія: спільні Postgres/Redis лишаються SPOF хоч скільки сервісів; orders-сервіс мертвий = бізнес мертвий.
- Незалежні команди → **один розробник**, цієї проблеми немає.
- Ціна: мережа між сервісами, **розподілені транзакції** (транзакційний enqueue pg-boss розпадається!), saga/outbox, розподілене трасування, N пайплайнів, передчасні межі (найдорожче для зміни).

**Канон:** модульний моноліт (структура `modules/` — це вже воно) + інтерфейси; якщо один модуль впреться — вирізати **його одного** по відомому шву (strangler), не big-bang. Мікросервіси — рішення для масштабу організації, не трафіку 50 закладів.

## 10. Тригер масштабування N=1 → N (перекалібровано)

Головний тригер — **доступність, НЕ навантаження**. При N=1: будь-який синхронний сплеск в event loop / краш / деплой = **повний** простій. Можна мати вільний CPU й страждати від SPOF та деплой-вікон.

**Чесний тригер:** «коли деплой-вікно або ризик SPOF дорожчі за вартість другого інстансу + Redis-pub/sub». Наступає **раніше**, ніж упреться CPU.

| Поріг | Вмикається |
|---|---|
| ~50 закладів / пік | (вже є) окремий worker-процес |
| Деплой-вікно/SPOF болять | N>1 API + RedisMessageBus + sticky НЕ потрібен (WS лишається на інстансі, що прийняв upgrade; потрібен лише крос-інстансний broadcast) |
| Аналітика тисне | read-replica для analytics-пулу |
| Гео-трафік росте | окремий Redis під pub/sub |

## 11. Модель довіри до входів (фрод / курʼєр-логіка)

Фундамент із філософської гілки, дистильований у інженерні правила. Кожен вхід проганяється крізь осі:

1. **Авторитетний / одноразовий** — де живе.
2. **Виграє від брехні? / Канал перехоплюваний?** — вірити на слово чи валідувати. Незмагальна ланка = обидва «ні» (відправник не виграє І потік не контролює зацікавлений). Інакше — змагальна.
3. **Дія під невизначеністю** — утримання-в-тертя чи автодія.
4. **Клас дії** — захисна (зворотна, низька планка) чи **засуджувальна** (червона лінія).

**Ключові висновки:**
- **Недовіра — до перехоплюваного *каналу*, не до *людини*.** Презумпція невинності — для актора; верифікація — для сигналу. Чесний курʼєр і шахрай шлють побайтово однаковий пакет, тож недовіра до пакета ≠ судження про людину.
- **Жоден поодинокий клієнтський сигнал не є доказом.** Доказ = **кореляція незалежних сигналів, які важко підробити узгоджено**. Це стосується і GPS, і heartbeat.
- **heartbeat — змагальний, не довірена ланка** (виправлення: спершу хибно названо «надійним/непідмінним»). Наявність валідує лише «щось із валідним токеном надіслало пакет», НЕ «курʼєр присутній/працює». Підробляється тривіальніше за GPS (скрипт у циклі). Мінімум — серверний nonce + підпис проти replay; висновок «онлайн» — лише слабкий флаг.
- **GPS — змагальний канал** (spoof-абельний), валідується **жорсткими фізичними інваріантами** (accuracy > 100 м, швидкість > 150 км/год → відхилити позицію). Це доказ хибності *даних*, не вирок людині.
- **Історичне порівняння (behavioral baseline)** — лише **мʼякий флаг/скоринг/тертя**, ніколи автокарання, ніколи єдина підстава. Отруюється дисциплінованим брехуном (consistent ≠ truthful), сліпий на cold-start (новий курʼєр / щойно вкрадений токен). Сиру історію писати з day-1 (дешево, неавторитетний/перераховуваний стан), шар порівняння → `planned-features`.

**ЧЕРВОНА ЛІНІЯ — найважливіше:**
> Ніколи не дозволяй системі виносити вирок, який актор не може спростувати дією.

- Засуджувальні дії (статус «підозрюваний/фрод») — **окремий клас**: лише **людина-в-петлі**, висока багатовісна планка, **спростовний вихід** (фото/дзвінок/диспетчер реально знімає флаг), флаг **знімний беззалишково** і **поза baseline** (інакше хибне спрацювання отруює майбутні судження → самопідтверджувана спіраль).
- Невизначеність розряджається в **тертя, не у вирок** (утримання від автодії ≠ звинувачення).
- Асиметрія: хибне засудження — окремий, найгірший клас (незворотне, таврівне, підриває supply-side з 1–5 курʼєрів, **самовідтворюване** — виштовхує чесних, виробляє антисистемну поведінку). Калібрувати: низька планка для **захисних** (зворотних) дій, дуже висока для **засуджувальних**.

**Наскрізний інваріант (звʼязує §8 і §11):**
> Жоден стан системи не має бути пасткою. Дані — з носієм; твердження — зі способом перевірки; вирок — зі спростуванням; людина — з виходом. Стан без виходу відчіпляє актора від причинності — лишається голий наратив сили. Легітимний (гідний, не-таврівний) вихід — це клапан тиску системи: misfit іде раніше, ніж стане руйнівником. Справедливість тут — окремий випадок надійності й вигідна обом сторонам при чесному обліку часу (несправедливість = прихований борг під грабіжницький відсоток).

## 13. Залишкові ризики (несемо чесно)

- **N=1 = повний простій** при синхронному сплеску в event loop / краші / деплої. Мітигація: худий публічний SSR-бандл, **Cloudflare-кеш SSR-HTML по `menu_version`** (обовʼязковий, не опція — публічний `/s/:slug` б'є весь інтернет, це DoS-поверхня й CPU-контенція), деплой поза піком, дисципліна памʼяті + плановий нічний рестарт (немає природної рециркуляції процесу).
- **Fallback покриває клієнта, але не власника/курʼєра** при падінні (адмінка й кур'єрський PWA гаснуть одночасно). Прийнято як профіль відмови MVP; додатковий аргумент за ранній перехід N>1.
- **Storage egress метрується** ($0.09/ГБ після 250 ГБ) → Cloudflare-кеш перед Storage **обовʼязковий** для контролю вартості; backup-job — інкрементальний (інакше сам генерує egress).
- **Розмір Supabase-compute під pg-boss** (його фонове обслуговування + партиційовані таблиці ділять IO з SSR-читаннями) — **зміряти на bootstrap**, не припускати; Micro може бути тіснуватим.

## 14. Перевірити на bootstrap (НЕ вірити на слово)

- Дефолтна версія Postgres на нових проєктах Supabase (не валідовано; неважливо, бо ні hypertables, ні конкретна версія не потрібні — але звірити).
- Чистота встановлення pg-boss-схеми під привілейованою роллю Supabase (схема `pgboss`, партиції, права) — не валідовано.
- Чи якась pg-boss-операція обслуговування використовує session-scoped advisory locks (session mode 5432 це покриває за конструкцією, але звірити).
- Реальний пул-бюджет: сумарні конекти 6543+5432 ≤ pool size плану.

## 15. Незмінні нагадування (зведено)

- CSS variables `var(--brand-*)` скрізь, нуль хардкоджених кольорів.
- БЕЗ cookies — виключно localStorage/sessionStorage (Safari блокує 3rd-party cookies в iframe).
- Embed mode: без `position:fixed`, localStorage cart, postMessage height (але embed → `planned-features`).
- crypto.randomUUID(), не Math.random(); timingSafeEqual для HMAC.
- Supabase RLS з першої міграції; tenant-isolation **і на аналітичному direct-пулі** вручну (привілейований конект обходить RLS).
- PII у pg-boss payload — НЕ класти; **claim-check**: у payload лише `order_id`, воркер дочитує під явним `WHERE location_id`.
- Idempotency, cron, broadcast, rate-limit — авторитет ПОЗА памʼяттю процесу вже на N=1.
- Курʼєрський екран: tap targets ≥48px, критичні ≥56px.
- graceful shutdown (SIGTERM → дозлити/повернути pg-boss-джобу, закрити WS з reconnect-кодом) + глибокий `/health` (окремо PG/Redis, degraded≠503 для некритичних) + **окремий liveness/heartbeat-моніторинг worker-процесу** (винесений воркер робить тихими відмови — мертвий воркер = таймаути не спрацьовують = PENDING висять назавжди, ніхто не бачить).

## 16. Наступний крок у код

Три точки входу (обрати одну):
- **(а)** Схема idempotency у Postgres: таблиця/констрейнт, що повертати на повтор, чистка старих ключів.
- **(б)** Контракт `deliver` із кореляцією: які осі, поріг тертя, технічно знімний флаг поза baseline.
- **(в)** `MessageBus`-інтерфейс broadcast так, щоб N=1→N був перемикачем.
